# スキルレビュー

公式ベストプラクティスとAgent Skills仕様に基づいてスキルをレビューし、具体的な改善提案を行う。

## レビュー手順

1. 対象スキルを特定（ユーザー指定 or カレントディレクトリ検索）
2. SKILL.md + ディレクトリ構成（`scripts/`, `references/`, `assets/`, `evals/`）を読み込む
3. 以下のチェックリストに照らして全項目を評価
4. レビューレポートを生成（テンプレートは末尾参照）
5. ユーザーと修正を進める（Critical優先、段階的に適用、各修正後に再検証）

## 1. Frontmatter (YAML)

### name
- [ ] 64文字以内である
- [ ] 小文字、数字、ハイフンのみ使用している
- [ ] ハイフンで始まらず、ハイフンで終わらない
- [ ] 連続ハイフン（`--`）を含まない
- [ ] 親ディレクトリ名と一致している
- [ ] 曖昧な名前を避けている（`helper`, `utils`, `tools`はNG）
- [ ] 予約語を含まない（`anthropic`, `claude`はNG）

### description
- [ ] 1024文字以内である
- [ ] 空でない
- [ ] スキルが**何をするか**を説明している
- [ ] スキルを**いつ使うべきか**を説明している
- [ ] 命令形で記述している（「Use when...」スタイル。「I can help」はNG）
- [ ] ユーザーの意図に焦点を当てている（内部実装ではなく）
- [ ] 具体的なトリガーワードやシナリオを含んでいる
- [ ] ファイル形式やタスクタイプを明示している
- [ ] 実装手段（ツール名・ライブラリ名）を含まない
- [ ] 明示的にドメインを言及しないケースもカバーしている
  - 良い例: `even if they don't explicitly mention "CSV" or "analysis"`
- [ ] 積極的（pushy）に適用範囲を示している
- [ ] トリガーラベルは「Use When:」で統一している（「使用タイミング:」等はNG）

### license（オプション）
- [ ] ライセンス名またはバンドルされたライセンスファイルへの参照
- [ ] 簡潔に記述している

### compatibility（オプション）
- [ ] 500文字以内である
- [ ] 環境要件がある場合のみ使用している
- [ ] 必要なランタイム・パッケージ・ネットワークアクセスを明記

### metadata（オプション）
- [ ] キー名が十分にユニークで衝突しない
- [ ] 文字列キーと文字列値のマッピングである

### allowed-tools（オプション・実験的）
- [ ] スペース区切りの事前承認ツールリスト
- [ ] エージェント実装によりサポート状況が異なることを理解している

## 2. SKILL.md Body

### 簡潔さ (Conciseness)
- [ ] 500行以下、5000トークン以下を目標としている
- [ ] エージェントが既に知っている情報を含まない
  - HTTP、PDF、DBマイグレーション等の一般知識は不要
  - 判断基準: 「エージェントがこの指示なしで間違えるか？」
- [ ] 冗長な説明より簡潔な例を優先している
- [ ] 各情報がトークンコストに見合う価値がある
- [ ] 「適切に処理する」「ベストプラクティスに従う」等の汎用アドバイスがない

### 構造
- [ ] 明確なセクション分けがある
- [ ] 目次またはナビゲーションがある（長い場合）
- [ ] 「When to Use」セクションがbodyにない（descriptionに含めるべき）

### Progressive Disclosure
- [ ] 詳細な情報は別ファイル（`references/`, `assets/`）に分離している
- [ ] 参照ファイルへのリンクは1階層深さまで（深いネスト禁止）
- [ ] 参照ファイルの**読み込み条件**が明確に説明されている
  - 良い例: 「APIが非200ステータスを返した場合、`references/api-errors.md`を参照」
  - 悪い例: 「詳細はreferences/を参照」
- [ ] 100行以上の参照ファイルには目次がある
- [ ] 個々の参照ファイルは焦点を絞った内容

### スキルスコーピング（coherent unit設計）
- [ ] 一貫した作業単位をカプセル化している
- [ ] 狭すぎない（1タスクに複数スキルが必要になる → オーバーヘッド・指示の矛盾リスク）
- [ ] 広すぎない（正確なトリガーが困難になる）
- [ ] 関数のスコープ設計と同じ考え方: 他スキルと合成しやすい粒度

## 3. コンテンツ設計

### 実際の専門知識に基づいているか
- [ ] 汎用的なLLM知識の焼き直しではない
- [ ] プロジェクト固有の規約・エッジケース・ツール選定を含む
- [ ] 情報ソースが明確（内部ドキュメント・インシデント・コードレビュー・Git履歴等）
- [ ] 実タスクでの実行→修正→改善のサイクルを経ている
  - ハンズオンタスクから成功パターン・修正点・入出力形式・提供したコンテキストを抽出
  - または既存のプロジェクト成果物（runbook, API仕様, スキーマ, 障害対応記録等）から合成

### 制御の粒度調整（Calibrating Control）
- [ ] タスクの脆弱性に応じて指示の具体性を調整している
  - 柔軟なタスク: 「何を見るか」の方針レベル → 理由（why）を説明
  - 脆弱なタスク: 正確な手順・コマンドを明示 → 逸脱禁止
- [ ] 1つのスキル内で柔軟/厳格を適切に使い分けている
- [ ] スキル自体がRigid（TDD等: 厳密に従う）かFlexible（パターン: 原則を適応）かを明示

### デフォルトの提供
- [ ] 複数のツール/手法がある場合、デフォルトを1つ選んでいる
- [ ] 代替手段は簡潔に言及するのみ（対等な選択肢として並べない）
- [ ] 「メニュー」ではなく「推奨＋エスケープハッチ」の構造

### 手順 vs 宣言
- [ ] 特定のインスタンスへの回答ではなく、再利用可能な手順を記述
- [ ] アプローチが汎化されている（具体的なカラム名→一般的な手法）
- [ ] 出力フォーマットテンプレートや制約（「PII禁止」等）は具体的でOK

## 4. 効果的な指示パターン

### Gotchasセクション
- [ ] 環境固有の「常識に反する事実」をリストしている
- [ ] 具体的で検証可能な内容である（「エラーを適切に処理」はNG）
- [ ] エージェントが遭遇する前にSKILL.md内で読めるようにしている
- [ ] 実行時に発見した間違いをGotchasに追加するフローがある

### テンプレート
- [ ] 特定のフォーマットが必要な場合、散文ではなく構造テンプレートを提供
- [ ] 短いテンプレートはインラインで、長いものは`assets/`に分離
- [ ] テンプレートは適応可能な余地を残している

### チェックリスト
- [ ] 複数ステップのワークフローに進捗追跡用チェックリストがある
- [ ] ステップ間の依存関係やバリデーションゲートが明記

### バリデーションループ
- [ ] 重要な操作後に「実行→検証→修正→繰り返し」のループがある
- [ ] 検証手段が明確（スクリプト、リファレンスとの照合、セルフチェック）
- [ ] バリデーション通過まで次のステップに進まない指示がある

### Plan-Validate-Execute
- [ ] バッチ処理・破壊的操作は中間プランを作成している
- [ ] プランを真実のソースに対して検証してから実行
- [ ] バリデーションスクリプトがエージェントの自己修正に十分なエラー情報を返す

### スクリプトのバンドルシグナル
- [ ] エージェントが毎回同じロジックを再発明している場合、`scripts/`にまとめている
  - チャート生成、特定フォーマットのパース、出力検証 等
- [ ] テスト済みスクリプトとして1回書いてバンドル

## 5. Scripts（scripts/がある場合）

### エージェント向け設計（必須）
- [ ] 対話型プロンプト（TTY入力）を使用していない（ハング原因）
- [ ] すべての入力をCLIフラグ・環境変数・stdinで受け付ける
- [ ] 必須引数が欠けた場合、明確なエラーとUsageを表示する
- [ ] `--help`が使用方法・フラグ・例を含む簡潔な出力を返す

### 出力設計
- [ ] 構造化出力（JSON, CSV, TSV）を優先している
- [ ] データはstdout、診断情報はstderrに分離している
- [ ] 出力サイズが予測可能（巨大な出力には`--offset`や要約のデフォルト）
- [ ] エージェントハーネスの自動切り詰め（10-30K文字）を考慮している

### エラーメッセージ
- [ ] 「何が間違いか＋期待値＋次のアクション」を含む
- [ ] 異なる失敗タイプに異なる終了コードを使用し、`--help`に文書化

### 堅牢性
- [ ] 冪等性がある（リトライしても安全。「既存ならスキップ」）
- [ ] 曖昧な入力はエラーで拒否（推測しない。enum/閉集合を使用）
- [ ] 破壊的操作には`--dry-run`フラグがある
- [ ] 安全なデフォルト（破壊的操作は`--confirm`/`--force`を要求）

### 自己完結性と依存管理
- [ ] 自己完結型、または依存関係をインラインで宣言している
  - Python: PEP 723（`# /// script` + `dependencies`）→ `uv run`で実行
  - Deno: `npm:`/`jsr:`インポート指定子でパッケージ自動解決
  - Bun: インポートパスにバージョン指定で自動インストール
  - Ruby: `bundler/inline`でgemをスクリプト内宣言
- [ ] バージョンを固定している（`npx eslint@9.0.0`, `uv run ruff@0.8.0`等）
- [ ] プラットフォーム制限が明記されている
- [ ] 前提条件がSKILL.mdまたは`compatibility`フィールドに記載

### ワンオフコマンド
- [ ] 既存パッケージで十分な場合はバンドルせず直接参照
- [ ] ランタイム別ツール: `uvx`/`pipx`(Python), `npx`/`bunx`(JS), `deno run`, `go run`
- [ ] 複雑になったら`scripts/`に移行

### スクリプト一覧の明示
- [ ] SKILL.md内に`## Available scripts`セクションで利用可能なスクリプトをリストしている
  - エージェントはリストがないとスクリプトの存在を認識できない
  - 各スクリプトの用途を1行で説明

### ファイルパス
- [ ] フォワードスラッシュを使用している（`scripts/helper.py`）
- [ ] Windowsスタイルのパスを使用していない
- [ ] スキルルートからの相対パスを使用している
- [ ] ファイル名が内容を示す説明的な名前である

## 6. Description最適化

### トリガー精度
- [ ] should-triggerケース（発動すべき）を想定できるか
- [ ] should-not-triggerケース（発動すべきでない隣接ドメイン）を想定できるか
- [ ] ニアミス（キーワードは似ているが別タスク）で誤発動しないか
- [ ] 明示的にドメインを言及しないカジュアルな依頼でも発動するか
- [ ] 複数ステップの長い依頼の中に埋もれていても発動するか

### トリガーの限界（理解しておくべきこと）
- エージェントは自力で処理できる単純なタスクにはスキルを参照しない場合がある
- descriptionが完璧でも、1ステップで済むタスクではshould-triggerが100%にならないことがある
- スキルが価値を発揮するのは、専門知識・ドメイン固有ワークフロー・非自明なエッジケースを要するタスク

### Evalクエリ設計（推奨: 約20件）
- [ ] should-trigger: 8-10件（フォーマル/カジュアル/typo/略語を混在）
- [ ] should-not-trigger: 8-10件（ニアミスを重視、明らかに無関係なものは無価値）
- [ ] フレーズの多様性: 明示的/暗黙的、簡潔/詳細、単純/複雑を網羅
- [ ] リアルなコンテキスト: ファイルパス、個人的文脈、カラム名等を含む

### トリガーテスト
- [ ] 各クエリを複数回実行してtrigger_rateを計測（非決定性対策、最低3回）
- [ ] should-trigger: trigger_rate > 0.5 で合格
- [ ] should-not-trigger: trigger_rate < 0.5 で合格

### 過学習防止
- [ ] trainセット（~60%）とvalidationセット（~40%）に分割
- [ ] 両セットにshould-trigger/should-not-triggerを比例配分
- [ ] trainセットの失敗のみで改善を行い、validationは検証のみに使用
- [ ] 特定フレーズへの最適化ではなく一般的なカテゴリで対応

### 最適化ループ（最大5回反復）
1. train/validationの両方で評価
2. trainセットの失敗を特定（should-triggerの未発動、should-not-triggerの誤発動）
3. descriptionを修正（狭すぎ→スコープ拡大、広すぎ→境界明確化）
4. 改善が停滞したら構造的に異なるアプローチを試す
5. validation pass rateが最も高いイテレーションを選択（最終版とは限らない）
- [ ] 1024文字制限を超えていないか定期的に確認

## 7. Eval（品質評価）体系

evals/が存在しない場合、Infoレベルで導入を推奨。

### テストケース設計
- [ ] `evals/evals.json`にテストケースを定義している
- [ ] 各テストケースにprompt, expected_output, files（任意）, assertionsを含む
- [ ] 最初は2-3件で開始し、結果を見て拡充
- [ ] プロンプトのバリエーション: カジュアル/精密、エッジケース含む
- [ ] 実際のユーザーが入力するようなリアルなコンテキストを含む

### 実行方法
- [ ] with_skill / without_skill（またはold_skill）の2構成で実行
- [ ] 各実行はクリーンなコンテキスト（残留状態なし）で行う
- [ ] 結果をワークスペースに`iteration-N/eval-{name}/{with,without}_skill/`で保存
- [ ] `timing.json`（total_tokens, duration_ms）を記録

### アサーション設計
- [ ] 初回実行結果を見てからアサーションを追加（事前に書きすぎない）
- [ ] 具体的・検証可能（「出力が良い」はNG、「軸ラベルがある」はOK）
- [ ] プログラムで検証可能なもの（valid JSON, 行数等）はスクリプトで検証
- [ ] 常にPASSするアサーションは削除（スキルの価値を示さない）
- [ ] 常にFAILするアサーションはアサーション自体を修正

### グレーディング
- [ ] 各アサーションにPASS/FAILとエビデンスを記録（`grading.json`）
- [ ] PASSには具体的エビデンスを要求（「たぶんOK」はNG）
- [ ] 出力/アサーションをLLMまたはスクリプトで評価
- [ ] ブラインド比較: 2バージョンの出力をどちらか隠してLLMジャッジに評価させる

### ベンチマーク集約
- [ ] `benchmark.json`に構成別の集約統計を保存
- [ ] pass_rate, time_seconds, tokens のmean/stddevを算出
- [ ] deltaで「スキルが何をコストし何を得るか」を定量化

### パターン分析
- [ ] 両構成で常にPASSするアサーションを削除/置換
- [ ] 両構成で常にFAILするアサーションを修正
- [ ] with_skillのみPASSするアサーションを分析（スキルの価値の所在）
- [ ] 結果が不安定（高stddev）な場合、指示の曖昧さを減らす
- [ ] 時間/トークンの外れ値は実行トレースで原因特定

### 人間レビュー
- [ ] アサーションでカバーできない品質（文体、視覚的デザイン、「しっくりくるか」）を確認
- [ ] 具体的フィードバックを`feedback.json`に記録（「見た目が悪い」はNG）
- [ ] 空のフィードバック = レビュー合格

### 反復改善ループ
- [ ] 3つのシグナル源: failed assertions + human feedback + execution traces
- [ ] 実行トレースを読む（最終出力だけでなく）
  - 無駄なステップの原因: 指示が曖昧 / 適用外の指示に従っている / 選択肢が多すぎ
- [ ] 改善はシグナル源すべてとSKILL.mdをLLMに渡して提案させる
- [ ] 一般化: テストケース固有のパッチではなく根本原因に対応
- [ ] 簡潔化: 指示を追加するだけでなく不要な指示の削除も検討
- [ ] 理由の説明: 「常にXせよ」より「XすべきなぜならY」の方が効果的
- [ ] エージェントが毎回同じロジックを再発明→`scripts/`にバンドル

## 8. ディレクトリ構成

### assets/ディレクトリ
- [ ] テンプレート（ドキュメントテンプレート、設定テンプレート）を格納
- [ ] 画像（図表、例示）を格納
- [ ] データファイル（ルックアップテーブル、スキーマ）を格納
- [ ] 長いテンプレートはインラインではなく`assets/`に分離している

### references/ディレクトリ
- [ ] ドメインまたは機能ごとに整理されている
- [ ] 必要な時だけ読み込まれるよう設計されている
- [ ] SKILL.mdと情報が重複していない
- [ ] 個々の参照ファイルは焦点を絞った内容

### Content Quality
- [ ] 一貫した用語を使用している（同義語の混在を避ける）
- [ ] 時間に敏感な情報を含まない
- [ ] 命令形/不定詞形で記述している
- [ ] 具体的な入出力例がある
- [ ] 基本と応用の両方の例がある
- [ ] 例のスタイルが一貫している
- [ ] データ形式にはstrictテンプレート、創造的タスクにはflexibleテンプレート

### MCP Tools
- [ ] 完全修飾ツール名を使用している（`ServerName:tool_name`）
- [ ] サーバープレフィックスを省略していない

### バリデーション
- [ ] `skills-ref validate ./my-skill` でフロントマターの検証を実施

## 9. Anti-Patterns（避けるべきこと）

- [ ] デフォルトなしで多くの選択肢を提示していない
- [ ] Windowsスタイルのパスを使用していない
- [ ] 時間に敏感な情報を含まない
- [ ] 曖昧な説明やスキル名を使用していない
- [ ] 深くネストしたファイル参照がない
- [ ] パッケージがプリインストール済みと仮定していない
- [ ] マジック定数を説明なしで使用していない
- [ ] 1つのデフォルトで十分な時に複数のアプローチを並列提示していない
- [ ] スクリプトでエラー条件を処理し忘れていない
- [ ] 一貫性のない用語を使用していない
- [ ] 汎用的なアドバイス（「エラーを適切に処理」「ベストプラクティスに従う」）がない
- [ ] エージェントが既に知っている一般知識を説明していない
- [ ] 特定インスタンスへの回答（ルックアップテーブル）ではなく手順を記述している
- [ ] 対話型プロンプトをスクリプトで使用していない
- [ ] スキルスコープが狭すぎも広すぎもしない

## 重大度分類

### Critical（必須修正）
- descriptionが空または不十分
- 500行を大幅に超えている（目標: 5000トークン以下）
- nameが親ディレクトリ名と不一致
- nameに連続ハイフンやハイフン始まり/終わりがある
- セキュリティ上の問題がある
- スクリプトにエラー処理がない
- スクリプトが対話型プロンプトを使用している（ハング原因）

### Warning（推奨修正）
- Progressive Disclosureが適用されていない
- 参照ファイルの読み込み条件が不明確（「詳細は参照」のみ）
- 例が不十分
- 用語の一貫性がない
- ワークフローが不明確
- デフォルトなしの複数選択肢を提示
- エージェントが既知の情報を冗長に説明
- descriptionがユーザー意図ではなく実装手段に焦点
- Gotchasセクションが必要だが欠如
- バリデーションループが必要だが欠如
- スクリプトに`--help`がない
- スキルスコープが狭すぎまたは広すぎ
- スクリプト出力が非構造化（stdout/stderrが混在）

### Info（改善提案）
- より簡潔にできる箇所がある
- 構造を改善できる
- 追加の例があると良い
- descriptionのトリガー精度を改善できる（evalクエリ設計の推奨）
- 制御の粒度をタスクに合わせて調整できる
- テンプレートの追加で一貫性を高められる
- Eval体系を導入してスキル品質を定量的に改善できる
- ブラインド比較で新旧バージョンの品質差を客観評価できる
- エージェント実行トレースの分析で無駄なステップを削減できる
- 繰り返し再発明されるロジックを`scripts/`にバンドルできる

## レポートテンプレート

```markdown
# スキルレビューレポート: {skill-name}

## サマリー
- 総合評価: {PASS | NEEDS_IMPROVEMENT | CRITICAL_ISSUES}
- Critical: {件数}
- Warning: {件数}
- Info: {件数}

## Critical（必須修正）
{修正必須の問題をリスト。エビデンスと修正例を含む}

## Warning（推奨修正）
{推奨される改善をリスト。Before/Afterの例を含む}

## Info（改善提案）
{オプションの強化をリスト}

## Description最適化
{トリガー精度の分析結果とdescription改善案}

## 具体的な推奨事項
{例を含む具体的なアクションアイテム}
```

## 概要

プロンプトを「直接編集して再生成」するのではなく、「**生成 → 評価 → プロンプト修正 → 再生成**」の閉ループで改善する。修正対象は常にプロンプト本体であって、生成された出力ではない。出力を直接修正しても再現性がないため意味がない。

## 前提条件

- 改善対象のプロンプトファイルがある（パス指定で渡される）
- プロンプトに渡す入力ファイル（Phase 1 構成・仕様書・要求文など）が確定している
- 次のいずれかが評価基準として使える:
  - **ゴールデン例**: あるべき出力の理想例（diff で比較できる）
  - **評価チェックリスト**: 「これが入っていれば合格」の項目集
- 反復履歴を残す作業ディレクトリ（**必ず `<workspace>/prompt-tuning/<project-slug>/` に切る**。詳細は Step 1）に書き込み権限がある
- git リポジトリ内である（各ラウンドをコミットして履歴化する）

## 大原則: ファイル局在化

**1 プロジェクトのチューニングに必要なすべてのファイル（入力・ゴールデン・参考データ・プロンプト・生成結果・評価レポート）は、単一の作業ディレクトリ `<workspace>/prompt-tuning/<project-slug>/` 配下に集約する。外部の絶対パスを参照しない。**

理由:
- 別セッションでのリファクタリング・整理・削除に巻き込まれて入力やゴールデンが消失すると、評価エージェントが指定パスを読めず、存在しないファイルを「実在する」とハルシネーションして評価レポートを返すことがある（実害事例あり）。プロジェクト単位で物理的に隔離することで事故を防ぐ
- サブエージェントに渡すパスが 1 ディレクトリ配下に閉じるので、間違った参照や読み忘れが起きにくい
- 後から別プロジェクトで再利用するときに、ディレクトリ単位でコピー・参照できる

## 用語

- **対象プロンプト**: 改善する prompt ファイル（例: `prompts/prompt-vN.md`）
- **入力ファイル**: 対象プロンプトに渡す具体ケース（例: Phase 1 構成、仕様書）
- **生成エージェント**: 対象プロンプト + 入力ファイルを受け取り、出力を作る subagent
- **評価エージェント**: 生成された出力を、ゴールデンまたはチェックリストと突き合わせる subagent
- **ゴールデン**: あるべき出力の理想例。生成エージェントには **絶対に見せない**

## ループの全体像

```
ラウンド N:
  1. 対象プロンプト v(N) を確定（前ラウンドの指摘を反映）
  2. 生成エージェントを spawn（対象プロンプト + 入力ファイル のみ渡す。ゴールデン・過去出力は禁止）
  3. 出力 output-v(N).md が生成される
  4. 評価エージェントを spawn（出力 + 対象プロンプト + ゴールデン or チェックリスト を渡す）
  5. 評価レポート eval-v(N).md を取得
  6. 指摘を 1〜数点ずつ対象プロンプトに反映 → v(N+1)
  7. 過学習スキャン（次節） → ヒットあれば修正
  8. コミット → ラウンド N+1 へ
収束条件:
  - 評価指摘がすべて PASS
  - または、ゴールデン一致度が目標値（例: 99%+）に達した
  - または、ユーザーが手動で停止
```

## Step 1: 作業ディレクトリの初期化と入力ファイルの集約

### 1-1. ディレクトリ構成（厳守）

ワークスペース直下に **必ず** 次の構成で作業ディレクトリを切る:

```
<workspace>/prompt-tuning/<project-slug>/
  inputs/       ← 入力ファイル・ゴールデン・参考データ（外部からコピーして集約）
  prompts/      ← 各ラウンドのプロンプト v1, v2, ...
  outputs/      ← 各ラウンドの生成結果（round-NN-output.md）
  evals/        ← 各ラウンドの評価レポート（eval-vN.md）
```

- `<project-slug>` は今回のチューニング対象を表す短い英数字スラッグ（例: `script-from-marp`、`outline-to-deck`）
- ディレクトリ名は `iteration-state/` 等の旧称を使わず、必ず `prompt-tuning/` を使う

```bash
mkdir -p <workspace>/prompt-tuning/<project-slug>/{inputs,prompts,outputs,evals}
```

### 1-2. 入力ファイル・ゴールデンを inputs/ に集約

**外部にある入力ファイル・ゴールデン例・参考データは、必ず `inputs/` にコピーする。外部の絶対パスをそのまま使い続けない。** 後の Step 2/3 でサブエージェントに渡すパスは、すべて `prompt-tuning/<project-slug>/` 配下のもののみ。

集約方法は以下のいずれか:

- **通常のコピー**: `cp <外部パス> inputs/<わかりやすい名前>.md`
- **git 履歴からの固定スナップショット**（外部ファイルが消失/移動するリスクがある場合に推奨）:
  ```bash
  git show <commit>:<外部パス> > inputs/<わかりやすい名前>.md
  ```
- **シンボリックリンク禁止**: シンボリックリンクは外部実体に依存して結局壊れるので、必ず実ファイルとしてコピーする

集約後の典型例:

```
inputs/
  golden.md           ← ゴールデン例（理想出力）
  source.md           ← プロンプトに渡す入力ケース（仕様書 / Phase 1 構成 / スライド md など）
  reference-A.md      ← 参考データがあれば追加（差分レポート、過去事例など）
```

### 1-3. パス指定の絶対ルール

以降の Step 2（生成エージェント spawn）と Step 3（評価エージェント spawn）で **サブエージェントに渡すファイルパスは、必ず `<workspace>/prompt-tuning/<project-slug>/` 配下のもののみ** にする。外部の絶対パス（別プロジェクトのディレクトリ、`iteration-state/` などの旧構成、テンポラリパス等）をサブエージェントに渡してはならない。

理由: 外部パスを直接渡すと、別セッションでのリファクタリング・整理・削除に巻き込まれてファイルが消失したときに、評価エージェントが「指定パスを読めず、存在しないファイルを実在するとハルシネーションして評価レポートを返す」事故が起きる（実害事例あり）。`inputs/` 配下に物理的に隔離することでこの種の事故を防ぐ。

## Step 2: 生成エージェントの spawn

### 2-1. 渡す情報

- 対象プロンプトのフルパス（`prompt-tuning/<project-slug>/prompts/prompt-vN.md`）
- 入力ファイルのフルパス（`prompt-tuning/<project-slug>/inputs/<name>.md`）
- 出力先パス（`prompt-tuning/<project-slug>/outputs/round-NN-output.md`）
- **読んではいけないファイルのリスト**: 同 `inputs/` 配下のゴールデン、同プロジェクトの過去ラウンドの出力・評価レポート・過去プロンプト、参考実装。これを必ず明示する

**渡すパスは必ず `prompt-tuning/<project-slug>/` 配下のもののみ**。外部の絶対パスを渡してはならない（Step 1-3 のルール）。

### 2-2. プロンプトテンプレート

```
あなたは生成エージェントです。指定プロンプトと入力ファイルから出力を作成してください。

## 読むファイル
1. 対象プロンプト: <workspace>/prompt-tuning/<project-slug>/prompts/prompt-vN.md
2. 入力ファイル: <workspace>/prompt-tuning/<project-slug>/inputs/<name>.md
3. その他必要な参照: <workspace>/prompt-tuning/<project-slug>/inputs/<other>.md

## 絶対に読んではいけないファイル
- <workspace>/prompt-tuning/<project-slug>/inputs/golden.md （正解が見えるとプロンプトの効果が測れない）
- <workspace>/prompt-tuning/<project-slug>/outputs/round-*.md （前回の構造を踏襲してしまうため）
- <workspace>/prompt-tuning/<project-slug>/evals/eval-*.md （評価結果からのリバースが起きるため）
- <workspace>/prompt-tuning/<project-slug>/prompts/prompt-v*.md のうち今回指定された v 以外（過去プロンプトも見ない）

## 出力先
<workspace>/prompt-tuning/<project-slug>/outputs/round-NN-output.md

## 完了条件
- ファイルが 1 つ書き出されている
- 対象プロンプトの規定をすべて適用している
```

### 2-3. なぜゴールデンを読ませないか

生成エージェントがゴールデンを参照すると、**プロンプトを介さずに直接コピー**するバイアスが入る。プロンプトの記述だけで何が再現できるかを測るのが本ループの目的なので、ゴールデンへの直接参照は禁止。

過去ラウンドの出力も同様。前回の構造を踏襲してしまい、プロンプト変更の効果が見えなくなる。

## Step 3: 評価エージェントの spawn

### 3-1. 渡す情報

- 生成された出力ファイル（`prompt-tuning/<project-slug>/outputs/round-NN-output.md`）
- ゴールデン例（`prompt-tuning/<project-slug>/inputs/golden.md`。**評価エージェントには見せてよい**。比較が役割なので）
- 対象プロンプトのフルパス（`prompt-tuning/<project-slug>/prompts/prompt-vN.md`。プロンプト側の問題箇所を特定するため）
- 評価軸または検証ポイント（後述）

**渡すパスは必ず `prompt-tuning/<project-slug>/` 配下のもののみ**。評価エージェントが「指定パスが見つからない」となったら、ハルシネーションで別ファイルを「実在する」と扱って評価を返してくる事故が起きる。Step 1-2 で `inputs/` への集約が済んでいることを確認してから spawn する。

### 3-2. 評価軸の与え方

ユーザーから指摘事項が出ていれば、**それぞれを独立した検証ポイントとして列挙する**。指摘ごとに PASS / FAIL を返してもらう。

新規ループの場合は次の標準観点を使う:
- 構造一致度（スライド数・順序・セクション分け）
- 文体一致（敬体/常体、用語の硬さ）
- 重要要素の欠落の有無
- ゴールデンに無い余分な要素の有無
- **過学習混入の有無**（次節）

### 3-3. プロンプトテンプレート

```
あなたは評価エージェントです。生成出力をゴールデンと比較・評価してください。

## 読むファイル
- 対象プロンプト: <path>（プロンプト側の問題箇所特定のため）
- 生成出力: <path>
- ゴールデン: <path>
- 入力ファイル: <path>

## 検証ポイント（指摘 1 点 = 1 観点）
1. <観点 1>: 期待される状態
2. <観点 2>: 期待される状態
...

## 報告フォーマット
- 各観点を PASS / FAIL で判定
- FAIL の場合: 出力のどこで failed か、プロンプトのどの記述が原因か
- 過学習スキャン結果（次節参照）

ファイルパスとサマリだけ。長い報告は不要。
```

## Step 4: 過学習スキャン（毎ラウンド必須）

過学習 = プロンプト本文が「特定ケース固有の例語・固有名・具体値」を含んでしまい、他ケースに使うと汚染する状態。これは毎ラウンド必ずチェックする。

### スキャン対象（grep ベース）

ループのドメインに応じて 3 層で分けて検出する:

**A. 対象ケース固有の用語**
- 入力ファイルに登場する固有名詞・クラス名・キーワード
- これらがプロンプト本文に直接書かれていたら過学習

**B. 対象ケース固有の具体値**
- 識別子・名前・数値・日付・URL・人名・地名
- これらがプロンプト本文に直接書かれていたら過学習

**C. 対象環境固有の用語**
- 特定リポジトリ名、社内用語、特定講座番号、特定プロダクト名
- 「広く使えるプロンプト」が目的なら、これらも過学習

### スキャンコマンド例

```bash
grep -in "<対象ケース固有用語>" <対象プロンプト>
grep -nE '"[^"]*"|= [0-9]+' <対象プロンプト>  # 文字列リテラル・数値リテラル
grep -in "<固有名詞>" <対象プロンプト>
```

### ヒットしたら

- 抽象表現（「同種のインスタンスを 2 件用意」「自然言語ラベル」「最小桁の数値」）に書き換える
- 必要なら「例」「参考」と明示し、「機械写し禁止」を併記する
- 自己点検チェックリストに「プロンプト本文の例語を出力にコピーしていない」を追加する

### スキャンを評価エージェントに任せる場合

評価エージェントの検証ポイントに「プロンプト本文に <対象ケース固有語> が含まれていないか」を必ず入れる。生成エージェントだけに任せると見落とす。

## Step 4.5: 指摘の種類別の反映方法

指摘の言語的レベルを見極めて反映する:

| 指摘の種類 | 反映方法 |
|---|---|
| **a. 表現の言い換え**（「もっと平易に」） | 該当ルールに言葉の選び方の指針 + 具体例 2〜3 個 + 「機械写し禁止」併記 |
| **b. ルールの追加**（「Y のときは Z やめて」） | 新しい条項を追加。条件と帰結を「〇〇 のときは △△ する」形式で明示 |
| **c. ルールの厳格化**（回帰した場合） | 「絶対」「必ず」「禁止」を入れる / 条項本体 + 自己点検チェックリスト の 2 箇所に書く / 反例を明示する |
| **d. スコープの分離**（「X だけに適用、Y には適用しない」） | 適用範囲と非適用範囲を対比形式で両方書く |
| **e. 概念のレベル調整**（「具体名ではなくカテゴリ」） | 抽象度のメタ規則を追加。「対象は型のレベルの概念で、特定インスタンス名ではない」 |

## Step 5: 指摘を反映するときの原則

### 原則 1: 出力を直接編集しない

ユーザーから「ここをこう直して」と言われても、**まずプロンプトを修正**する。出力を直接編集しても次回以降の生成では再現されないので意味がない。

唯一の例外: 1 回限りの納品物として今すぐ手元の出力を修正する必要がある場合のみ、出力も同時に編集してよい。ただし必ず同じ指示をプロンプトにも反映する。

### 原則 2: 1 指摘 = 1 修正 = 1 検証

指摘を複数まとめて反映してから検証すると、どの修正が効いたか・どれが回帰を起こしたかが切り分けられない。原則として 1 指摘ずつプロンプトに反映し、再検証する。

軽微な指摘が並んでいる場合は、互いに干渉しないことを確認した上でまとめてもよい。

### 原則 3: 指摘を「ルールに昇格」させる

ユーザーの指摘は「特定箇所への修正」と「ルールへの昇格」の 2 段階で扱う:

- 修正: 該当箇所を直す
- ルール昇格: プロンプトの規約セクションに「同型のケースで同じ判断をせよ」と書く

ルール昇格を怠ると、別のケースで同じ問題が再発する。

### 原則 4: 検証の前に必ず diff を取る

プロンプトを修正したら、修正前後の diff を確認する。意図しない部分を変更していないか、関連箇所に矛盾を生んでいないかを目視で確認する。

### 原則 5: 「完了宣言の前に実ファイルを開く」

評価エージェントの報告を信じきらない。ユーザーから「見出しに X が入っているか確認してから完了宣言してほしい」のような指摘を受けたら、**自分でファイルを開いて grep し**、引用付きで報告する。

評価エージェントは出力を読んでまとめるが、まとめの過程で重要箇所を見落とすことがある。

## Step 6: コミットと履歴化

各ラウンドの最後に必ずコミットする。コミットメッセージは次の形式:

```
iter v<N>: <主な変更点 1 行>
```

例:
```
iter v3: S7 必ず 2 枚規則を強化、過学習スキャンで「具体値」混入を発見・除去
```

コミット粒度を細かくしておくと、後で「どの修正でスコアが上がったか／回帰が出たか」を git log で追える。

## Step 7: 収束判定とループ停止

次のいずれかで停止する:

- 評価エージェントの検証点が全て PASS
- ゴールデン一致度が目標値（例: 99%）に達した
- 同じ指摘が 3 ラウンド連続で再発（プロンプト表現を変えても効かない場合は別アプローチへ）
- ユーザーが手動で停止指示

停止したら最終プロンプトと最終出力をコミットし、サマリーを `prompt-tuning/<project-slug>/CONVERGENCE_SUMMARY.md` に書き出す。

## 並列バリアント研究（任意）

複数の異なる方向でプロンプトを並行して試したい場合、3 パターン × N ラウンドで並列研究を行うこともできる。各パターンの担当 subagent を組成し、ラウンド終了時に他パターンの評価を共有させると、知見がクロスポリネーションする。

詳細手順は [references/parallel-research.md](references/parallel-research.md) を参照（必要なときだけ読む）。

## Gotchas

### ゴールデン直接コピーで偽陽性スコア

生成エージェントがゴールデンを読むと、出力はほぼゴールデン同型になり評価が「99% 一致」を返す。**プロンプト単体の再現性は測れていない**。生成 prompt に「絶対に読んではいけないファイル」を明示し、評価レポートに「過学習スキャン結果」を必須項目として組み込む。

### 自分で diff を取る

評価エージェントの自己評価スコアは bias がかかる。最終ラウンドでは必ず手動で `diff <output> <golden>` を取り目視確認する。

### プロンプトの抽象例が出力に転記される

プロンプト内に具体例を書くと、生成エージェントがそのままコピーする（過学習の元）。例は最小限にし、「例」「参考」「機械写し禁止」を併記する。過学習スキャンで毎ラウンド検出する。

### 複数指摘の一括反映で回帰が起きる

「5 個反映 → 3 個 OK 2 個回帰」が起きやすい。原則は 1 指摘 = 1 修正 = 1 検証。

### 評価エージェントが曖昧記述を勝手に補完して PASS を返す

「経路 A でも S7 は 2 枚」と書いても「経路 A なので 1 枚で OK」と補完されることがある。プロンプトに **禁止条項を明示**（「〜の解釈は禁止」）し、評価エージェントの検証ポイントに該当項目を独立観点で入れる。

### スコープ box にメタ制約が漏れる

入力ファイルに講座制作者向けの「テストの話はしない」「実コードは扱わない」のようなメタ制約があると、生成エージェントが出力にそのまま転記する。プロンプトに「メタ制約を box に書かない」絶対禁止を明示。

### カテゴリ概念と具体インスタンスの混同

「定義スライド」を作るとき、題材固有の具体クラス名を取り上げてしまう。本来は **型のレベルの概念** を定義し、具体クラスは後段で初出させる。プロンプトと評価チェックポイントの両方で明示する。

### コミット忘れで履歴が追えない

ラウンドごとに「プロンプト + 出力 + 評価」の 3 点セットでコミットする。

### subagent の spawn コスト

各 spawn にトークンコストがかかる。Agent Teams を使うと約 7 倍（agent-teams-guide 参照）のため、シンプルな単一 subagent 反復が基本。

# 並列バリアント研究（任意）

複数の異なる方向で同じプロンプトを並行して試したい場合の手順。シンプルな単一反復で十分なら使わない。Agent Teams 機能を使うため、トークンコストが約 7 倍になる点に注意（agent-teams-guide スキル参照）。

## 使うべき条件

次の **すべて** を満たすときだけ並列バリアントを採用する:

- 単一プロンプトの反復だけでは収束が遅い、または明らかに局所最適に止まっている
- 改善方向が複数仮説あり、どれが最良か事前に判断できない（例: 構造重視型 / 文体重視型 / カタログ型）
- ユーザーが明示的に並列研究を要求した
- 課金プランが Max 以上（agent-teams-guide の前提条件）

## チームの組成

`agent-teams-guide` を起動して仕様を確認した上で、次の構成で組む:

- `team-lead`: メインセッション。各バリアントの結果を統合する役
- `researcher-A`: バリアント A を担当（例: 構造分解の原則を強化する方向）
- `researcher-B`: バリアント B を担当（例: 文体規約を強化する方向）
- `researcher-C`: バリアント C を担当（例: スライド種別カタログを強化する方向）
- `meta-merger`（任意）: 全バリアントの最良要素を統合するエージェント

各 researcher は独立した出力ファイル系列を持つ:

```
prompts/prompt-A-r1.md, prompt-A-r2.md, ...
prompts/prompt-B-r1.md, prompt-B-r2.md, ...
prompts/prompt-C-r1.md, prompt-C-r2.md, ...
outputs/output-A-r1.md, ...
evals/eval-A-r1.md, ...
```

## ラウンドの進め方

各ラウンドで:

1. 各 researcher が自分のプロンプトを 1 ラウンド分改善（前ラウンドの自己評価 + ピアの評価を反映）
2. 各 researcher が自分のプロンプトでスライドを生成
3. 各 researcher が自己評価
4. 各 researcher は `SendMessage` で他 researcher に「自分が見つけた汎用性向上の鍵」を共有
5. team-lead は進捗を集約

ピア共有ルール:
- 共有する内容は「汎用化の手法」「過学習回避の発見」「構造原則」など、特定バリアントに依存しない知見
- 自分のバリアント固有の表現は共有しない（他バリアントが模倣すると独立性が崩れるため）

## メタ統合

3 バリアントの r4-5 ぐらいで指摘がほぼ収束したら、`meta-merger` を spawn して最良要素を統合する:

- 各バリアントの強みを抽出
- 矛盾点を解消（どれを採るか優先順位を決める）
- 統合プロンプト `prompt-FINAL-rN.md` を作成
- 統合プロンプトで生成・自己評価して、各バリアント単独より良いか確認

## 収束の判定

- 3 バリアント独立に同じスコア（例: 全て 99%）に到達 → 構造的収束
- メタ統合がそれを上回る（例: 99.5%）→ 確定
- ゴールデンとの diff が「レンダリング非影響」「ゴールデン側のバグ修正」のみ → 実質完成

## チームの解散

ループが完了したら必ずクリーンアップ（agent-teams-guide §4）:

1. 全 researcher と meta-merger に `shutdown_request`
2. `teammate_terminated` を全員分受信したら `TeamDelete`
3. 孤立 tmux セッションがあれば `tmux kill-session`

## Gotchas

### バリアント数を増やしすぎない

3 バリアントが一般的な上限。4 以上にすると context bleeding（メンバー間の仮定が混線）が起きやすく、また並列発火コストが急増する。

### 各 researcher が他バリアントを参考にしすぎる

ピア共有を活用しつつも、自分のバリアントの独自性を保つ。「全員が同じプロンプトに収束」してしまうと並列研究の意味がない。

### メタ統合のタイミングが早すぎる

各バリアントが r3 程度のときにメタ統合すると、まだ各バリアント自体が改善余地を残しているため統合の価値が薄い。各バリアントが自己ベスト水準に到達してから統合するのが目安。

### コミットが追えなくなる

3 バリアント × N ラウンドで 30 ファイル以上が動く。各ラウンドの最後に `git add iteration-state/` でまとめてコミットし、メッセージに「ラウンド N 完了、A=X% B=Y% C=Z%」のサマリを書く。

### Agent Teams のクリーンアップ忘れ

ループを途中で中断したとき、Agent Teams のチームが残ったままだと次回のセッションで干渉する。`TeamDelete` を必ず実行する。

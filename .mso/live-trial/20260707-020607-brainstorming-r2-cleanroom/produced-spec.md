# コネクタ kind マッチング設計

## 背景と問題

ブロック接続（歯車・インベントリ・流体）で、意図しない繋がり方が起きる。
例: シャフト端と歯車の歯が直接繋がる。

接続可否は `BlockConnectorComponent.OnPlaceBlock`（`moorestech_server/Assets/Scripts/Game.Block/Component/BlockConnectorComponent.cs:63-112`）の1箇所だけで決まり、判定は実質2つ。

まずは `TTarget` ジェネリックによるドメイン型フィルタ（`:69`）で、gear/inventory/fluid の跨ぎは弾かれる。
次に offset / directions による座標一致（`:76-100`）だけで接続が成立する。

問題はドメイン内にある。歯車もシャフトも両方 `IGearEnergyTransformer`（`VanillaGearTemplate.cs:29` / `VanillaShaftTemplate.cs:27` がどちらも `connectSetting, connectSetting` を渡す）で、`IBlockConnector` が持つのは `connectorGuid` / `offset` / `directions` のみ。歯なのかシャフト端なのかを区別する情報がコードにもスキーマにも無く、座標が噛めば繋がる。これが直接原因。
インベントリも同じ `BlockConnectorComponent<IBlockInventory>` の同じコードパスで、同一の根本原因・同一の修正で解ける。

## ゴール

コネクタに「kind（役割タグ）」を持たせ、適合した kind 同士だけが接続するようにする。
歯車を主対象とし、シャフト端×歯車の歯のような意図しない接続を止める。
加えて、回転軸が直交する歯車同士（歯×歯）が噛み合わない問題も止める（kind は回転不変なので扱えない、別の関心事）。
インベントリ・流体は今回は単一 kind ベースライン（現状維持）とし、同じ仕組みが一律で効く状態にする。

## 非ゴール

後方互換・パフォーマンス最適化・将来拡張性は考慮しない（AGENTS.md）。
インベントリ/流体の kind 細分化は今回行わない（仕組みだけ通す）。
ベベルギア（回転軸が直交する歯車が噛み合う）は作らない。直交軸の歯×歯は常に不接続とする。
別リポジトリの実ゲームデータ（`../moorestech_master/server_v8`）のタクソノミ authoring は本 spec の対象外（mooreseditor 側フォローアップ）。

## 設計方針の決定事項

| 決定 | 内容 | 根拠 |
|---|---|---|
| 適合モデル | 中央の適合ペア表（分散属性ではない） | N対N の関係データは中央正規化テーブルが SSOT |
| kind の識別 | GUID マスタ + foreignKey（enum ではない） | mod・将来の kind 追加をマスタ定義で扱う |
| ペアの対称性 | 無順序・対称（ロード時に (a,b)(b,a) 両方を集合へ） | 適合は相互関係（A が B と繋がるなら B も A と繋がる）。向きの制約は kind ではなく整列軸で別途扱う |
| 既定挙動 | fail-closed（表に無い kind ペアは不接続） | 目的が意図しない接続を無くすこと。安全側に倒す |
| ゲート位置 | `BlockConnectorComponent.OnPlaceBlock` の座標一致直後 | 唯一の接続可否判定箇所。self/target コネクタが揃っている |
| kind 参照層 | 共通抽象 `IBlockConnector` に持たせる | ゲートが `IBlockConnector` だけを見れば済み、ドメイン別キャスト不要で全ドメイン一律 |
| 向きの扱い | kind とは別に「整列軸」ベクトル（BlockDirection で回転する）を持たせ、両方が持つときだけ軸平行を要求 | kind は回転不変で軸の向きを表せない。向きは幾何量なので回転する別フィールドで扱う |

## データモデル

### 1. connectorKinds マスタ（新設）

kind の実体。要素は次の2フィールド。

- `connectorKindGuid`: uuid, autoGenerated
- `name`: string

### 2. IBlockConnector への kind 追加

共通抽象 `IBlockConnector`（`VanillaSchema/blocks.yml` の `globalDefineInterface`）に次を追加し、gear/inventory/fluid の全コネクタが必ず1つ持つ（required）。

- `connectorKindGuid`: uuid, foreignKey → connectorKinds

### 3. connectorCompatibility ペア表（新設）

「kind a と kind b は接続してよい」を1行で表す中央表。

- `a`: uuid, foreignKey → connectorKinds
- `b`: uuid, foreignKey → connectorKinds

foreignKey を3つ追加するため、スキーマ編集後に validate-schema で C# バリデーション追加漏れを確認する。

### 4. IBlockConnector への整列軸追加（向きの関心事）

kind は回転不変なので回転軸の向きを表せない。向きは幾何量として別フィールドで扱う。
共通抽象 `IBlockConnector` に次を追加（optional、省略可）。

- `alignmentAxis`: vector3Int, optional

歯コネクタ（`gearTooth`）はここに回転軸（基本歯車で `[0, 0, 1]`）を設定する。`directions` と同じく BlockDirection で回転する。
`shaftEnd` / `itemPort` / `fluidPort` など向きを問わないコネクタは省略する（省略側は軸チェックをスキップ）。

## ランタイム

### MasterHolder

`MasterHolder` に `ConnectorKindMaster` と `ConnectorCompatibilityMaster` を追加（既存 `ItemMaster` 等と同型）。
後者はロード時に対称ペア集合 `HashSet<(Guid, Guid)>`（(a,b) と (b,a) の両方）を構築し、次を提供する。

- `bool CanConnect(Guid selfKind, Guid targetKind)`

### ゲートの挿入

`BlockConnectorComponent.OnPlaceBlock` の座標一致判定の直後、`isConnect = true` にする前に kind 適合を確認する。

```
if (!MasterHolder.ConnectorCompatibility.CanConnect(
        outputConnector.connector.ConnectorKindGuid,
        target.connector.ConnectorKindGuid)) continue;
```

`selfConnector` / `targetElementConnector` はその場に揃っている（`BlockConnectorComponent.cs:94-96`）ため、ドメイン別キャスト不要で gear/inventory/fluid 全ドメインに一律で効く。
`option`（`GearConnectOption.IsReverse` 等）は接続後の挙動専用で、今回のゲートとは独立。触らない。

### 整列軸ゲート（向きチェック）

kind 適合の直後に、向きの一致を確認する。両コネクタが `alignmentAxis` を持つ場合のみ、**軸が平行**（外積ゼロ、= 同方向または逆方向）であることを要求する。片方でも省略していればスキップ（現状維持）。

```
if (selfAxis.HasValue && targetAxis.HasValue &&
    Vector3Int.Cross(selfAxis.Value, targetAxis.Value) != Vector3Int.zero) continue;
```

`alignmentAxis` は各コネクタの**所属ブロックの BlockDirection で回転**させてから比較する（self は自ブロック、target は相手ブロックの向き）。`directions` と同じ回転で、`BlockConnectorConnectPositionCalculator` がコネクタ座標を計算する際に回転済み軸も併せて算出し、コネクタ情報と一緒に保持する。

これにより次が正しくなる。

- 軸Z歯車 × 軸X歯車（直交、共通垂直軸で隣接）: 歯×歯だが軸が非平行 → 不接続。
- 軸Z歯車 × 軸Z歯車（横並び）: 歯×歯で軸平行 → 接続。
- 軸Z歯車 × 軸Z歯車（Z方向に縦積み）: 歯は向き合わず（歯は ±X,±Y）、代わりに軸(シャフト)コネクタ同士が接続。

### null-directions パスの改修（必須）

現状 `OnPlaceBlock` には kind ゲートを通せない抜け道が1つある。
入力コネクタの `directions == null`（どの向きからでも接続可）だと、`BlockConnectorConnectPositionCalculator.CalculateConnectorToConnectPosList`（`:32` `result.Add(inputConnectorPos, null)`）がコネクタ参照を捨て、`OnPlaceBlock:82-86` が connector 情報なしで無条件 `isConnect = true` にしてしまう。ここを通ると kind チェックが効かず fail-closed が破れる。

対策として `BlockConnectorConnectPositionCalculator` を「directions が null でも connector 参照は常に保持し、方向制約だけを optional にする」形へ小改修し、全パスで必ず kind ゲートを通す。これは fail-closed 成立の必須条件。

## 移行と初期 kind タクソノミ

方針は「現状維持ベースライン → 個別に締める」の2段構え。

まず現状の挙動を再現する最小 kind を各ドメインに割り当て、compat 表に同 kind 行を入れて何も壊れない状態を作る。
その上で、問題のペアだけ kind を分けて締める。

### 歯車（挙げられた不具合を直す確定案）

- kind: `gearTooth`（歯車の歯）、`shaftEnd`（シャフト端・歯車の軸受け）
- compat 行: `{gearTooth, gearTooth}`、`{shaftEnd, shaftEnd}`
- 整列軸: `gearTooth` コネクタに回転軸（基本向きで `[0, 0, 1]`）を設定。`shaftEnd` は省略。
- 結果: 歯×歯、軸×軸だけが繋がり、シャフト端×歯は表に無いため fail-closed で不接続。さらに歯×歯は軸平行時のみ接続。当初の不具合と直交軸の誤接続が両方直る。
- 将来「大歯車×小歯車」等の別 kind 噛み合いが要れば `{bigTooth, smallTooth}` を1行足すだけ（中央表の利点）。

### インベントリ・流体

`itemPort` / `fluidPort` の単一 kind でベースライン（現状維持）。compat 行は `{itemPort, itemPort}`、`{fluidPort, fluidPort}`。
現時点で切りたい具体ペアは無いため細分化しない。将来切りたくなったら kind を分けて compat 行を足すだけで対応できる。

### 移行スコープ

本 spec 実装に含める:

- VanillaSchema（connectorKinds / connectorCompatibility / IBlockConnector への foreignKey + alignmentAxis）
- SourceGenerator 再生成
- MasterHolder（マスタ2つ + CanConnect）
- ゲート挿入（kind + 整列軸）+ null-directions 改修 + 整列軸の回転算出
- リポジトリ内テストmod（`forUnitTest` / `EditModeInPlayingTestMod`）の JSON への kind 付与 + gearTooth への整列軸付与 + compat 表

含めない（分離）:

- 実ゲームデータ `../moorestech_master/server_v8` のタクソノミ authoring（mooreseditor 側フォローアップ）

## クライアント

変更不要。
`GearConnectorView` / `InventoryConnectorView`（`GearConnectorView.cs:25-35` 等）はコネクタの `offset` / `directions` の位置線を描くだけで、接続成立の可否は描いていない。接続はサーバー権威のため、プレビューとサーバーの食い違い（desync）は起きない。
自動生成モデルに `connectorKindGuid` が増えてコンパイルが通ればよく、挙動変更なし。compat による色分けは YAGNI で対象外。

## テスト（サーバー CombinedTest）

- 不具合再現→修正の確定テスト: シャフト端を歯車の歯に隣接配置 → 未接続をアサート。歯車×歯車（歯）→ 接続。シャフト×シャフト → 接続。
- 直交軸テスト: 軸Z歯車と軸X歯車を共通垂直軸で隣接配置（両方 gearTooth が向き合う）→ 軸非平行で未接続。軸Z×軸Z 横並び → 接続。軸Z×軸Z 縦積み → 軸(シャフト)接続になる。
- null-directions 抜け道テスト: `directions == null` の入力コネクタに非適合 kind を隣接 → 未接続（改修を検証）。
- MasterHolder: 新マスタのロード + `CanConnect` の対称性（(a,b)(b,a) 両方 true）。
- ベースライン回帰: 既存の接続系テスト（`GearBeltConveyorTest` / `ChestLogicTest` / `FilterSplitterTest` / `FluidTest` 等）が、テストmodへの単一 kind ベースライン付与後も全てパスすること。

## 自己反証（設計が壊れないことの確認）

- ブロック回転（向き）: kind は回転不変なので回転軸の向きを表せない。これは `alignmentAxis`（BlockDirection で回転する幾何量）と軸平行ゲートで扱う。直交軸の歯×歯は軸非平行で不接続、平行なら接続。← 当初 kind だけで扱おうとして見落とした穴を明示的に塞いだ。
- 多対多（1 kind が複数 kind と接続）: compat 表に行を足すだけで表現できる。
- 同 kind 自己接続（teeth↔teeth）: 表に明示行が必要。全ての正当接続が表に載る＝完全に明示的で SSOT。
- kind GUID が connectorKinds に無い / どの行にも無い: `CanConnect` が false → 不接続（安全側）。
- ブロック破壊（`OnRemoveBlock`）: kind 無関係で従来通り。

## リスク

- 共有 enum ではなく GUID マスタ + foreignKey を採用したため、`globalDefineInterface` への foreignKey 付与と SourceGenerator 生成が想定通り動くかは実装時に検証する（`globalDefineInterface` は本番初使用）。
- テストmod 全コネクタへの kind 付与漏れがあると foreignKey バリデーションで検出される（コンパイル/検証駆動で漏れを機械検出）。

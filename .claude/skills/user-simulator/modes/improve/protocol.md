# improve モード — 漏れ/誤検知の即時改善

シミュレーターの外し（FN: ユーザーの追加指摘 / FP: 適用済み指摘の否定）を、**その場のハンドオフ発行**と
**新セッションでのコピペ1発改善**で回収する。「溜めてから改善」はしない（ユーザーが覚えていられないため）。

## ハンドオフ発行（外しが起きたメインセッションが即座に行う）

1. ユーザーの指摘をまず対象docへ適用する（タスクを止めない）。
2. `modes/improve/pending/<YYYYMMDD-HHmm>-<slug>.md` を発行する:

```markdown
# 改善ハンドオフ: <slug>
- 種別: FN | FP
- 対象doc: <絶対パス>（当時のcommit: <ref>）
- 判事の予測レポート: （全文貼り付け）
- ユーザーの指摘（原文）: 「...」
- 漏れ/誤検知の原因仮説: （メインセッションの一言。知識不足/選択ミス/反証で誤破棄 等）
- 文脈: （このタスクのゴールと、指摘に至る会話の要点3行）
```

3. `modes/improve/misses.md` に採点1行を追記する。
4. ユーザーに1行だけ提示する:
   > 改善ハンドオフ発行。新しいセッションで → `/user-simulator improve <YYYYMMDD-HHmm>-<slug>`

## 改善セッションの手順（`/user-simulator improve <id>` で起動）

1. `pending/<id>.md` と `misses.md` の過去分を読む。**同根分析**: 過去の外しと根本原因が同じか。
   同根なら既存知識の修正・強化で対応（新規追加より優先）。
2. 原因を分類して対処する:
   - 知識不足 → `knowledge/` に追記（既存ファイル優先。新規ファイルはindex.mdに1行トリガー付きで登録）
   - 選択ミス（知識はあったのにロードされず） → index.mdのトリガー記述を直す
   - 反証で誤破棄（FN） / 反証が甘い（FP） → `agents/refuter.md` の基準を直す
   - プロトコル欠陥 → `agents/judge.md` / 各protocol.mdを直す
3. **ゴールデン昇格と再演**: このハンドオフを `goldens/<id>.md` に昇格し、当時のdocに対して判事を再実行して
   「今回の外しが直る」ことを確認する。さらに既存goldensから直近2件を再演し「既存の的中が壊れず・FPが増えない」
   ことを確認する。通らなければ知識の書き方を変えて再試行（通るまで採用しない）。
4. **ADR追記（必須）**: `decisions.md` に1行追加する。
   `| # | <外しの要約> | <どの知識/エージェントをどう変えたか> | <見送った代替案>: <理由> | miss:<id> / golden:<id> | <詳細リンク> |`
   変更した知識ファイル側にも出自ヘッダ（`<!-- decision #N / miss <id> -->`）を付ける。
5. ハンドオフを `applied/<id>.md` へ移動し、末尾に**検討過程まで**追記してコミットする:
   同根分析の結果・検討して見送った修正案とその理由・変更ファイル一覧・ゴールデン再演の結果。
   decisions.md の「詳細」列はこの applied ファイルを指す（改善1件の詳細記録はここが正）。
   複数の外しにまたがる構造的な再設計を行った場合のみ、`decisions/YYYY-MM-DD-<topic>.md` に
   まとめの検討記録を別途書く（前例: `decisions/2026-07-24-v2-redesign.md`）。

## 剪定（FPが続く場合）

同じ知識エントリ由来のFPが2回記録されたら、improveセッションはそのエントリを弱める/削除する。
削除もdecisions.mdに行を残す（追記型: 旧行は消さず「→ #N で剪定」と注記）。

# preanswer モード — 質問の事前予測回答

AskUserQuestion（設計裁定・C型質問）を出す**直前**に毎回通す。目的はユーザーの認知負荷の削減:
ユーザーは「答えを考える」のではなく「予測へ拒否権を行使する」だけで済むようにする。

前段に design-question-triage を必ず通すこと（A型調査・B型原則で消える質問はそもそもここに来ない）。

## メインセッションの手順

1. C型質問リスト（質問文・選択肢・各選択肢のトレードオフ）をscratchpadのファイルに書く。
2. Fable判事を起動（review モードと同じ3行契約で `mode: preanswer`、docにはこのファイルを渡す）。
3. 判事の予測を受け、**確信度で提示形式を変える**:
   - **確信高** → 質問を出さず「前提宣言（拒否権つき）」に降格する。
     `- <決定>: <1行根拠>（シミュレーター予測・確信高。違ったら指摘してください）`
   - **確信中** → AskUserQuestionを出すが、予測選択肢を先頭に置き `（予測: 根拠1行）` を付ける
   - **確信低** → 素の質問として出す（予測を書かない。当てずっぽうの注記はノイズ）
4. ユーザーの実際の回答/拒否権行使と予測を照合し、**必ず** `modes/improve/misses.md` に採点を追記する
   （的中/外れ。損失関数の帳簿はユーザーの頭ではなくここが持つ）。外れが出たら改善ハンドオフを発行する。
5. 承認された予測裁定を対象docのADRに記録するときは、出所を
   `シミュレーター予測→ユーザー承認 YYYY-MM-DD` と書く（ユーザー裁定との偽装混同禁止）。

## 判事の出力契約

```markdown
- Q: <質問1行>
  予測: <選択肢> / 確信: 高|中|低 / 根拠: <過去裁定・原則の実名を1行>
```

確信の基準:
- **高**: 同型の裁定が過去に2回以上ある、または原則表の直接適用（例: YAGNI先送り傾向・冪等選好）
- **中**: 類似裁定が1回、または複数原則からの類推
- **低**: 根拠が「一般的にはこうする」しかない — それは予測ではないので正直に低とする

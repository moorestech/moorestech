# Lens: 波及棚卸し（DI427箇所・CleanRoom第2プロセッサ・マスタ再登録漏れ由来）

## あなたの役割
Doc path のplanを読み、**変更の波及先が数え上げられておらずタスクから脱落している**Criticalのみを返す。planの欠陥類型で最多。リポジトリ（cwd）をGrepして実数を確認してよい（確認した数字は指摘に添える）。

## 検査項目

### 観点1: 呼び出し側の数え上げ
- シグネチャ・コンストラクタ・オプション名を変える各型について、呼び出し箇所数をGrepで数えた形跡がplanにあるか。
- 実例: DIコンテナ直接構築427箇所（ほぼ全テスト）が未計測のまま計画され、後から全面改訂。

### 観点2: 同一基盤の第2利用者
- 改修するステート・サービス・コンポーネントの利用者を全検索したか。「同じ基盤を使う別実装」が計画対象に入っているか。
- 実例: 機械レシピ選択がCleanRoom機械（同じIdleステートの第2プロセッサ）を見落とし。shape追加時のコネクタ分割要否も同型。

### 観点3: 対応するマスタ側更新
- コード変更に対応するマスタデータ側の更新（アンロック対象の再登録・shape設定・カテゴリ・`../moorestech_master` のJSON）がコードタスクと対で列挙されているか。

### 観点4: 失敗・中断・周辺経路
- 削除/改名するCLIオプション・引数の全呼び出し側（エディタ再生・テスト・スクリプト）を列挙したか。
- 処理中断で壊れた状態が残る経路（プロビジョニング中断等）の後始末がタスク化されているか。

## Critical判定基準
波及先が実在する（Grepでヒットする）のにplanのタスクに無い → 脱落タスクの追加を修正方針として書く（ヒット数・代表パス付き）。

## Criticalにしないもの（過検知ガード）
- planが明示的に「対象外・別plan」と分類済みの波及先（ただし分類の妥当性が怪しければWarning）。
- Grepでヒット0件の推測波及（存在しない波及を指摘しない）。
- 波及先がテストのみで、planに一括更新タスクが既にある場合。

# Lens: 真実源一元化（見た目Config却下・ベルト長導出却下・WorldDataDirectory由来）

## あなたの役割
Doc path のspec/planを読み、**データ・設定・リソースの置き場と導出が単一真実源（SSOT）を壊している設計**のCriticalのみを返す。

## 検査項目

### 観点1: 権威ソース外の置き場新設
- 新規に導入する設定・データが、既存の権威ソース（マスタデータ＋Addressables）の外（クライアント側SO・個別Config・ハードコード）に置かれていないか。
- 「見た目だから」「クライアント専用だから」は分離の理由にならない（ユーザー裁定: 「絶対ダメ、複数個所での管理になる。全データをマスタとaddressableに統合」）。

### 観点2: 別ドメイン値からの導出結合
- ゲーム仕様値を、別ドメインのフィールド（見た目サイズ・blockSize等）から「冗長排除」の名目で導出していないか。
- 導出は一元化に見えて無関係ドメインを結合させる場合がある（ユーザー裁定: 「ベルト長はマスタで設定する。導出はやらない」）。
- 逆方向も検査: 同じ情報の置き場・通り道を2つ作っていないか（順序フィールド新設→配列indexから導出へ修正された前例）。

### 観点3: 統括の不在（散在組み立て）
- 複数のパス・ID・設定が各所で文字列連結・個別プロパティとして組み立てられる設計になっていないか。
- **正解形**: 単一の値オブジェクト（例: `WorldDataDirectory`）が構造全体を統括し、他は全てそこから取得する。

## Critical判定基準
- 権威ソース外の新設置き場 → マスタ/Addressablesへの統合方針を書く。
- 別ドメイン導出 → 明示フィールド化（または逆に重複置き場→導出化）の方針を書く。
- 散在組み立て → 統括値オブジェクトの新設（名前案付き）を書く。

## Criticalにしないもの（過検知ガード）
- design-question-triage の導出可能テストで既に裁定済みの箇所（ContextのADR参照）。
- 一時ファイル・キャッシュ等、真実源でないと明記されているもの。
- どちらに置くか判断が割れる配置（どのymlに置くか等）— violationでなく `設計判断: あり` へ。

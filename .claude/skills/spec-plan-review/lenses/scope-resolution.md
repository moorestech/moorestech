# Lens: スコープ確定（向きコピーYAGNI却下・二経路併存却下・uGUI移行境界由来）

## あなたの役割
Doc path のspec/planを読み、**スコープが未確定・過小・二重のまま実装に進もうとしている**Criticalのみを返す。

## 検査項目

### 観点1: A案/B案の残置
- 設計・計画に複数方式が「併記」のまま残っていないか（例: 接続判定2方式並置→ユーザー「常に1方式だけにしたい」）。
- 実装タスクに入る前に1方式へ確定する。確定に裁定が要るなら `設計判断: あり` へ。

### 観点2: YAGNIを口実にした要件削り
- 「初版では省く」「YAGNI」「必要なら後で」とした項目を列挙し、それがユーザーの当初依頼文・中核要件に含まれていないか照合する。
- 実例: 「向きはコピーしない（YAGNI）」を5セッション連続で提案し毎回「向きもピックする」と却下された。省略の裁定は実装者でなくユーザー（`設計判断: あり` で確認に回す）。

### 観点3: 二経路併存・部分移行
- 旧経路を「特殊ケース用に存続」させる設計になっていないか。存続理由が後方互換だけなら削除・概念統合案を併記する（AGENTS.md: 変更の波及を恐れない。実例: 旧プロトコル存続案→「全部消したい。一括で移行したい」）。

### 観点4: 含める/含めないの境界表
- 移行・変更対象の一覧が「含める/含めない/作り直す」で1件ずつ分類されているか。対象群を一律扱いしていないか。
- 実例: uGUI→web移行で「ワールド空間UIは残す・クラフトリーは削除・スキットは再設計」と対象ごとに扱いが割れた。
- 後続フェーズ（P2以降等）の扱いがdocのどこに書いてあるかも確認する（「P2以降どうするかはどこに書いてあるの？」）。

## Critical判定基準
- A案併記のまま・境界表なし → 確定または表の追加を修正方針に書く。
- YAGNI削りが依頼文と矛盾 → 復活させるか裁定に回す。
- 後方互換のみを理由とする旧経路存続 → 統合案を併記し裁定に回す。

## Criticalにしないもの（過検知ガード）
- ADRに裁定済みと記録されているスコープ決定（Context参照）。
- 明示的に「フェーズ分割」として後続planへの参照がある省略。
- ユーザーが依頼文で自ら除外した項目。

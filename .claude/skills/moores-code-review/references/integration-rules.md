# 統合ルール（Step 5 / Step 6 / Step 7 の規約）

決定論スクリプト・レンズ群の生結果をそのままユーザーへ転送しない。以下で1つの統合結果にマージする。

## 1. 実コード検証（最初に必ず行う）

- **レンズのCritical** — サブエージェントが実コードを読んで出しているが、適用前に該当箇所をReadして裏取りする。precedent-alignmentの `[検証済み]` 注記はspot-checkで足りる。
- **決定論チェックのconfirmed** — 裏取り不要。そのまま統合結果に載せる（誤検出はスクリプトのバグとして報告・修正する）。
- 検証で否定された指摘は統合結果から落とす（落とした件数だけ最終報告に残す）。

## 2. 重複排除

- 複数系統（決定論・複数レンズ）が同一問題を指摘したら**1件に統合**し「N系統一致（高確度）」と明記する。
- 複数レンズのCriticalが同じ箇所を指す場合、両方の修正方針を満たす**1つの編集**にまとめる。

## 3. 確定修正の自動適用

検証を通過した指摘のうち、次を**すべて**満たすものは確認を挟まず自動でインライン適用する（デフォルト動作）:
- 具体名（ファイル/クラス/メソッド/フィールド）と修正方針が挙がっている
- 修正の仕方に選択の余地が無い（妥当な実装が事実上1つ）
- 波及判断が不要、または波及先が修正方針に列挙されている

適用時の規則:
- レンズが具体名を挙げたら**その具体名どおりに編集**する。同等処理を別構造で実装し直さない。
- 修正方針が波及（「全呼び出し元に配線」「全JSONに追加」）を含むなら、列挙対象を**すべてReadしてすべてに適用**する。一部だけ適用して「スコープ外」と切らない。
- 未指示のscope-shrink・renameを混ぜない。`Library/ Temp/ obj/ bin/`・`*.meta`・Prefab/シーン等のUnity YAMLは触らない。

## 4. 設計判断の保留

次はここでは適用せず、Step 7のAskUserQuestionへ選択肢付きでまとめて回す:
- 複数の妥当な選択肢がある（型分割の粒度、イベント新設の要否の裁定等）
- スコープに影響する（全JSON更新・アーキテクチャ変更・別PRへの切り出し候補）
- 決定論confirmedのうち修正が設計作業になるもの（200行分割・ディレクトリ再編・partial解消等）

## 5. 検証

- .csを修正したら `uloop compile --project-path ./moorestech_client` でエラー0を確認する。
- 影響範囲のテストがあれば `--filter-type regex` で限定実行する。

# 統合ルール（回収・照合・適用・報告の規約）

決定論スクリプト・moores設計レンズ群・汎用reviewer群・Codex外部監査・Fable全般レビューの生結果を、そのままユーザーへ転送しない。以下で1つの統合結果にマージする。

## 0. 5系統の性質の違い

- **決定論チェック（`deterministic_checks.py`）** — AGENTS.md・moorestech規約の機械判定可能分（partial・try-catch・200行・10ファイル・デフォルト引数・SerializeField命名・master_default_fallback・packet_response_root）の `confirmed`。**検出は正確で裏取り不要**（誤検出はスクリプトのバグ扱い）。ただし修正が設計作業になるものは §3/§4 の判断フローに乗せる。
- **moores設計レンズ（`lenses/`）** — moorestech固有の設計規約（ドメイン境界・サーバー状態同期3点セット・DataStore分離・マスタデータ防御・型構造・前例一致）。実PRレビュー指摘由来。実コード（patch+cwd）を読んで `Critical` のみ返す。
- **汎用reviewer（`reviewers/`）** — 言語横断のコード品質（命名・重複・デッドコード・一方向フロー・Unity規約・React作法等）。全数調査で採用実績のある23観点のみ採録。出力は `Critical` のみ。
- **Codex外部監査** — 別モデルの独立第三者視点。設計妥当性・致命的不具合・リグレッションを俯瞰で見る。**コードを実行しないため行番号・事実主張を誤ることがある**。重大度(Critical/High/Medium/Low)付き。
- **Fable全般レビュー（`generalists/`）** — チェックリスト非依存の俯瞰監査。報告前にRead/Grepで**自己裏取り済み**の契約なので、Codexより軽い照合で採用できる（`[検証済み]` の事実だけspot-check）。

独立した複数機構が同じ問題を指摘したら、それは確度の高いsignal。

## 1. 実コード検証（最初に必ず行う）

- **Codexの各指摘** — 該当ファイルを実際にReadし、主張（行番号・「XがYを参照」等）が正しいか確認。**誤りは破棄、正しいものだけ採用**。
- **レンズ/reviewerのCritical** — サブエージェントが実コードを読んで出しているが、適用前に該当箇所をReadして裏取りする。precedent-alignmentの `[検証済み]` 注記はspot-checkで足りる。
- **FableのCritical** — `[検証済み]` 注記の事実をspot-check（全件再検証は不要）。`未検証` の懸念はCodexと同じ扱いで裏取り。
- **決定論チェックのconfirmed** — 裏取り不要。そのまま統合結果に載せる。
- 検証で否定された指摘は統合結果から落とす（落とした件数だけ最終報告に残す）。

## 2. 重複排除

- 複数系統（決定論・レンズ・reviewer・Codex・Fable）が同一問題を指摘したら **1件に統合**し「N系統一致（高確度）」と明記する。
- 複数系統のCriticalが同じsurfaceを指す場合、両方の修正方針を満たす **1つの編集** にまとめる。片方を無視・優先しない。両立不能な場合のみ、より具体的・詳細な方を採用し、その旨を最終報告に1行記す。

## 3. 確定修正の自動適用

検証を通過した指摘のうち、次を**すべて**満たすものは確認を挟まず自動でインライン適用する（デフォルト動作）:
- 具体名（ファイル/クラス/メソッド/引数名/フィールド名）と修正方針が挙がっている
- 修正の仕方に選択の余地が無い（妥当な実装が事実上1つ）
- 波及判断が不要、または波及先が修正方針に列挙されている

適用時の規則:
- レンズ/reviewerが具体名を挙げたら**その具体名どおりに編集**する。同等処理を別構造で実装し直さない。
- 修正方針が波及（「全呼び出し元に配線」「型変更に追従」「全JSONに追加」）を含むなら、列挙対象を**すべてReadしてすべてに適用**する。一部だけ適用して「スコープ外」と切らない。
- 「集約/一本化/共有/再設計/判別共用体化/新クラス新設/レイヤ抽出」を指示されたら、指示どおり実行する。自分の判断で「最小侵襲」に読み替えて縮小しない。

## 3.5. 単独系統のcosmetic-only修正の扱い（照合して自動適用）

cosmetic/規約整形クラス（未使用引数・未使用フィールドの除去/配線、rename、`#region`再編・メンバー並べ替え、staticアクセス→DI取得、後方互換コード削除、その他**挙動を変えない**整形）は、具体的な修正内容（ファイル/対象/直し方）が示されている限り、単独系統であっても §3 どおり質問せず自動適用する。規約整形をAskUserQuestionへ回してはならない。

誤検知への防御は「質問」ではなく「実コード照合」で行う:
- 修正方針を実コードと照合し、事実誤認（対象が存在しない・既に規約準拠・実際は挙動が変わる）なら**破棄**し、報告に破棄理由を書く。
- 照合を通過したら適用する。単独系統のcosmetic適用は報告に「単独系統cosmetic」と明記する。

**構造変更**（ファイル切り出し・判別共用体化・型分割・レイヤ抽出・ロジック修正）は §3 の波及ルールに従い全件適用対象。

## 4. 設計判断の保留（末尾でAskUserQuestion）

次に該当する指摘は適用せず保留し、報告のあとに **AskUserQuestionでまとめて**方針を仰ぐ:
- 複数の妥当な選択肢がある（型分割の粒度、イベント新設の要否の裁定等）
- スコープに影響する（全JSON更新・アーキテクチャ変更・別PR切り出し）
- レンズ/reviewer/Codexの指摘同士が両立せず設計判断が要る
- CodexのHigh/Mediumで対応方針が一意に定まらないもの
- 決定論confirmedのうち修正が設計作業になるもの（200行分割・partial解消・try-catch再設計等）

cosmetic/規約整形（§3.5）は設計判断ではない。AskUserQuestionに含めず自動適用する。保留0件ならAskUserQuestionはスキップ。

## 5. 安全規則（適用は判断ではなく機械的制約）

- syntax/build/import breakを起こさないよう整合させる。
- 編集・移植コードのデバッグ出力（`Debug.Log`/`print`/`console.log`）は削除する。
- 未指示のscope-shrink/rename、無関係なchurnを混ぜない。
- `dist/ build/ node_modules/ vendor/ Library/ Temp/ obj/ bin/`・lockfile・`*.meta`・Prefab/シーン等のUnity YAMLは触らない。
- .csを修正したら `uloop compile --project-path ./moorestech_client` でエラー0を確認。影響範囲のテストがあれば `--filter-type regex` で限定実行する。

## 6. 偽装の禁止（4カテゴリcontextを書くとき）

- ユーザーのrevert指示を「合意」と誤読しない（revert=現状維持≠agreement）。
- 会話の含意・推論を「合意」として扱わない。合意は発言として存在するか否かの2値。
- 自分の判断を「ユーザー合意」と書き換えない。「今回のスコープ外」等の私的判断は「目指さない」欄に、合意とは書かず記す。
- 規約からの帰結を「合意済み」と書いて代替案を狭めない。

# moorestech設計レンズ・ダイジェスト

実装前・タスクレビュー時のチェックリスト。詳細と前例は `.claude/skills/moores-code-review/lenses/*.md`。
（実PRレビュー指摘 PR978/987/988/996/997/1000 由来）

1. **前例一致** — この形の問題をリポジトリが既にどう解いているか検索したか。前例は機構でなく**役割**で選ぶ。無言の乖離は禁止（乖離するなら理由を明記）。
2. **依存方向** — 汎用基盤（基底コンポーネント・共通サービス・Master・Template）に上位ドメインの語彙・`Func<bool>`述語・デフォルト値を漏らしていないか。判断は具体側、基盤は `SetHoge` でのプッシュ受けのみ。
3. **ポーリング排除** — `Update()` 内で毎tickの同値判定・状態導出をしていないか。UniRxの変化通知Subscribeか、変化を起こす操作直後のプッシュへ。`Update()` は物理進行専用。
4. **サーバー状態3点セット** — 新規サーバー可変状態に「イベントパケット＋初期データ（Handshake/va:get*）＋クライアント購読」が揃っているか。他プロトコル応答から間接導出するApplierは禁止。
5. **DataStore分離** — 読み取り `I*Lookup`（static公開可）と変更 `I*Mutation`（DI注入のみ）に分離したか。staticに変更系を露出しない。
6. **データ防御禁止** — スキーマ `optional: true`・`?? Default`・ローダーでのプリフィルで欠損を吸収していないか。正解は必須化＋`default`定義＋全JSON一括更新（波及を恐れない）。
7. **型で排除** — 「どれか1つが埋まっていたら有効」struct、N択1必須役割のnullable表現をしていないか。判別子＋ペイロード型 / 束ねinterface＋ctor必須へ。既存共通サービスと重複するロジックは委譲へ。
8. **配置規約** — `PacketResponse/` 直下はIPacketResponse実装のみ。DTOは別階層。1ディレクトリ10ファイル・1ファイル200行以内。ディレクトリは依存の下流→上流が見える構成。

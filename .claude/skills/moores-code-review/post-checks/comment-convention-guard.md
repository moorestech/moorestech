# Post-Check: コメント規約ガード（文字数はスクリプト計測済み）

## あなたの役割
2 つの入力を検査する:
1. **文字数候補の検証** — `deterministic_checks.py` が計測した超過候補（JSON の `candidates.comment_length`）について、例外該当を判定し、非該当には短縮案を出す。**文字数を自分で数え直さない**（スクリプトの `count` が正）。
2. **意味的な冗長コメントの検出** — 最終 diff の追加コメント行（`+`）のうち、文字数に関わらず **名前と情報が重複しているだけのコメント**（`isOpen // 開いているか`、`setX // x をセット` 等の自明コメント）を検出する。

## 入力契約（起動 prompt 4 行）
```
Read this : <このファイルの絶対パス>
Candidates : <最終 diff に対する deterministic_checks 出力 JSON の絶対パス>
Patch path : <最終 diff の絶対パス>
User prompt : <4 カテゴリ context の絶対パス>
```

## 1. 文字数候補の検証手順
各候補（`file` / `line` / `comment` / `count` / `kind` / `limit`）について:
- **例外判定** — 次に該当したら**違反にしない**（黙って落とし、件数だけ報告）:
  - 非常に複雑なアルゴリズムの説明。
  - load-bearing な根拠コメント（なぜ必要か・不変条件・順序保証・競合回避。`comment-rationale-guard` が保全する種別）。**根拠コメントを短縮対象に挙げてはならない**。
  - JSDoc/TSDoc 構造・ライセンスヘッダ・外部 API 仕様の引き写し。
- **非該当なら短縮案を出す** — `limit` 以下へ意訳・短縮（「〜を計算する」→「〜算出」等）。`kind: method` はメソッド名以上の情報（副作用・前提・戻り値の意味）だけ残す。`kind: class` は 1 項目 20 文字以下の箇条書き・主要 3 つ程度へ整形。
- **英訳ペア行の追従** — 日本語行を短縮したら直後の English 行も対応して直す案を含める。

## 2. 冗長コメントの検出基準
- コメントの情報が変数・メソッド名と完全重複 → 削除。名前から読めない情報（単位・前提・範囲）が混ざるならそれだけ残す。
- 自明な setter/getter コメント → 削除。

## 自動適用の可否（修正方針にラベルを付ける）
- **機械的**: 名前重複コメント削除、自明 setter コメント削除、語尾を削るだけの単純短縮 — Step 6 の安全規則のもとで自動適用してよい。
- **要判断**: 短縮で情報が落ちる懸念、アルゴリズム説明か根拠コメントかの判別が際どい、箇条書きへの再構成 — 自動適用せず Step 7 の AskUserQuestion へ。

## 出力フォーマット
違反が 1 件でもあれば:
```
Critical: あり

修正方針:
- <ファイル:行>: <現コメント>（N文字/種別）→ <短縮案 or 削除> [機械的|要判断]
- ...

例外として落とした候補: N 件（<種別内訳>）
```
0 件なら:
```
Critical: なし
```

# code-review-orchestration

コード成果物を提示・マージする前に、**外部AI監査**（Codex を第三者監査人として使う）と**多観点コードレビュー**（code-reviewer の並列サブエージェント群）を **同時に** 走らせ、両者の指摘を1つの統合結果にまとめる。

2つのレビューは性質が異なり補完的なので両方かける価値がある:
- **外部監査(Codex)** — 別モデルの独立視点。設計の妥当性・致命的不具合・リグレッションを俯瞰で見る。ただしコードを実行せず行番号を誤ることがある。
- **code-reviewer** — プロジェクト規約に特化した多数のサブエージェントが観点ごとに精査。実コードを読んで判定する。

## 前提条件

- `codex` CLI が PATH にあること（`which codex` で確認）。無ければ外部監査をスキップし code-reviewer のみ実行し、その旨を報告する。
- `external-audit` スキルと `code-reviewer` スキルが利用可能なこと。本スキルはこの2つを内部から起動する。各スキルの内部仕様（Codex の呼び方、code-reviewer のサブエージェント設計）はそれぞれのスキルに委譲する。
- レビュー対象が、このセッションで自分が作業したコミット範囲・staged diff・unstaged diff として特定できること。対象ファイル群を作業実績から確定できる状態にする。

## ワークフロー

### Step 1: 対象と共通コンテキストを準備する

両レビューに **同じ入力** を渡すため、先に1度だけ用意する。

1. **自分の作業範囲を特定** — このセッションで自分が生成・変更した成果物を、コミット済み・未コミットの両方から確定する。会話中の commit SHA、作業開始時点の HEAD、`git log --oneline`、`git status --short`、`git diff --cached`、`git diff` を照合し、レビュー対象の「コミット範囲」と「未コミット差分」を分けて記録する。単一コミットとは限らないし、HEAD だけとも限らない。
   - コミット済み成果物がある場合: 対象コミット SHA 群と、それらを表す最小範囲（例: 単一コミットなら `<sha>^..<sha>`、連続コミットなら `<first>^..<last>`）を記録する。
   - 未コミット成果物がある場合: staged / unstaged を分けて対象に含める。外部監査プロンプトには `git diff --cached` と `git diff` も確認対象として明記する。
   - 対象ファイル一覧は、確定したコミット範囲・staged diff・unstaged diff から作る。自分がこのセッションで触っていない既存ブランチ差分を混ぜない。
   - ユーザーがレビュー範囲を明示した場合は、その指定を優先する。
2. **4カテゴリの共通コンテキストを書く** — これは code-reviewer の必須入力であり、外部監査プロンプトにも同じ内容を載せる。会話履歴から以下を逐語または自分の判断として埋める（埋め忘れると code-reviewer が false-positive Critical を量産する）:
   - **目指す（ゴール）** — この変更で達成したい挙動。
   - **目指さない（非目標）** — 意図的にスコープ外にした項目。ユーザーが選択・発言したものは合意として、自分の判断のものは「自分の判断として」と明記（「ユーザー合意済み」と偽装しない）。
   - **許容するトレードオフ** — 合意済みの妥協。
   - **尊重すべき制約** — プロジェクト規約・既存アーキテクチャ・ハード制約。

### Step 2: 外部監査(Codex)をバックグラウンドで起動する

Codex は数分かかるので **先にバックグラウンド起動** し、待っている間に Step 3 を走らせる。これが「同時実行」の実体。

1. 監査プロンプトを一時ファイル（`/tmp/code-review-audit-<timestamp>.md`、リポジトリ内には置かない）に書く。プロンプトには必ず含める:
   - レビュー対象として Step 1 で確定したコミット範囲と未コミット差分（Codex は自分でリポジトリを読むので、確認すべき `git diff` コマンドを明示する）
   - 背景（仕様・計画ドキュメントのパス、達成ゴール）
   - **明示的なレビュー観点** — 曖昧なプロンプトは曖昧な結果しか返さない。デフォルトは3観点: (1) アーキテクチャ的不整合・既存パターンとの乖離 (2) 設計の妥当性・将来の設計上の懸念 (3) 致命的不具合・エンバグ・リグレッション。ユーザーが観点を指定したらそれを使う。
   - Step 1 の4カテゴリコンテキスト
   - 出力形式の指定（観点ごとに 重大度/該当ファイル:行/問題/推奨対応、問題なしなら明記、憶測でなく実コード根拠を示す）

   プロンプト本文のテンプレート（`{...}` を埋めて使う。観点はユーザー指定があれば差し替える）:
   ```markdown
   あなたは {リポジトリ名} の外部監査者です。「{成果物名}」の実装を監査してください。
   レビュー対象は、このセッションで私が作業した成果物だけです。以下を確認してください。
   - コミット済み差分: {該当する場合は `git diff <commit-range>`。無ければ「なし」}
   - staged 差分: {該当する場合は `git diff --cached`。無ければ「なし」}
   - unstaged 差分: {該当する場合は `git diff`。無ければ「なし」}
   - 対象ファイル: {Step 1 で確定した対象ファイル一覧}

   ## 背景
   - 設計仕様 — {仕様ドキュメントのパス、無ければ省略}
   - 達成ゴール — {このレビューで完成しているべき挙動}

   ## 監査の観点（以下を必ず個別に評価）
   ### 1. アーキテクチャ的不整合・既存コードパターンとの乖離
   ### 2. 今後の設計上の懸念・設計の妥当性
   ### 3. 致命的な不具合・エンバグ・リグレッション
   {ユーザーが観点を指定した場合はそれに差し替える。各観点に、対象成果物に即した着眼点を2〜4行で具体的に書く}

   ## 目指す / 目指さない / 許容するトレードオフ / 尊重すべき制約
   {Step 1 の4カテゴリをそのまま貼る}

   ## 出力形式
   観点ごとに、見つかった指摘を「重大度(Critical/High/Medium/Low)・該当ファイル:行・
   問題・推奨対応」の形で列挙。問題がなければ「問題なし」と明記。
   憶測ではなく実コードの根拠を示すこと。{回答言語}で回答。
   ```

2. バックグラウンドで起動する:
   ```bash
   codex exec --sandbox read-only --skip-git-repo-check - < /tmp/code-review-audit-<timestamp>.md
   ```
   Bash ツールの `run_in_background: true` で起動し、シェルIDを控える。

### Step 3: code-reviewer を起動する（Codex 実行中に並行）

Codex がバックグラウンドで走っている間に `code-reviewer` スキルを起動する。code-reviewer は `agent/` 配下の全サブエージェントを1メッセージ内で並列発火する設計なので、これ自体が並列レビューになる。Step 1 の4カテゴリコンテキストをそのまま渡す。

code-reviewer のサブエージェント群と Codex が同時に走っている状態が、本スキルの「同時実行」の到達点。

### Step 4: 両方の結果を回収する

- code-reviewer の全サブエージェントの返却を受け取る。
- Step 2 のバックグラウンド Codex の出力を回収する（まだ終わっていなければ完了を待つ）。
- 一時プロンプトファイルを削除する。

### Step 5: 統合する（本スキルの中核）

2つの生結果をそのままユーザーに転送しない。以下のルールでマージする。

1. **外部監査(Codex)の指摘は実コードで検証する** — Codex はコードを実行せず、行番号や事実（「X が Y を参照している」等）を誤ることがある。各指摘について該当ファイルを実際に Read し、主張が正しいか確認する。**誤りは破棄、正しいものだけ採用する。** code-reviewer の指摘はサブエージェントが実コードを読んで出しているが、Critical は同様に裏取りする。
2. **重複排除** — 外部監査と code-reviewer が同じ問題を指摘していたら1件に統合し、「両レビューが一致」と明記する（独立した2機構の一致は確度が高い signal）。
3. **重大度で処理を決める**:
   - **Critical** — 提示前に必ず対応。インラインで修正できるものは修正する。設計判断を要するもの（複数の妥当な選択肢がある、スコープに影響する）は `AskUserQuestion` でユーザーに方針を仰ぐ。
   - **Warning** — 明確な理由がない限り修正する。修正しない場合、理由が「尊重すべき制約」や合意済みトレードオフに基づくことを確認する（私的判断を「合意」と偽装しない）。
   - **Info** — 判断次第。将来フェーズに申し送るものは記録する。
4. **検証** — コードを修正したら、プロジェクトのビルド/テストを実行して通ることを確認してからコミットする。

### Step 6: 統合結果を提示する

1つの統合ビューで提示する。最低限:
- Critical / Warning / Info の件数サマリ
- 各指摘について「どのレビューが出したか（外部監査 / code-reviewer / 両方）」「対応（修正済み / 非修正＋理由 / ユーザー判断待ち）」
- 適用した修正と、検証（ビルド・テスト）結果
- 意図的に非修正とした指摘は理由を一言添える

## Gotchas

- **`codex exec` のフラグ順序** — `--sandbox` `--skip-git-repo-check` 等のオプションはサブコマンド（`resume` 等）より **前** に置く。`codex exec resume --sandbox ...` はフラグが拒否される。新規実行なら `codex exec --sandbox read-only --skip-git-repo-check - < file`。
- **監査プロンプトは一時ファイルに、リポジトリ外に置く** — `/tmp` 等に書き、Step 4 で削除する。リポジトリ内に置くと誤ってコミットされる。
- **Codex の指摘を生のまま信用しない** — Codex はコードを実行しない。行番号・「A が B を参照」などの事実主張が誤ることがある。Step 5-1 の実コード検証は必須。逆に、Codex が正しく重大バグを見つけることもある（実行せずとも設計レベルの欠陥は検出できる）ので軽視もしない。
- **4カテゴリコンテキストを埋めないと code-reviewer が誤検知する** — code-reviewer は空コンテキストを「合意なし」と解釈して既定 Critical を出す。Step 1 で必ず4つ埋める。本スキルは同じコンテキストを両レビューに渡すので、ここで手を抜くと両方が荒れる。
- **「同時」の実体はバックグラウンド起動** — Codex を `run_in_background` で先に投げ、その完了を待たずに code-reviewer を起動する。逐次に走らせると単に遅いだけになる。
- **codex が無い環境** — `which codex` が失敗したら外部監査はスキップし、code-reviewer のみ実行する。スキップした事実を Step 6 の報告に明記する（黙ってシングルレビューにしない）。
- **観点を明示しないと外部監査が浅くなる** — Codex プロンプトに具体的なレビュー観点を列挙する。「レビューして」だけでは表層的な指摘しか返らない。

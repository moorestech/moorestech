# code-reviewer

抽象的なコードレビュースキル。レビュー観点は `agent/` 配下のサブエージェントテンプレートとして保持する。SKILL.md はオーケストレーター: 成果物を渡して全サブエージェントを並列起動し、返ってきた指摘を反映してからユーザーに提示する。

## 中核ルール

1. **コード・型定義・データスキーマ・API契約・設計成果物を提示する直前に、必ず本スキルを通す**
2. **メインエージェントはどのサブエージェントを使うか選ばない**。毎回 `agent/` 配下の全テンプレートを並列で起動する。対象外かどうかの判定は各サブエージェントに任せる
3. ユーザーからの指摘は **新しいレビュー観点を追加すべきシグナル** として扱う

メインが選ばない理由: 「このレビューは多分要らない」という判断ミスで観点が抜ける事故を防ぐため。選ばないことを仕組みで保証する。

## プロセス

### Step 1: 全サブエージェントを並列起動する

**メインは `agent/*.md` を Read しない。** 下の「Bundled agents」セクションがカタログであり、そこに並んでいる全エージェントを毎回並列で発火する。テンプレート本文を読むのはサブエージェント自身の責任。

#### Step 1.0: コンテキスト準備チェックリスト（必須・発火前に完了する）

**これを怠ると false-positive Critical を量産する。** サブエージェントはコンテキストを「目指す / 目指さない / 許容するトレードオフ / 尊重すべき制約」の 4 カテゴリに分解して読むよう訓練されている。メインは発火前に **4 つ全てを prompt に記述済みか** を確認する。

| カテゴリ | 何を書くか | 埋まっていないとどうなるか |
|---|---|---|
| **目指す（ゴール）** | 今回の変更で達成したい挙動・機能。ユーザーが最初に依頼した内容を1〜3行で要約 | サブエージェントが「何が成功か」を判定できず、過剰にベストプラクティスを持ち込む |
| **目指さない（非目標）** | 今回のスコープから意図的に外した項目（例: 「UI表示中の自動更新はしない」「push ゲートの根本修正はしない」） | **非目標を Critical として再フラグされる**（二重指摘） |
| **許容するトレードオフ** | ユーザーと合意済みの妥協（例: 「症状パッチとして採用、根本は別issue化」「static singleton パターンを踏襲、DI化はしない」） | **トレードオフを欠陥として指摘される**（合意済みなのに） |
| **尊重すべき制約** | プロジェクト規約・既存アーキテクチャ・ハード制約（例: try-catch禁止、後方互換ハック禁止、既存 static singleton パターン） | 規約違反の提案が返ってくる |

**原則: ユーザーとの会話の中で明示・暗黙に合意したものは、必ずコンテキストとしてサブエージェントに渡す。** 「自明だから省略」は禁止。サブエージェントは会話履歴を見られない。

**ただし埋め忘れに対するセーフティネットあり**: Pattern A/B/C/D に該当するキーワード（`GearNetworkDatastore` / `GetCancellationTokenOnDestroy` / `isChanged` 単一サイトガード / `OnEnter` で 1 回 fetch 等）が成果物に出てくる場合、subagent 側が `subagent-common-rules.md` Step 0.36 で **空カテゴリに verdict block を自動採用** する。だからといってここを埋めない理由にはならない（埋めたほうが追加証拠の持ち込みもできるので最強）。

非目標・トレードオフが 1 つも無いケースは稀。見つからないなら、コンテキスト準備が不十分である可能性を疑い、会話履歴を再スキャンする。

#### コンテキスト 4 カテゴリのコピペテンプレ（T6 追加・C/D 再発防止）

**メインがコンテキストを毎回ゼロから書こうとして埋め忘れるのが C/D 再発の構造的原因**（ユーザー T6 直筆: 「subagent 作成時のコンテキスト受け渡しの問題です」）。以下のテンプレを **prompt の 4 カテゴリ部分にコピペし、不要行を削るだけ** で済むようにする。会話履歴に該当発言があるなら逐語、無いなら自分の判断として（「ユーザー合意済み」と書かない）記述する:

```
## 目指す（ゴール）
- {今回の変更で達成したい挙動を 1〜3 行}

## 目指さない（非目標）
- 【UI 系で頻出】UI 表示中の自動更新 / 再 fetch（OnEnter で 1 回取得して以降は固定する仕様）  ← Pattern D
- 【症状パッチ採用時】push ゲート (例: SimpleGearService.isChanged) の根本欠陥修正は今回スコープ外（別 issue 化）  ← Pattern C
- 【既存 static 維持時】既存の static singleton パターンの全廃 / DI への全面移行  ← Pattern A 関連

## 許容するトレードオフ
- 【UniTask race 系で頻出】post-await の ct.ThrowIfCancellationRequested() では Unity Destroy 遅延実行のため race を閉じない仕様（unitask-cancellation-semantics.md 参照）。await 後がフィールド代入のみで可視副作用ゼロなら Critical 扱いにしない  ← Pattern B
- 【症状パッチ採用時】単一呼び出しサイトのローカルガード追加で症状を抑える。同欠陥が他経路でも発火する可能性は受容  ← Pattern C 補強

## 尊重すべき制約
- 【既存 static singleton + DI 受け取り時】host (例: GearNetworkDatastore) は static singleton 運用、注入される依存も singleton 寿命。両者の寿命は揃っており、DI は単なる解決手段の違い。「ライフサイクル不整合」「static と DI の混在」を Critical 化しない  ← Pattern A
- 【プロジェクト規約】try-catch 禁止 / 後方互換ハック禁止 / 未使用コード削除
```

**運用**: 該当しない行は削除する。該当するが会話で明示されていないなら「（自分の判断として）」と前置きして書く（**「ユーザー合意済み」とは書かない**。逆方向のミス: 合意の偽装に該当する）。

#### Step 1.05: 再レビュー前事前検証チェックリスト（T9 追加・最重要・Step 1.1 発火前に実行）

T9 で 7 周連続同 4 パターン再発したため追加。コピペテンプレを埋めた直後・Agent 並列発火の直前に、メインが以下を全て yes にできない限り発火を保留する:

- [ ] **Pattern A 該当キーワード**（`GearNetworkDatastore` / `Initialize(I*Protocol)` / `private static * _instance` を持つクラスが DI で依存を受け取る形）が成果物中に**存在しない** OR 「尊重すべき制約」欄に Step 0.35 Pattern A verdict block の対応行を**逐語コピペ済み**
- [ ] **Pattern B 該当キーワード**（`await` + `GetCancellationTokenOnDestroy` / `UniTaskCompletionSource` + Unity Destroy）が成果物中に**存在しない** OR 「許容するトレードオフ」欄に Step 0.35 Pattern B verdict block の対応行 + 「post-await ガードでは閉じない仕様確認済み（unitask-cancellation-semantics.md）」を**逐語コピペ済み**
- [ ] **Pattern C 該当キーワード**（`SimpleGearService.isChanged` / 単一サイト症状パッチ / push ゲート）が成果物中に**存在しない** OR 「目指さない」欄に Step 0.35 Pattern C verdict block の対応行を**逐語コピペ済み**
- [ ] **Pattern D 該当キーワード**（`OnEnter` / `OnAttach` で 1 回 fetch、UI 表示中の再 fetch なし）が成果物中に**存在しない** OR 「目指さない」欄に Step 0.35 Pattern D verdict block の対応行を**逐語コピペ済み**
- [ ] 上記 4 パターンが 1 つでも該当している場合、`unitask-cancellation-semantics.md` Frame N シーケンス（Pattern B 該当時のみ）と Step 0.35 verdict block 全体の所在を prompt 末尾に「## 引用元（subagent が逐語引用すべき一次資料）」として明示している

**運用ルール**:

1. 該当キーワードがあるのに verdict 逐語コピペが無い場合、**発火しない**。Step 1.0 のコピペテンプレに戻って verdict block の対応行を逐語で貼る
2. Step 0.36 の自動採用は safety net であり、メインが明示貼りをサボる理由にしない。明示貼りが確定行動、自動採用は補助
3. このチェックを通過した後でも 4 パターンの Critical/Warning が返ってきた場合、それは subagent 側のルール失効として `update-skill` で再強化する

**T9 の根拠**: T2/T4/T6/T7/T8 と 5 回連続で「再発防止」を入れたが、メイン側の検証ゲートが無いと埋め忘れに気付かないまま発火してしまう構造が残っていた。Step 1.05 はメインの埋め忘れを発火前に物理的に検出する最後のゲート。

#### Step 1.1: 全エージェント並列発火

1. 下の Bundled agents セクションを見て、全エントリを列挙する（Glob/Read は不要）
2. **1メッセージ内で** 各エントリに対して並列に Agent ツールを呼ぶ。カタログ行に `(sonnet)` `(haiku)` `(opus)` のモデル指定があれば Agent 呼び出しの `model` パラメータに渡す（指定が無ければ省略）。prompt にはテンプレートへの **絶対パス** と成果物・4カテゴリコンテキストだけを渡す:
   ```javascript
   Agent({
     description: "3〜5語の説明",
     subagent_type: "general-purpose",
     model: "sonnet",  // ← カタログ行で `(sonnet)` 等の指定があれば設定
     prompt: `まず以下の2ファイルを Read で読み、共通ルール → 個別テンプレートの順に従ってください。

共通ルール: /absolute/path/to/.agents/skills/code-reviewer/references/subagent-common-rules.md
個別テンプレート: /absolute/path/to/.agents/skills/code-reviewer/agent/{name}.md

---

## レビュー対象

${theArtifact}

## 目指す（ゴール）

${whatWeWantToAchieve}

## 目指さない（非目標）

${explicitNonGoals}

## 許容するトレードオフ

${acceptedTradeoffs}

## 尊重すべき制約

${anyHardConstraints}`
   })
   ```
3. 順次呼び出しは禁止。全部を1メッセージにまとめる

この設計により、メインのコンテキストはカタログの1行サマリだけで済み、テンプレート本文はサブエージェント側でのみ展開される。

### Step 2: 各サブエージェントは自己判定で早期終了する

各サブエージェントの本文先頭には **Applicability check** セクションが必須で、以下の契約で動く:

- 渡された成果物が自分のスコープ外なら、共通ルールの出力形式で即座に早期終了する
- スコープ内なら通常のレビューを実行する

これによりメインエージェントは「どれを呼ぶか」を考えなくてよい。全部投げて、必要なものだけが実際に働く。

### Step 3: 指摘を適用する

返ってきたパンチリストを深刻度順に処理:

- **Critical**: 提示前に必ずインラインで修正
- **Warning**: 明確な理由がない限り修正
- **Info**: 判断次第
- **NOT_APPLICABLE マーカー**: 無視

**生のレビュー結果をユーザーに転送しない。** 自分で修正を適用し、改善された成果物を提示する。意図的に適用しなかった指摘があれば、提示時に一言だけ触れる。

## 新しい観点の追加 / 既存観点の更新

ユーザーからレビュー指摘を受けたら、それを永続化するのがこのスキルの本質。**この作業（サブエージェントの追加・更新・テスト・Bundled agents カタログ追記・事例履歴の記録）は専用スキル `code-review-subagent-updater` に切り出されている。** レビュー指摘を受けて観点を恒久化するときは同スキルを起動すること。根本原因分析・新規/更新判断・テンプレート構造・テスト手順・過去の全事例履歴（False-negative / False-positive / Pattern A–D 再発ログ / 合意の偽装）はすべて同スキル配下にある。

新規エージェントを追加した場合は、必ず下の Bundled agents カタログに 1 行追記する。このカタログが Step 1 の発火元であり、ここに載っていないエージェントは起動されない。

## Bundled agents

Step 1 はこのリストを読んで全エントリを並列発火する。各エントリには `agent/` からの相対パスと 1 行の概要のみを書く(本文を展開しない)。

- `agent/schema-design-reviewer.md` — 型定義・データスキーマ・API契約向け。直交性違反(1フィールドに2つの意味)、判別共用体の欠落、永続セーブ DTO を型別フィラー列へフラット化(variant 増加に耐えない)、スコープ誤り、位置情報の冗長保持、汎用レスポンスにドメイン特化 convenience を載せる抽象度違反。
- `agent/architecture-lifecycle-reviewer.md` — アーキテクチャ設計・画面構成・ルーティング/モーダル合成・新規クラスのディレクトリ/namespace/アセンブリ配置向け。ライフサイクル要件の列挙漏れ、クロスセクション不変条件の伝搬忘れ、Web SPA反射のドメインミスマッチ、共有リソースの暗黙破壊経路、修正時の過剰スイング、責務と不一致なモジュール配置、横断的関心事(接続・認証等)の feature アセンブリ同梱、汎用 state/API へのドメイン特化責務混入、protocol 内 nested domain adapter / mutation semantics。Common/ への移動提案は「定数/実質定数（状態なし・副作用なし・入力に対して常に同じ値）」に限定し、runtime クラスや contract interface を Common 化する提案は禁止。
- `agent/bug-fix-intent-reviewer.md` — バグ修正コード変更・修正案向け。設計意図の未読解(static/singleton/service locator を触る修正でソースを読んだ形跡がない)、症状パッチを根本修正と偽った提示、却下後の反対極への振り切り、選択肢羅列による思考停止、blast radius の過剰。
- `agent/dead-code-and-scope-reviewer.md` — コード変更後の残骸検出。デッドコード(テスト専用public メソッド)、参照数減少によるスコープ縮小機会(ローカル関数化)、インターフェース実装への誤った制限ラベル、不要な using/フィールド。UniRx 系 IObservable/Subject ファサードの読み取り専用プロパティは除外。
- `agent/region-internal-reviewer.md` — C# の `#region Internal` 規約違反検出と適用機会検出。クラス直下で private メソッドを囲う使い方、`#endregion` 後ろにコードが続く書き方、ローカル関数以外を詰め込む使い方、別名でアンチパターンを隠すケース、40 行超メソッド/バリデーション群/参照1箇所の private メソッドでローカル関数化の機会を逃しているケース。
- `agent/unitask-pattern-reviewer.md` — Unity + UniTask / UniRx の非同期・購読パターン向け。`Task.Run(async ...)` の fire-and-forget、bool フラグによる one-shot 購読管理、`.Wait()` / `.Result` のメインスレッドブロック、エディタ専用に同期版コピペを用意するパターンを検出し `async UniTask + .Forget()` + `UniTask.SwitchToTaskPool()` + IDisposable 保持への置換を提案。
- `agent/unity-convention-reviewer.md` — Unity プロジェクト固有規約(AGENTS.md 準拠)向け。`#if UNITY_EDITOR` のファイル末尾配置違反、エディタ/プロダクションのコピペ分離（統一設計原則違反）、`#if UNITY_EDITOR_*` で囲まれた public メソッドのプロダクション側黙って no-op、日英 2 行セットコメントの折り返し違反、コードを言い換えただけの自明なコメント、集約 null ガードで済むところに冗長ガードを置いているケース、Prefab/Scene 前提 UI 部品の動的生成、SerializeField 参照漏れ・配置ルール違反を検出。
- `agent/comparison-operator-reviewer.md` (sonnet) — C# の二項比較演算子の向き統一規約。`>` `>=` の使用を検出して `<` `<=` への書き換えを提案する。「左から右に小さい値→大きい値」の数直線通りの向きに揃えることで grep / 視線移動コストを下げる。ジェネリクス・ラムダ `=>`・シフト `>>` は除外。diff の追加/変更行のみを対象とし、既存ファイルの未触行は Info に留める。
- `agent/skill-proposal-reviewer.md` (sonnet) — スキル化提案（create-skill / skill-creator のワークフロー出力）が L1（タスク特化）/ L2（中間）/ L3（汎用）の 3 抽象度で並列提示されているかを検証。単一抽象度提案、ラベル不統一、3 案の抽象度縮退、各レベル 4 項目（name / scope / description / triggers）の欠落、おすすめ理由の薄さを検出。スキル提案以外の成果物（実装済み SKILL.md、コード変更）はスコープ外として早期終了。

## このスキルが存在する理由

ユーザーがデータモデルや型定義について「X するな」と指摘するたびに、それは **レビュー観点が欠けていた証拠**。このスキルの目的は、それらの観点を蓄積して **同じ指摘を二度と受けないようにする** こと。指摘を受けたら、次のアクションには必ずこのスキル配下のサブエージェントを追加・更新する作業を含める。

## メインエージェント側のミス類型（再発防止）

サブエージェントが false-positive Critical を返す原因の大半は、**メインエージェント（このスキルを呼ぶエージェント自身）のコンテキスト受け渡し不足**である。以下はレビュー失敗の実例と対策:

- **症状パッチ / 根本修正の区別が Critical 化された**: メインがトレードオフを「症状パッチ採用、根本は別issue化（ユーザー合意済み）」と明記していなかった → Step 1.0 の「許容するトレードオフ」欄に書く
- **UI open 時 1 回 fetch を不整合として Critical 化された**: メインが「UI表示中の自動更新は非目標（ユーザー合意済み）」を明記していなかった → 「目指さない」欄に書く
- **既存 static singleton の DI 化が提案された**: メインが「既存の static singleton パターンを踏襲する（プロジェクト規約）」を明記していなかった → 「尊重すべき制約」欄に書く
- **static singleton 内の DI 受け取りを「ライフサイクル不整合」と Critical 化された（criterion 6 の鏡像）**: 受け取り側も注入元も singleton 寿命なら不整合は無い。メインが「ホスト/依存ともに singleton 寿命で揃えている」を明記していないと、構文だけ見て Critical 化される → 「尊重すべき制約」欄に「ホストは static singleton、注入される依存も singleton 登録（両者の寿命は揃っており、DI は単なる解決手段）」と書く
- **UniTask の `await` 後 `ThrowIfCancellationRequested` を Critical で要求された**: メインが「Unity の `Destroy` は遅延実行のため post-await ガードでは race を閉じない」「await 後はフィールド代入のみで可視副作用無し」を明記していないと、`await` + `GetCancellationTokenOnDestroy` を見ただけで反射的に Critical 化される → 「許容するトレードオフ」または「尊重すべき制約」に「post-await ガード単体では race を閉じないことは仕様確認済み（unitask-cancellation-semantics.md）。同期 Cancel 可能な CTS が必要なら別途検討」と書く

ユーザーから「そのレビュー指摘は不適切」と言われたときは、まず **メインのコンテキスト渡しに抜けが無かったか** を自己点検する。サブエージェントのルール不備だけが原因とは限らない。そのうえで、サブエージェント側にも「コンテキスト欄が埋まっているときはそれを尊重せよ」というルールがあるかを確認し、無ければ追加する。

### 逆方向のミス: 合意の偽装（fabricated agreements）

サブエージェント側のルールを強化した直後、メインが過剰補正で「ユーザーが合意したこと以外も合意済みトレードオフとして書く」誤りを起こしやすい。典型的には:

- **ユーザーの revert 指示を「合意」と誤読する**: ユーザーが「レビュー指摘を取り下げて元に戻して」と指示した場合、それは「今回のレビューサイクルで適用しない」であって「その設計判断でよいと合意した」ではない。Revert = 現状維持 ≠ Agreement
- **会話の含意・推論を「合意」として扱う**: ユーザーが明示的に発言していない項目を「文脈から合意済みと推定した」として非目標欄に書かない。合意は発言として存在するか否かの 2 値で判定する
- **自分の判断を「ユーザー合意」と書き換える**: 「今回のスコープ外」「blast radius を広げない」等の私的判断は、自分の責任として「目指さない」欄に書く。**「ユーザー合意済み」という文言は使わない**
- **私的判断口調を「許容するトレードオフ」に書いて subagent の指摘を降格させる**: メインが「コメントが明確な責務を持つため private メソッドとして残す方針」のような私的判断を「許容するトレードオフ」欄に書くと、subagent が Info の指摘を「合意済み方針を尊重」として降格させてしまう。**私的判断口調の典型例（書いてはいけないパターン）**: 「○○として残す方針」「○○はしない」「現状維持で問題なし」「○○の必要なし」「コメントが意図を明示しているため○○」「責務が明確なので○○」。これらは「目指さない」欄か、何も書かずに subagent に判定させるべき。**書いてよい形**: ユーザー直筆引用「コードはまだ変えない」「private のままで進める」「ローカル関数化は次回」など、発言として存在するもののみ。
- **規約からの帰結口調を「許容するトレードオフ」に書いて代替案を狭める**: メインが「○○禁止規約に従うためコードで AddComponent」のように規約解釈の中間結論を勝手に「合意済み」と書き換えると、subagent が他の代替案を検討する機会を物理的に奪う。多くの場合、規約は禁止行為のみを列挙しており複数の代替案がある。**規約からの帰結口調の典型例（書いてはいけないパターン）**: 「○○禁止規約に従うため○○」「○○できないので○○するしかない」「規約上○○できないので○○」「○○禁止のため○○を採用」。**書いてよい形**: ユーザー直筆引用「動的付与で OK」「RequireComponent は今回使わない」など、ユーザーが代替案を比較した上で選択したことが発言として存在する場合のみ。

上記 2 件の具体的な事例（2026-05-15 region-internal / 動的 AddComponent）と、過去の全偽装事例・Pattern A–D 再発ログは `code-review-subagent-updater` スキルの `references/case-history.md` に保存されている。

**判定チェック（コンテキスト準備時の必須ステップ）:**

コンテキスト 4 カテゴリの各行に書く前に、以下のどれに該当するかを自問する:

1. **ユーザーが会話内で明示的に発言した** → 合意として書いてよい。可能なら発言の要点を短く引用する
2. **ユーザーが選択肢として選んだ（AskUserQuestion 等）** → 合意として書いてよい
3. **自分の判断で今回スコープに入れない** → 「目指さない（今回のスコープ外）」として、**合意済みとは書かない**
4. **前回のレビューで却下したが、ユーザーは再検討していない** → 合意として書かない。サブエージェントに通常通りレビューさせ、必要ならまた Critical が出てくる
5. **規約・既存設計を根拠にして代替案を狭めようとしている** → AGENTS.md 等の該当行を **逐語で読み返し**、その規約は本当にその代替案を強制しているかを自問する。多くの場合、規約は禁止行為のみを列挙しており複数の代替案がある。規約からの帰結を「合意済み」と書かず、代替案検討を subagent に委ねる

ユーザーから「その項目は合意していない」「そのCriticalはそのまま出てほしかった」と指摘されたときは、上記チェックを自己検証し、どの行を偽装していたかを正直に列挙する。偽装 → 削除 → 再レビュー の順で復旧する。

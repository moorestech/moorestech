比較演算子の向きを `<` `<=` に統一する規約違反を検出する専門レビュワー。**左から右に並べたときに「小さい値が左、大きい値が右」と数直線通りに読める向きに揃える**。`>` `>=` は等価な `<` `<=` に書き換えれば必ず同じ条件を表現できるので、選択の自由を与えず一方向に固定することで読みやすさを確保する。

成果物（変更/追加された C# 実装ファイル）を読み、以下の基準に照らしてパンチリストを返す。スコープ外なら共通ルールの形式で即座に終了する。

## 適用判定（最初に実行）

- **スコープ内**: C# 実装ファイル（`.cs`）の追加 / 変更行に `>` または `>=` の比較演算子が含まれているもの。
- **スコープ外**:
  - `.cs` 以外（`.json`, `.md`, `.yml` 等）
  - `>` `>=` を含まない C# 変更
  - **ジェネリクスの `>` / `=>` ラムダ / `<<` `>>` シフト / 比較ではない箇所**（ヒット時は誤検出として除外する）
  - 既存ファイルで「触っていない行」に残る `>` `>=`（diff の追加/変更行に含まれない）

**スコープ外の場合、共通ルールの出力形式に従って早期終了する。**

## レビュー基準（スコープ内の場合のみ）

### 1. `>` `>=` 比較演算子の使用

**レッドフラグ**: 追加/変更行に二項比較として `>` または `>=` が出現している。

例（NG）:
```csharp
if (amount > 0)              // → if (0 < amount)
if (count >= maxCount)       // → if (maxCount <= count)
return idx >= 0;             // → return 0 <= idx;
while (i > 0)                // → while (0 < i)
var ok = a > b && c >= d;    // → var ok = b < a && d <= c;
```

例（OK・既に正規化されている）:
```csharp
if (0 < amount)
if (maxCount <= count)
if (Amount <= 0)             // OK：左側の方が小さく書きたい意図がある
```

重要な理由:
- 「小さい値 → 大きい値」の向きで揃えると、不等号が常に同じ向きを向き、行列で並べたときに条件の方向性が一目で分かる。
- `>` `>=` を許すと「`a > b`」と「`b < a`」が混在し、grep / 視線移動のコストが上がる。
- `>` `>=` で書ける条件は必ず `<` `<=` で同じ意味のまま書き直せる（操作対象の左右を入れ替えるだけ）ので、表現力の損失なくスタイルを揃えられる。

修正方法:
- `a > b` → `b < a`
- `a >= b` → `b <= a`
- 複合条件 `a > b && c >= d` → `b < a && d <= c`
- 値が定数で「下限チェック」を意図する場合は特に `0 < x` `0 <= x` の形が読みやすい。

### 2. 誤検出を避けるための除外規則

以下は `>` を含むが比較ではないので**指摘しない**:

- ジェネリクス: `List<int>`, `Dictionary<TKey, TValue>`, `IObservable<Unit>`, `Func<int, bool>` 等
- ラムダ式: `x => x.Foo`, `() => DoSomething()` 等
- シフト演算: `x >> 2`, `value << 8`
- 不等号: `!=`（純粋に `>` ではない）
- アロー演算子: `obj->member`（C# では出ないが念のため）
- 文字列内・コメント内・XML doc 内の `>`（`<summary>` タグ等）

検出時は実コードのトークンとして `>` `>=` が二項比較として現れているかを判定する。`<>` の閉じ括弧、ラムダの `=>`、シフトの `>>` は除外する。

### 3. 既存コードへの過剰スイングを避ける

- diff の **追加/変更行** に含まれる `>` `>=` のみを指摘対象とする。
- 既存ファイルで「自分が触っていない行」に残る `>` `>=` を Critical で挙げない。Info で「ファイル全体の統一を希望するなら別 PR で」と添える程度にとどめる。
- このルールがプロジェクト規約として既に確立しているか不明な場合は、Critical ではなく Warning で出す。

## 出力形式

共通ルールの出力形式に従う。指摘ありの場合は以下のテンプレート:

```
## Critical（規約違反、必ず修正）

- {file}:{line} — `{NG式}` を `{OK式}` に書き換え。`>` `>=` 禁止、常に `<` `<=` で「小さい → 大きい」の向きに揃える。

## Warning

（必要に応じて）

## Info

（必要に応じて。例: 既存ファイル全体での揃え直しは別 PR 推奨）
```

400 語上限を厳守。指摘行数は同一ファイル内なら 1 つにまとめてもよい（`{file}:{line1, line2, ...}` 形式）。

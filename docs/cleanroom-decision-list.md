# クリーンルーム 決定リスト（新仕様書の入力）

- 由来: PR #965 の設計書 (`specs/2026-06-05-cleanroom-design.md`)・バランス確定書・実装コードから抽出
- 用途: 新しい仕様書をゼロから書くための唯一の入力。文体・構成・用語は引き継がない
- 凡例: 無印 = 設計時に承認済みの決定 / **⚠ = 実装中に事後決定されたもの（要再承認）**

## 部屋検出

1. クリーンルームは境界ブロックで完全に囲まれた3D密閉空間として検出する。
   理由: 「部屋」という空間単位に純度状態を持たせるため。
2. flood-fill は6近傍。斜めの隙間はリーク扱い。
   理由: 密閉判定を「面で塞ぐ」に統一し、判定を単純化。
3. 境界を貫通してよいのはドア・アイテムハッチ・パイプコネクタのみ。それ以外の壁の欠けは即リーク。
4. リーク判定は「fill中に実際に触れた壁の外接箱(+1)」で縛り、上限体積4096を安全網とする。
   理由: 未密閉構造の探索コストを自室の壁だけで抑えられ、隣接部屋・装飾壁の影響を受けない（誤答2案との比較論証済み）。
5. 再検出の同期即時全走査は禁止。設置/破壊を dirty シードに積み、tick予算（8192セル）内で差分更新、超過分は繰越。
   理由: 巨大な未密閉空間への fill で1tickが破綻するのを防ぐ。
6. 機械等の占有セルは部屋に属するが体積V（=空セル数）に算入しない。
   理由: 機械を詰めるほど実効Vが減り濃度が跳ねやすくなる＝詰め込みの代償。

## 純度モデル

7. 純度は不純物濃度 C = N / V（個/m³）。N は部屋が保持する連続状態。
8. 清浄機は「毎秒一定体積 q を処理」し、除去レート = n·q·C。固定個数除去は不採用。
   理由: 固定除去は dN/dt が定数になり「0クランプか発散か」の二値に崩壊して多段クラスが成立しない。体積処理式は平衡濃度 C_eq = A_total/(n·q) に安定収束する。
9. 汚染 A_total = 機械稼働 + ハッチ搬送レート + a_volume·V + a_surface·S + a_connector·接続点数。
   理由: 空間比例の基礎汚染で「巨大空室のデメリット」を作る。
10. クラス成立は二条件: C ≤ 濃度閾値 かつ ACH = n·q/V ≥ 必要換気回数。
    理由: ACH要求が「大部屋の代償」を局所流体シミュなしで成立させる。
11. ヒステリシス: 降格は素の閾値、昇格は濃度×0.8 かつ ACH×1.25 を満たして初めて発火。【裁定 2026-07-08: ACH側含め承認】
    理由: 閾値付近でのクラス点滅防止（ACH側がないと電力変動起因の点滅が残る）。
12. ハッチ汚染はアイテム1個ごとの加算ではなく搬送レート換算（k_hatch × 直近1秒窓の個/秒）。
    理由: 個数加算はベルト速度・スタックサイズで不自然に回避できる。
13. 【裁定 2026-07-08】ドア通過による汚染バーストは導入しない。ドアは境界貫通ブロックとして接続点数（a_connector）に寄与するのみ。
    経緯: 旧設計は通過バースト15個/通過だったが、実装は通過検知が未統合の死に機能だった。機構ごと廃止。
14. 【裁定 2026-07-08】部屋の状態は「密閉成立（クラス付き）」か「不成立」の2値のみ。旧設計の Valid/Degraded/Invalid 3段階と猶予タイマは廃止。
    理由: 猶予・孤立の機構はややこしさに見合わない。クラス揺れの吸収はヒステリシス（決定11）だけで行う。密閉が崩れた機械は即・稼働不能（決定22）。

## チップ抽選

15. チップのグレードは独立 ItemId（Lv1〜Lv4）。ItemStack に品質フィールドは持たせない。
16. 3レバーの直交分離: 部屋クラス=最大Lv天井＋格下げ率 / 品質モジュール=天井内で上位寄せ / EUV失敗=出力なし率。
    理由: 失敗・格付け・分布を分離し、プレイヤーが出力を推論できる。
17. 抽選の処理順: EUV失敗 → 天井 → 基礎分布抽選 → 格下げ → （品質シフト） → 確定。
18. 抽選は決定的シード（blockInstanceId ＋ セーブされるサイクルカウンタ）から導出。
    理由: セーブ/ロードで結果が再現される。共有 Random の相関も排除。
19. 【裁定 2026-07-08】クラス外（Out）の部屋では機械は稼働しない（旧設計の「稼働するが出力なし」を廃止）。
20. アイテムを部屋外へ出しても変化しない。純度は「製造の瞬間に参照される部屋の属性」でありアイテム状態ではない。
    理由: アイテム単位の汚染度トラッキングの複雑さを回避。

## 製造機と部屋の関係

21. 半導体製造機は有効な部屋内でのみ稼働。部屋が無効なら外置きと同じ＝停止。
22. 【裁定 2026-07-08】部屋が無効・Out の機械は専用の「稼働不能」ステートに入る（Idle とも Processing とも別の第3状態）。旧実装の「Processing のまま凍結＋電力消費継続」は廃止。
    新仕様書で決めること: 稼働不能突入時の加工進捗・投入済み材料の扱い、復帰条件。
23. 部屋クラスがレシピ要求に満たない場合は停止ではなく天井クランプで出力を制限（稼働は継続）。

## 部屋の同一性と永続化

24. 部屋IDは揮発。再検出をまたぐ同一性はセル重なりで判定する。結合は N 合算、分割は N_old × 重なりセル数 / 旧セル数 で按分（N の保存則を維持）。
    注: この引き継ぎは「再検出後も存在し続ける部屋」のためのもの（部屋内へのブロック設置・間仕切り等でクラスがリセットされない）。
25. セーブは部屋ごとに N・クラス行・全セル座標を保存。ロード時は再検出してから最大セル重なりで復元。V/S は保存せず再導出。
26. 【裁定 2026-07-08】孤立部屋（幽霊）システムは持たない。密閉が崩れた部屋の状態は即消滅し、再密閉は常に新品（N=0・クラスは Out から証明し直し）。
    理由: 猶予付きの状態保持は複雑さに見合わない。余計な処理は持たない。

## 空気清浄機

27. 電力駆動（ElectricMachine系）。電力不足時は q × 電力割合 に減衰。
28. フィルターは仕事量ベース消費（除去した不純物の累計が容量に達するごとに1個消費）。残0の台は除去能力0。
    理由: 平衡時のフィルター消費が A_total に比例し「汚い部屋ほどフィルターを食う」動機が直接生まれる。

## 初期パラメータ（調整前提の出発値・決定ではない）

| クラス | 濃度上限(個/m³) | Lv天井 | 格下げ率 | 必要換気(1/秒) |
|---|---|---|---|---|
| A | 10 | Lv4 | 0% | 0.017 |
| B | 50 | Lv3 | 5% | 0.0083 |
| C | 200 | Lv2 | 15% | 0.0042 |
| D | 1000 | Lv1 | 35% | 0.0014 |
| Out | >1000 | なし | — | — |

- 汚染係数: a_volume 0.10 / a_surface 0.05 / a_connector 0.50 / 機械稼働 2.0 / k_hatch 0.30
- 清浄機: q=5.0 m³/秒（満電1台）・要求電力100・フィルター容量5000
- EUV失敗率: 20%（例値）
- 昇格ヒステリシス: 濃度×0.8・ACH×1.25
- 検証用の基準例: 内寸5×5×3（V=75, S=110, 接続点2, 稼働機械1, 清浄機1台満電）→ A_total=16.0 → C_eq=3.2 → クラスA成立。10×10×5（V=500）に拡げると1台ではクラスB止まり＋ACH未達。
- 守るべき2不変条件: 「小部屋は1台で最良クラス」「大部屋は増設必須」。数値調整はこれを壊さない範囲で行う。

29. 【裁定 2026-07-08】共有境界の計上は「面する全部屋に全額」。1つのハッチが2部屋に面するなら搬送汚染・接続点数を両部屋にフル計上する。
    理由: 実装が最単純で過小計上がなく、部屋間直結を僅かに不利にして集約を促す方向に働く。

## 新仕様書で新たに決めるべきこと（旧仕様の穴）

- 稼働不能ステートの詳細（進捗・投入済み材料の扱い、復帰条件、クライアントへの状態表示）
- 汚染係数・抽選分布をマスタデータ化するか（旧実装は閾値表のみマスタ、係数はハードコードで不揃い）
- クライアント側の可視化（部屋クラス・濃度・フィルター残の表示）。旧仕様はサーバーのみで完全に対象外だった
- semiconductorChips マスタの実データ（旧実装は空のまま）

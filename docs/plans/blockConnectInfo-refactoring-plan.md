# blockConnectInfo.yml廃止とインターフェースベーススキーマへの移行計画

## 概要

現在の`blockConnectInfo.yml`は、`connectType`（Inventory/Gear/Fluid）を手動選択する必要がありミスが起こりやすい。これを廃止し、Inventory/Gear/Fluidそれぞれのコネクタスキーマへ分離する。

現在も`VanillaSchema/ref/blockConnectInfo.yml`、`connectType`、`connectOption`、`fixedParameter`は残っているため、この移行方針自体は有効。ただし旧プランには現行コードと合わないパス、SourceGeneratorのスコープ制約、修正対象漏れがあるため、本ドキュメントではそれを反映する。

### connectorGuidについて

現在の`blockConnectInfo.yml`には`connectorGuid`（autoGenerated UUID）が定義されており、ベルトコンベアのアイテムエンティティなどで使用されている。このGUIDは各コネクタを一意に識別するために必須であり、新しいスキーマでも維持する。

## 意思決定事項

### 1. 共通コネクタインターフェースをどこまで持つか

`IBlockConnector`に含める共通プロパティは、現時点では以下に絞るのが安全。

- `connectorGuid`
- `offset`
- `directions`

`option`はInventory/Gear/Fluidで生成型が異なるため、`IBlockConnector`に含めるとC#インターフェースの戻り値型不一致でSourceGenerator後に崩れる可能性がある。`option`を共通化したい場合は、SourceGeneratorが「各具象Option型のプロパティを共通インターフェース型として公開できるか」を先に検証する。

### 2. `IBlockConnectOption`を使うか

Gear/Fluid/Inventoryの`option`オブジェクトに`IBlockConnectOption`を実装させる案は残せる。ただし現行コードでは`GearConnectOption`/`FluidConnectOption`相当の具体型へキャストしているため、共通インターフェースだけでは実用上ほぼ不足する。

判断:
- 型安全な具体Option型を主に使うなら、`IBlockConnectOption`は不要でもよい。
- 将来、Optionを共通リストや共通APIで扱う予定があるなら、空インターフェースとして残す。

### 3. `globalDefineInterface`を使うか

別ファイルの`inventoryConnects.yml`、`gearConnects.yml`、`fluidInventoryConnects.yml`から参照するインターフェースは、`blocks.yml`の`defineInterface`ではなく`globalDefineInterface`に置く必要がある。

SourceGenerator側ではローカル`defineInterface`は同一スキーマ内からしか参照できない。旧プランのまま`defineInterface`に追加すると、別ファイル参照時に`InterfaceNotFound`またはスコープ診断で失敗する可能性が高い。

### 4. `mooresmaster.SandBox/schema`も同期するか

ゲーム本体は`VanillaSchema`を参照するが、`mooresmaster/mooresmaster.SandBox/schema`にも同じ旧スキーマが残っている。SourceGeneratorサンプル・検証環境を壊さないため、実装時に同期するかを決める。

推奨: `VanillaSchema`変更と同じコミット、または直後の検証コミットで`mooresmaster.SandBox/schema`も同期する。

## Phase 1: スキーマ変更

### 1.1 共通インターフェース追加

**ファイル**: `VanillaSchema/blocks.yml`

`globalDefineInterface`に追加する。既に`globalDefineInterface`がなければ、トップレベルに新設する。

```yaml
globalDefineInterface:
- interfaceName: IBlockConnector
  properties:
  - key: connectorGuid
    type: uuid
  - key: offset
    type: vector3Int
  - key: directions
    type: array
    optional: true
    items:
      type: vector3Int
```

`autoGenerated`や`default`は具象スキーマ側に置く。インターフェース側は実装クラスに要求するプロパティ形状だけを定義する。

`IBlockConnectOption`を採用する場合のみ、同じ`globalDefineInterface`に追加する。

```yaml
- interfaceName: IBlockConnectOption
  properties: []
```

### 1.2 inventoryConnects.yml書き換え

**ファイル**: `VanillaSchema/ref/inventoryConnects.yml`

変更前: `ref: blockConnectInfo` + `fixedParameter`
変更後: 直接定義 + `implementationInterface: [IBlockConnector]`

`connectType`は削除する。`option`は空オブジェクトとして残すか、Inventoryでは削除するかを実装前に決める。既存JSON移行量を抑えるなら空`option`を残す。

### 1.3 gearConnects.yml書き換え

**ファイル**: `VanillaSchema/ref/gearConnects.yml`

`gearConnects.items`を直接定義し、各要素に`implementationInterface: [IBlockConnector]`を付ける。`option`には`isReverse`を持たせる。

### 1.4 fluidInventoryConnects.yml書き換え

**ファイル**: `VanillaSchema/ref/fluidInventoryConnects.yml`

`inflowConnects.items`と`outflowConnects.items`を直接定義し、各要素に`implementationInterface: [IBlockConnector]`を付ける。`option`には`flowCapacity`と`connectTankIndex`を持たせる。

現行の`fluidInventoryConnects.yml`は`fixedParameter`なしで`blockConnectInfo`を参照しているため、Inventory/Gearよりも`connectType`手入力ミスの余地が大きい。優先的にテストする。

### 1.5 blockConnectInfo.yml削除

**ファイル**: `VanillaSchema/ref/blockConnectInfo.yml` -> 削除

`mooresmaster/mooresmaster.SandBox/schema/ref/blockConnectInfo.yml`も同期対象にする場合は同時に削除する。

### 1.6 SourceGenerator再生成トリガー

**ファイル**: `moorestech_server/Assets/Scripts/Core.Master/_CompileRequester.cs`

`dummyText`の値を変更してSourceGeneratorを再実行する。旧プランの`Dummy.cs`は現行ツリーに存在しない。

## Phase 2: C#コード修正

コンパイル後、生成された型名に合わせてC#コードを修正する。生成型名はSourceGenerator結果を確認してから確定する。

### 主な変更パターン

1. `ConnectOption` -> `Option`
2. `BlockConnectInfo` -> Inventory/Gear/Fluid別の配列型または共通インターフェース受け
3. `BlockConnectInfoElement` -> 各コネクタ固有の生成型、または`IBlockConnector`
4. `GearConnectOption` / `FluidConnectOption` -> 新しい具体Option型
5. `Mooresmaster.Model.BlockConnectInfoModule` using削除

## Phase 3: JSONマスターデータ移行

### 変更内容

| 変更前 | 変更後 |
|--------|--------|
| `connectType: "Inventory"` | 削除 |
| `connectType: "Gear"` | 削除 |
| `connectType: "Fluid"` | 削除 |
| `connectOption: {...}` | `option: {...}` |
| `connectorGuid: "..."` | 維持 |

`connectorGuid`は各コネクタを一意に識別するために使用されており、削除してはならない。既存JSONデータに含まれる`connectorGuid`はそのまま維持する。

### 対象ファイル

1. `moorestech_server/Assets/Scripts/Tests.Module/TestMod/ForUnitTest/mods/forUnitTest/master/blocks.json`
2. `moorestech_client/Assets/Scripts/Client.Tests/EditModeInPlayingTest/ServerData/mods/EditModeInPlayingTestMod/master/blocks.json`
3. `../moorestech_master/server_v8/mods/**/master/blocks.json`
4. `mooresmaster/mooresmaster.SandBox/TestMod/blocks.json`（SandBox同期する場合）

### JSON変更例

変更前:

```json
{
  "connectorGuid": "12345678-1234-1234-1234-123456789abc",
  "offset": [0, 0, 0],
  "connectType": "Gear",
  "directions": [[1, 0, 0]],
  "connectOption": {"isReverse": true}
}
```

変更後:

```json
{
  "connectorGuid": "12345678-1234-1234-1234-123456789abc",
  "offset": [0, 0, 0],
  "directions": [[1, 0, 0]],
  "option": {"isReverse": true}
}
```

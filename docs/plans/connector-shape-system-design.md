# コネクタ形状システム設計書

## 背景・問題

現在のブロック接続は `BlockConnectorComponent<TTarget>` の位置判定のみで確立される。
「自分の出力コネクタ位置」と「相手の入力コネクタ受入位置」が一致すれば無条件で接続するため、以下の問題がある。

1. **形状の不一致を検出できない**: 歯車の歯の部分とシャフトの端面のように、物理的に噛み合わないコネクタ同士が接続してしまう
2. **向きの不一致を検出できない**: 同じ「歯」同士でも、歯車が90度回転していて回転軸が直交している場合は噛み合わないはずだが、接続してしまう

1は静的な形状互換表で解決できるが、2は「同じ形状・向き依存」の問題であり、表では表現できない。
ドメイン固有の判定ロジックを注入できる仕組みが必要となる。

## ゴール

- コネクタに「形状」を定義し、形状同士の互換性をマスタデータで設定できるようにする
- 形状互換表で表現できないドメイン固有の接続条件（歯車の軸平行チェック等）を、型安全に注入できるようにする
- 既存マスタデータを変更せずに現行挙動を維持できる（形状未設定 = 制約なし）

## 非ゴール（スコープ外）

- クライアント側の設置プレビューでの接続可否表示（将来課題）
- 設置そのものの禁止（接続が確立しないだけで、ブロックは自由に設置できる。現行挙動を踏襲）
- 実マスタ（moorestech_master）への形状付与（shapeGuidはoptionalのため既存挙動は不変。テスト用マスタにのみ付与）
- 判定ロジックのyamlからの選択（判定クラスはドメインごとにC#で固定。YAGNI）
- ベベルギア等の直交軸接続（軸平行のみを正とする）

## 決定事項

1. **形状定義は `blocks.yml` トップレベルの `connectorSettings` オブジェクト配下にまとめる**。`connectorSettings.connectorShapes`（shapeGuid + name）に形状を定義する
2. **互換表は無向ペアのリスト `connectorSettings.connectableShapePairs`（shape0 / shape1）**。同一形状同士も暗黙では繋がらず、必要なら (A, A) を明示登録する
3. **`IBlockConnector` に `shapeGuid`（uuid, optional）を追加。未設定 = 形状制約なし（何とでも繋がる）**。`directions: null = 無制限` と同じ意味論
4. **形状互換表チェックは `BlockConnectorComponent` 本体で必ず適用**し、注入する判定クラスは「追加条件」のみを担う（表チェックの書き忘れを構造的に防止）
5. **判定は接続確立時（`OnPlaceBlock`）のみ**
6. **判定クラスは第2型パラメータ `TConnectJudge` で型レベル束縛**（`new()` 制約、static キャッシュ）。閉じたジェネリック型が検索キーとなるため、両側ブロックが同じ判定クラスを使うことがコンパイル時に保証される
7. **歯車の向き判定は「噛み合い軸の平行チェック」**。軸情報は `gearConnectOption` にオプショナルな `meshingAxis`（vector3Int, ローカル座標）として追加する
   - 当初は BlockDirection のみから導出する想定だったが、コネクタのdirectionsから軸は一意に導出できない（例: BigTestGear は面ごとに単方向コネクタを持ち、[-1,0,0] に垂直な軸は Y/Z の2候補があり曖昧）ため、データ追加が必要と判断
   - `meshingAxis` 未設定 = 向き制約なし（shapeGuid と同じワイルドカード意味論）

## スキーマ設計

### blocks.yml トップレベル（`data` の隣に追加）

形状定義と互換表は `connectorSettings` オブジェクトに1階層まとめる。

```yaml
- key: connectorSettings
  type: object
  optional: true
  properties:
  - key: connectorShapes
    type: array
    optional: true
    items:
      type: object
      properties:
      - key: shapeGuid
        type: uuid
        autoGenerated: true
      - key: name
        type: string
  - key: connectableShapePairs
    type: array
    optional: true
    items:
      type: object
      properties:
      - key: shape0
        type: uuid
        foreignKey:
          schemaId: blocks
          foreignKeyIdPath: /connectorSettings/connectorShapes/[*]/shapeGuid
          displayElementPath: /connectorSettings/connectorShapes/[*]/name
      - key: shape1
        type: uuid
        foreignKey:
          schemaId: blocks
          foreignKeyIdPath: /connectorSettings/connectorShapes/[*]/shapeGuid
          displayElementPath: /connectorSettings/connectorShapes/[*]/name
```

各コネクタの `shapeGuid` にも同じ foreignKey を付与する（同一ファイル内参照は `schemaId: blocks` + トップレベルパスで表現。実装時に edit-schema skill でエディタ側の解決可否を確認する）。

### globalDefineInterface: IBlockConnector への追加

```yaml
- key: shapeGuid
  type: uuid
  optional: true
```

インターフェースへの追加に伴い、実装側の各コネクタ要素にも同プロパティを追記する:

- `VanillaSchema/ref/inventoryConnects.yml`（inputConnects / outputConnects の items）
- `VanillaSchema/ref/gearConnects.yml`（gearConnects の items）
- `VanillaSchema/ref/fluidInventoryConnects.yml`（inflowConnects / outflowConnects の items）

### gearConnectOption.yml への追加

```yaml
- key: meshingAxis
  type: vector3Int
  optional: true
```

歯コネクタには歯車の回転軸（例: SmallTestGear なら [0,0,1]）、シャフトコネクタにはシャフトの軸方向を設定する。
歯車機械のシャフト受け口にも設定でき、向き不問にしたいコネクタは未設定のままにする。

前回リファクタの教訓: option を匿名 object で書くと SourceGenerator が重複クラスを生成するため、必ず既存の `gearConnectOption` ref に追記する。

## ランタイム設計

### 判定インターフェースとコンテキスト

配置: `Game.Block.Interface/Component/`（IBlockConnectorComponent の隣）

```csharp
public interface IConnectorConnectJudge
{
    bool CanConnect(ConnectJudgeContext context);
}

public readonly struct ConnectJudgeContext
{
    public readonly IBlockConnector SelfConnector;       // 出力側コネクタ
    public readonly IBlockConnector TargetConnector;     // 入力側コネクタ
    public readonly BlockPositionInfo SelfPositionInfo;  // 出力側ブロック（向き含む）
    public readonly BlockPositionInfo TargetPositionInfo;
}
```

### BlockConnectorComponent の変更

```csharp
public class BlockConnectorComponent<TTarget, TConnectJudge>
    : IBlockConnectorComponent<TTarget>
    where TTarget : IBlockComponent
    where TConnectJudge : IConnectorConnectJudge, new()
{
    private static readonly TConnectJudge Judge = new();
}
```

`OnPlaceBlock` の判定を3段にする:

1. **位置一致**（現行ロジックそのまま）
2. **形状互換表**: 双方の shapeGuid を `ConnectorShapeCompatibility.CanConnect(Guid? self, Guid? target)` で照合。どちらかが未設定なら常に true
3. **ドメイン判定**: `Judge.CanConnect(context)`

注意点:
- 相手側検索 `TryGetBlock<BlockConnectorComponent<TTarget, TConnectJudge>>` も閉じた2型パラメータで行う。これにより判定クラスの不一致が型不一致 = 接続不可として自然に表現される
- 相手インプットが `directions == null`（方向無制限）で接続する経路でも、形状表と Judge は必ず通す。この経路では相手側コネクタが特定されないため、`ConnectedInfo` 生成と同様に相手コネクタ null 相当の扱いとし、形状チェックはワイルドカード（制約なし）として通過させる。Judge 側もコネクタ null を「制約なし = true」として扱う契約とする（GearConnectJudge は null なら軸チェックをスキップ）
- 相手側 `BlockPositionInfo` は `WorldBlockDatastore.GetBlock(outputTargetPos).BlockPositionInfo` から取得

### 形状互換表のロード

配置: `Core.Master`（MasterHolder の隣）

- 生成モデル（blocks.json トップレベルの connectorSettings 配下 connectorShapes / connectableShapePairs）から、順序正規化した `HashSet<(Guid, Guid)>` を構築する
- 構築タイミングは `MasterHolder.Load` 時。`ConnectorShapeCompatibility` が static に保持し、`CanConnect(Guid? a, Guid? b)` を公開する

### ドメイン別判定クラス

| ドメイン | 判定クラス | 配置 | 内容 |
|---|---|---|---|
| インベントリ | `DefaultConnectJudge` | Game.Block.Interface/Component/ | 常に true（表チェックのみで足りる） |
| 流体 | `DefaultConnectJudge` | 同上 | 同上 |
| 歯車 | `GearConnectJudge` | Game.Gear/Common/ | 噛み合い軸の平行チェック |

閉じた型の一覧（テンプレート・コンポーネントの参照をすべてこの3つに置換）:

- `BlockConnectorComponent<IBlockInventory, DefaultConnectJudge>`
- `BlockConnectorComponent<IFluidInventory, DefaultConnectJudge>`
- `BlockConnectorComponent<IGearEnergyTransformer, GearConnectJudge>`

### GearConnectJudge の判定ロジック

1. 双方の `IBlockConnector` を `GearConnectsElement` にキャストし、`Option.MeshingAxis` を取得（歯車ドメインの生成型は1種のみのため直接キャスト。前回リファクタの `GearConnect.FromConnectedInfo` と同方式）
2. どちらかの `meshingAxis` が未設定なら true（向き制約なし）
3. 各ローカル軸を `BlockDirection.GetRotation()` によるワールド変換（`GetCoordinateConvertAction` と同じ回転行列 + RoundToInt）でワールド軸に変換
4. 外積がゼロベクトル（= 平行、逆向き含む）なら true、それ以外は false

これにより:
- 歯と歯: 両歯車の回転軸が平行なときのみ噛み合う（90度回転した歯車は拒否）
- シャフトとシャフト: 軸が同一直線方向のときのみ接続
- 歯とシャフト: そもそも形状互換表（決定事項2）で拒否

## テスト計画

新規テスト（creating-server-tests skill 準拠）:

1. 形状互換表: 互換ペアあり → 接続、ペアなし → 非接続、片側未設定 → 接続
2. 歯車の向き: 同一平面の歯車同士（軸平行）→ 接続、90度回転した歯車（軸直交）→ 非接続
3. 形状分離: 歯コネクタとシャフトコネクタが隣接位置で向き合っても非接続
4. meshingAxis 未設定の歯車 → 向き不問で接続（現行挙動）

回帰確認:

```
uloop compile --project-path ./moorestech_client
uloop run-tests --project-path ./moorestech_client --filter-type regex --filter-value "BlockConnector|BeltConveyor|FilterSplitter|Fluid|GearNetwork"
```

その後 EditMode 全件パスを確認する。

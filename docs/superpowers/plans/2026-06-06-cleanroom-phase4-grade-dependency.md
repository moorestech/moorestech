# クリーンルーム フェーズ4 グレード依存 決着メモ

- 日付: 2026-06-06
- 対象: クリーンルーム フェーズ4（製造機統合）の前提＝「出力チップのグレード（Lv）表現と抽選機構」をどう用意するか
- 種別: **決定メモ**。フェーズ4 TDDプランの前提を1つに固定する。

---

## 1. 問題

クリーンルームのクラスは「最大グレードの天井＋汚れによる下位グレードへの格下げ（down-bin）」を課す（設計書§4）。これは**出力チップが Lv1〜LvN の区別を持つ**ことを前提にする。だが:

- `ItemStack` に品質/Lvフィールドは**無い**（`ItemStackMetaData` はスケルトンのみ・未使用）。グレードは**独立 ItemId**で表すのがアップグレード設計書§3の確定事項。
- その「独立 ItemId のレベルファミリー」「機械の出力レベル分布抽選」「抽選順序の決定性（§7.2）」は、アップグレードシステムの**フェーズB（品質軸）**に割り当てられている。
- **そのフェーズB計画は未作成・未実装**（`2026-06-05-upgrade-system-phase-a.md` が品質軸を明示的に対象外と宣言し、`2026-XX-XX-upgrade-system-phase-b.md` は存在しない）。

つまりフェーズ4が乗るべき土台（レベル抽選）が宙に浮いている。

## 2. すでに在る土台（フェーズA成果）

アップグレード**フェーズA は実装計画済み**で、フェーズ4が再利用できる足場を残している:

- `MachineModuleEffect`（効果集計・純粋関数＋clamp）に **Quality 軸の集計分岐の「穴」が確保済み**（A3-1 / 完了条件で明記）。
- 生産性の追加産出で **決定的乱数**（共有 static Random を避け、保存される `_processedCycleCount` ＋ `_blockInstanceId` から導出）を確立済み（A3-4）。これは §7.2 が要求する「決定的シード経路」そのもの。
- 仮想容量予約（開始前に最大産出分を出力容量に予約）パターンも確立済み（A3-4）。

→ レベル抽選の**順序固定（§7.2）と決定的乱数源は、フェーズAの流儀をそのまま継げる**。

## 3. 選択肢

| 案 | 内容 | 評価 |
|---|---|---|
| **A. アップグレードのフェーズBを待つ** | レベルファミリー＋出力レベル抽選をフェーズBで作り、フェーズ4はそれに相乗り | フェーズB計画が未作成。クリーンルームが無関係な機能の完成待ちで**ブロックされる**。✗ |
| **B. フェーズ4が「レベル抽選コア」を半導体チェーン限定で実装（推奨）** | フェーズ4内で「半導体チップのレベルファミリー（独立ItemId）＋機械の出力レベル分布抽選」を、フェーズAの決定的乱数・§7.2順序に**準拠して**実装。クリーンルームのクラスはその抽選への**部屋クラス修正子**として乗せる | クリーンルームを単独で完結できる。後述の通り将来のフェーズBと**衝突しない**形に設計可能。○ |
| C. クリーンルームとアップグレードBを合同で1プラン化 | 両者が同じ抽選を共有するので一緒に作る | スコープ肥大・2機能を結合。クリーンルームの独立性を失う。△ |

## 4. 決定：案B（フェーズ4がレベル抽選コアを担う）

**フェーズ4は、半導体チェーンのチップに限定した「レベルファミリー＋出力レベル分布抽選」を、フェーズAの決定的乱数・§7.2抽選順序に準拠して実装する。クリーンルームのクラスは、その抽選にかかる修正子（天井クランプ＋down-bin）として実装する。**

### 4.1 なぜ衝突しないか（将来のアップグレードBとの関係）

- §7.2 の抽選順序は「①レシピ完了 → ②生産性で個数決定 → ③各出力に品質（レベル）決定」。フェーズ4が実装するのは **③のレベル決定の中核（基礎レベル分布テーブル＋決定的抽選）**。
- 将来アップグレードのフェーズB（品質モジュール）は、この**③に確率シフトを足すだけ**になる（`MachineModuleEffect` の Quality 穴 → 分布シフト量を供給）。フェーズ4が③のコアを先に作る＝フェーズBは「修正子の供給側」に縮む。
- クリーンルームの修正子も同じ③にかかる。**順序固定（クリーンルーム天井 → 品質シフト → down-bin → レベル確定）と単一の決定的乱数源**を共有すれば、後からモジュールが入っても整合する。

→ 実装の重複は無い。フェーズ4が「③レベル決定」の置き場所と乱数規約を確定し、アップグレードBは後からそこへ shift を差すだけ。

### 4.2 フェーズ4が用意するもの（コアの最小セット）

- **チップのレベルファミリー**: `ICチップ_Lv1..Lv4` を独立 ItemId で定義（**決定的 GUID**＝既存GUID固定 or baseGuid+level 派生。ランダム禁止＝セーブ破損防止、アップグレード設計書§4.3）。半導体チェーンの ICチップ基盤の先に置く。
- **機械の基礎出力レベル分布テーブル**: 露光装置等が持つ `Lv1:p1 / Lv2:p2 / …`（マスタ。初期はクラス天井内で一様でも可）。
- **決定的抽選**: フェーズA の `_processedCycleCount` ＋ `_blockInstanceId` 由来シードを流用。
- **クリーンルーム修正子の合成点**: §7.2 の③で「(1) 部屋クラスの最大Lvで天井クランプ → (2) 基礎分布で抽選 → (3) down-bin率で確率的に1段格下げ」。

### 4.3 マスタ追記

- レベルファミリー定義（アップグレード設計書§4.3の `baseItem / maxLevel / combineCount` 方式）を**半導体チップに適用**。SourceGenerator が `ICチップ_Lv1..Lv4` と段間合成レシピを生成。
- 露光装置レシピの出力に「レベル分布テーブル」を持たせる枠を `machineRecipes` 系スキーマへ追加（フェーズ4プランで詳細化）。

## 5. フェーズ4プランへの引き渡し事項

1. レベルファミリー＝**半導体チップ限定**で実装（汎用化はアップグレードBに委ねる）。GUIDは決定的に。
2. レベル決定は §7.2 の③として実装し、**単一の決定的乱数源**（フェーズA流儀）に一本化。
3. クリーンルーム修正子の合成順序を固定: **室Invalid停止ゲート → （稼働）→ 完了時：天井クランプ → 基礎分布抽選 → down-bin格下げ → Lv確定**。
4. `VanillaMachineOutputInventory.InsertOutputSlot` の出力 ItemStack 生成直前に③を差す。multi-block 占有は `BlockPositionInfo.MinPos..MaxPos` × `TryGetRoomContainingBlock`。
5. 将来アップグレードBが品質シフトを差せるよう、`MachineModuleEffect` の Quality 穴を分布シフトの入口として温存（埋めはしないが場所を壊さない）。

## 6. リスク

- フェーズ4が「③レベル決定」のコアを持つ＝アップグレードBがこのコアAPIに合わせて書かれる必要がある（依存方向が逆転）。**フェーズ4完了時に③のAPI（分布テーブル型・抽選関数シグネチャ・乱数規約）を明文化**し、アップグレードB計画の入力にする。
- 半導体限定で作ったレベルファミリー機構を後で汎用化する際の移植コスト。ただし「独立ItemId＋決定的GUID＋SourceGenerator生成」という骨格は共通なので、汎用化は generator のパラメータ化で済む見込み。

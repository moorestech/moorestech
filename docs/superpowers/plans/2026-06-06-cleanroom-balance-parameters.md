## 0. 単位と基準系

- 不純物総数 `N`: 個（無次元カウント、double）。
- 体積 `V`: m³（= 部屋の空セル数。1セル=1m³とする）。
- 表面積 `S`: m²（部屋の空セルが境界ブロックに面する面の数）。
- 濃度 `C = N/V`: 個/m³。
- 汚染レート `A`: 個/秒。
- エアフィルター処理体積 `q`: m³/秒。除去レート = `n·q·C`（個/秒）。
- 換気回数 `ACH = n·q/V`: **本書は内部単位を 1/秒で統一**。直感用に 1/時（×3600）も併記する。
- tick: 50ms（20/秒）。毎tickの積分量 = レート × 0.05。

**基準部屋（worked example の土台）**: 内寸 5×5×3 = **V=75 m³**、表面積 **S≈110 m²**（床25+天井25+壁60）。

## 1. クラス定義（濃度閾値・最大グレード・down-bin率・必要換気回数）

クラスは清浄なほど上位（A が最良）。半導体チップは独立 ItemId の Lv1〜Lv4（フェーズ4／決着メモ参照）。

| クラス | 濃度上限 `classThreshold`（個/m³） | 最大グレード（チップLv天井） | down-bin率（汚れ由来の下位格下げ） | 必要換気 `requiredAirChangeRate`（1/秒 ／ 1/時） |
|---|---|---|---|---|
| **A** | ≤ 10 | Lv4 | 0% | 0.0167 ／ 60 |
| **B** | ≤ 50 | Lv3 | 5% | 0.0083 ／ 30 |
| **C** | ≤ 200 | Lv2 | 15% | 0.0042 ／ 15 |
| **D** | ≤ 1000 | Lv1 | 35% | 0.0014 ／ 5 |
| **Out** | > 1000 | なし（半導体製造不可。**MaxGrade=0 ＝抽選せず出力を生成しない**。サイレントに Lv1 を出してはならない） | — | — |

> **「クラス」はコード上の列挙型ではない（コードマップ v2）。** A〜D は `cleanRoomThresholds.yml` の閾値行のラベルにすぎない。コードは濃度 C／ACH をマスタ閾値に照らして都度 `CleanRoomEffect`（MaxGrade/DownBinRate）を算出する。ただし**ヒステリシスのために「現在の閾値行インデックス（int）」を `CleanRoom` が保持・セーブする**（§6）。
> 1/秒列と 1/時列は `1/秒 = 1/時 ÷ 3600` で正確に対応させること（テスト期待値は 1/秒 側を使う）。

- **クラス成立の二条件**（設計書§3）: `C ≤ classThreshold(class)` **かつ** `ACH = n·q/V ≥ requiredAirChangeRate(class)` の**両方**を満たす最良クラス。濃度が足りても換気が足りなければ降格。
- **最大グレード天井**: そのクラスの最大Lvを超えるチップは出ない（上限クランプ）。
- **down-bin率**: 天井内で生成された出力のうち、この割合を1段下位Lvへ格下げ（フェーズ4の抽選修正子。詳細は決着メモ §3）。

### 1.1 ヒステリシス（クラス揺れ防止）

クラス境界は**上げ閾値と下げ閾値を別**にする。`classThreshold` を「下げ（悪化）方向の閾値」とし、上げ（改善）方向は **0.8倍**で別に持つ（10% ではなくマージン20%）。

| 遷移 | 使う閾値 |
|---|---|
| 改善（C→B など、上位へ上がる） | `classThreshold(上位) × 0.8` を下回って初めて昇格 |
| 悪化（B→C など、下位へ落ちる） | `classThreshold(現クラス)` を超えたら降格 |

例: B→A への昇格は C ≤ 8（10×0.8）で発火、A→B への降格は C > 10 で発火。間の 8〜10 は現状維持＝点滅しない。

### 1.2 Valid / Degraded / Invalid と猶予

| 状態 | 条件 | 機械挙動（フェーズ4） |
|---|---|---|
| Valid | 密閉成立・クラス維持 | 通常稼働 |
| Degraded | 密閉判定が一時崩壊／クラス急落直後。**猶予タイマ作動中** | 停止しない（稼働継続） |
| Invalid | 猶予切れもリーク継続 | 稼働停止（外置きと同等） |

- **Degraded 猶予秒数 `graceSeconds` = 5.0 秒**（= 100 tick）。建築編集・チャンクロード順による一瞬の無効化で即停止しないため。
- 猶予中に密閉が回復すれば Valid へ。猶予切れまで未回復なら Invalid。

## 2. 汚染源係数（A_total の各項）

```
A_total = A_machine + A_hatch + A_door
        + a_volume   · V
        + a_surface  · S
        + a_connector · 接続点数
```

| 係数 | 初期値 | 単位 | 意味・根拠 |
|---|---|---|---|
| `a_volume` | **0.10** | 個/(m³·秒) | 空間維持コスト。巨大空室を不利に |
| `a_surface` | **0.05** | 個/(m²·秒) | 壁面・シール面からの漏れ |
| `a_connector` | **0.50** | 個/(接続点·秒) | ハッチ/ドア/パイプコネクタ1個あたり。搬送口を増やすほど汚れる |
| `A_machine` | **2.0** | 個/(稼働機械·秒) | 半導体製造機が `Processing` 中に排出。詰めるほど汚染増 |
| `A_hatch`（レート係数 `k_hatch`） | **0.30** | 個/(アイテム搬送) | `A_hatch = k_hatch · 直近スループット(個/秒)`。1個ごと加算でなく**レート換算**（設計書§5） |
| `A_door`（バースト `burst_door`） | **15** | 個/通過 | プレイヤー1通過あたりの瞬間加算。自動化を促す大きめ値 |

> 接続点数は `CleanRoomBoundaryKind` 別に重み分けも可能（初期は一律 `a_connector`）。ドアの恒常項は `a_connector` に含め、通過バーストは別途 `A_door`。
>
> **単位の重要注意（バーストとレートを混ぜない）**: `burst_door` は「個/通過」の**瞬間量**であり、「個/秒」の `A_total` に直接合算してはならない（1tickしか乗らないため N 増分が `15 × 0.05 = 0.75` になり仕様の 1/20 に化ける）。バーストは `CleanRoom.AddImpurity(burst)` で **N へ直接加算**する。テストは A_total ではなく「通過1回で N が +15 される」ことをアサートすること。
>
> **接続点・ハッチ計量の重複排除**: 接続点カウントは**セル単位ではなく `BlockInstanceId` 単位**で行う（マルチセル境界ブロックの多重カウント防止）。2部屋の共有境界にあるハッチの扱い: 恒常項 `a_connector`・`A_hatch` は**面する各部屋に計上**（両部屋が汚染リスクを負う）、`burst_door` は**面する全部屋へ全額加算**。バーストの読み出しは consume-and-clear ではなく **peek（非破壊）→ tick終端で advance** に分離し、部屋の評価順で結果が変わらないようにする。

## 3. エアフィルター（CleanRoomAirFilter） ＋ フィルター ＋ 電力

| パラメータ | 初期値 | 単位 | 意味 |
|---|---|---|---|
| 処理体積 `q` | **5.0** | m³/秒（1台・満電時） | 除去レート = `n·q·C` |
| 要求電力 `requiredPower` | **100** | （既存電力単位に合わせる） | 満電で q を発揮。不足時は `q × (currentPower/requiredPower)`。**スキーマのキー名は既存規約どおり `requiredPower`**（`ElectricMachine` の前例。`requestPower` という独自名は使わない） |
| フィルター除去容量 `filterCapacity` | **5000** | 個/フィルター1個 | 仕事量ベース。除去した不純物の累計がこの値に達したら1個消費。**blockParam としてマスタデータ化する**（C# ハードコード禁止。本書は調整前提の数値であり、調整がコード変更になってはならない） |
| フィルターアイテム識別 | `filterItemGuid`（blockParam） | — | スロット内の**任意アイテムを消費してはならない**。許可フィルターアイテムを param で識別し、消費時にアイテム種チェックを行う |
| 高級フィルター | 後追い | — | 除去効率の高い上位種別。初期は1種のみ |

- **電力割合換算**は既存 `MachineCurrentPowerToSubSecond.GetSubTicks(currentPower, requiredPower)` と同じ流儀。
- **フィルター仕事量消費**: 自台の除去寄与 `q·C·(電力割合)·0.05`（毎tick）を累計し、`filterCapacity` ごとに1個消費。平衡時消費は A_total に比例 → 汚い部屋ほど食う。**この配線（データストア→`ApplyRemovedImpurity`→摩耗）は必ず統合テストで固定する**（平衡 t 秒後の摩耗累計 ≈ A_total×t を ±10% でアサート。配線を消してもテストが通る状態を禁止）。
- フィルター切れ（残0）の台は `RemovalVolumePerSecond = 0`（除去しない）。
- **離散化の安定条件**: 陽的オイラーのため `n·q·Δt/V < 2` が必要（Δt=0.05）。現行 q=5 では1セル1台が物理上限なので安全だが、**高級フィルターで q を上げる場合はこの条件を再確認すること**。

## 4. 整合する基準例（worked example）

基準部屋 V=75, S=110、接続点 2（ハッチ1・ドア1）、稼働機械 1台、ハッチ無搬送・ドア無通過の定常:

```
A_total = A_machine·1 + 0 + 0 + a_volume·75 + a_surface·110 + a_connector·2
        = 2.0 + 0.10·75 + 0.05·110 + 0.50·2
        = 2.0 + 7.5 + 5.5 + 1.0 = 16.0 個/秒
```

エアフィルター n=1, q=5 満電:
```
C_eq = A_total / (n·q) = 16.0 / 5 = 3.2 個/m³   → 濃度はクラスA域（≤10）
ACH  = n·q/V = 5/75 = 0.067 /秒（240/時）       → A の要求 0.0167 を満たす
→ クラス A 成立（Lv4 まで製造可）
```

エアフィルターを 0 台にすると `dN/dt = 16` で N が増え続け、C は V でしか割られないため発散 → やがて Out（停止）。1台で十分清浄、という基準。部屋を 10×10×5（V=500, S≈400）に拡げると:
```
A_total ≈ 2.0 + 0.10·500 + 0.05·400 + 1.0 = 73 個/秒
C_eq(1台) = 73/5 = 14.6 → クラスB止まり。さらに ACH = 5/500 = 0.01 < 0.0167 で A 要求も未達。
→ 大部屋で A を維持するには複数台（n↑）が必要。「大部屋の代償」が数値で出る。
```

この2例が「小部屋は1台で最良クラス／大部屋は増設必須」という設計意図（設計書§3.体積の役割）を満たすことを、フェーズ2テストの平衡アサーションで固定する。

> **占有セル規則を厳密適用した実テスト値（フェーズ3 keystone）**: 上の V=75 例は直感用の**理想形**（占有なし・機械稼働あり）。実際の統合テストは室内のエアフィルター1台が1セルを占有するため、**機械なし基準部屋で V=74, S=109, 接続点2 → A_total = 7.4+5.45+1.0 = 13.85 個/秒 → C_eq = 2.77（A行域）**、摩耗累計 ≈ A_total×t（t=300秒, ±10%）を使う。数値を調整する際は本書とフェーズ3 keystone の両方を同時更新すること。

## 5. 検出・再検出パラメータ（フェーズ1の確定値を本書に集約）

| パラメータ | 値 | 備考 |
|---|---|---|
| flood-fill 近傍 | **6近傍** | **6近傍では空気は斜めに移動できないため、エッジのみ接触の斜め壁は気密＝リークにならない**（設計書§2「コーナーだけで密閉した2壁片も自然に正しくなる」と整合。「斜め隙間＝リーク」という旧記述は誤りなので注意） |
| `MaxRoomVolume` | **4096** | 病的巨大ケースの安全網。超えたら invalid。根拠: 大部屋例 V=500（10×10×5）の8倍超を許容しつつ、未密閉構造のリーク探索コストを抑える。大部屋戦略を殺さないかプレイテストで再評価 |
| 機械・コンベア・エアフィルターの体積算入 | **空セルを塞ぐ（占有セルは V に算入しない）** | 機械を詰めると実効 V が減り濃度が跳ねやすくなる＝詰め込みの代償。**ただし部屋の `Cells`（帰属判定用セル集合）には占有セルを含める**: flood-fill は内部ブロックを通過し、`Cells`＝全内部セル、`Volume`＝`Cells` のうち空セル数。機械の「部屋内」判定は Cells で行う（V から除外したセルの機械が「室外」扱いになってはならない）。内部ブロックの設置/削除でも V が変わるため dirty 対象 |
| 部屋結合時の N | **合算** | |
| 部屋分割・形状変更時の N | **N_new = Σ N_old·overlap/\|Cells_old\|**（旧部屋との重なりセル数比） | **縮小＝濃度保存（除去セル分の N は消滅）／拡張＝N 保存（新規セルは清浄空気扱い＝希釈）**。「各新部屋 N = C·V_new」は縮小・等分割の特殊形でしかない点に注意。**按分の分母は V（空セルのみ）ではなく \|Cells\|（占有セル含む全内部セル数）**: overlap は Cells 同士で数えるため、分母を V にすると機械入り部屋で Σ寄与 > N_old となり保存則が壊れる（フェーズ2 Task 2）。閾値付近の揺れはヒステリシスで吸収 |
| ロード時 room 対応 | **保存 Cells と再検出部屋の最大セル重なり** | コードマップ§1.2/1.3。複数レコードが同一部屋にマッチした場合は**合算**（後勝ち上書きで N を消さない。結合の規則と一貫させる） |
| 再検出 dirty 処理の1tick上限 | **8192 セル/tick** | 同期即時全走査は禁止。超過分は次tickへ繰越。**担当: フェーズ2（データストア化）** |
| リーク判定境界の局所化 | **「触れた壁AABB+1」。担当: フェーズ2（データストア化）** | 設計書「リーク判定境界の局所化」節。フェーズ間で先送りループにしない（フェーズ2の DoD に含める） |

## 6. セーブ形式（純度状態）

`CleanRoomSaveData`（1部屋1レコード。コードマップ v2 の命名）:

| フィールド | 型 | 内容 |
|---|---|---|
| `impurityCount` | double | N |
| `thresholdIndex` | int | 現在の閾値行インデックス（ヒステリシス保持用。クラス列挙は持たない） |
| `status` | int | Valid/Degraded/Invalid |
| `graceRemainingSeconds` | float | 猶予残（Degraded時） |
| `cells` | int[][]（x,y,z の配列） | 同一性照合用の全セル署名（初期は全セル。重ければアンカー方式へ） |

- V/S は再検出から再導出するため保存しない。I/O 中継中アイテム/流体は各ブロックの `IBlockSaveState`（グローバルスキーマ非改変、`FuelGearGeneratorItemComponent` と同方式）。
- **Degraded 中（壁が壊れた直後）の孤立状態も保存対象に含める**。検出中の部屋だけ保存すると、猶予中にセーブ→ロードで N が消え「猶予内再封で N 継続」と矛盾する。status/graceRemainingSeconds フィールドはこのために存在する（検出中の部屋は定義上 Valid なので、保存しないならこの2フィールドは死にフィールドになる）。
- **Invalid 化した孤立状態は次の再検出時に破棄する**（無期限に溜めない。何時間も前に解体した部屋の汚染が跡地の新部屋へ「転生」してはならない）。
- **ロード→再検出→Restore の順序に注意**: Restore 後の最初の tick で再度全再検出が走って N・閾値行インデックスがリセットされる事故（dirty フラグ未クリア）を非回帰テストで防ぐ。ロード直後に1tick進めてもクラス相当（閾値行）と N が維持されることをアサートする。

## 7. 調整手順（テストでの固定方法）

1. フェーズ2テストは**基準部屋の C_eq とクラス**を上の worked example の値でアサート（係数を定数で注入）。
2. 数値変更時はこのテーブルと worked example を同時更新し、テストの期待値を1箇所で差し替える。
3. プレイテストでのチューニングは本書の初期値を起点に、「小部屋1台で最良／大部屋は増設」「汚い部屋ほどフィルター消費」の2不変条件を壊さない範囲で行う。

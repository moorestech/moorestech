# moores-code-review レビュー記録（ADR風）設計

日付: 2026-07-23 / 裁定: sakastudio（会話内AskUserQuestion 2件）

## 目的
moores-code-review実行ごとに「何をレビューし・何を指摘し・何を適用し・何を裁定したか」を
不変の記録として残す。PR1045では一時ファイル削除により事後監査がセッションログ考古学になった。
ADR（Architecture Decision Record）の原則（決定＋理由＋結果・コードと同居・不変・小さくシンプル）を適用する。

## 決定事項
1. **置き場所**: `.claude/skills/moores-code-review/records/YYYY-MM-DD-<topic>.md`（1レビュー実行=1ファイル）。
   skills-sync既存パイプラインでリポジトリ管理され、機能PRのdiffを汚さない（ユーザー裁定）。
2. **diffは保存しない**: `base SHA`＋`reviewed head SHA`の2ハッシュのみ記録（ユーザー裁定）。
   eval/fixtures.tsvの`base=`直指定と同形式のため、記録からリプレイfixtureを再生成できる。
   - 未コミット変更込みレビュー時は「dirty込み・ハッシュのみでは完全再現不可」と注記し`git diff --stat`要約を残す。
   - レビュー後の修正適用でheadが動くため「レビュー対象head」と「修正適用コミット」を別項目で記録する。
3. **内容は統合結果＋系統別1行判定表**: raw全文は保存しない。各系統の`Critical有無＋要旨1行`と
   セッションIDを残す（深掘りはセッションログで可能）。AskUserQuestion裁定・適用修正・破棄指摘は必須セクション。
4. **不変性**: 記録は事後改変しない。同ブランチ再レビューは`-r2`付き新ファイル＋相互リンク。
   マージ後に判明した事後結果（人間レビュー指摘等）のみ「事後結果」セクションへ追記可（ADRのコメント追記に相当）。
5. **フロー組み込み**: SKILL.md Step 7の一時ファイル削除を「記録生成→削除」に変更。新機構は作らない。
6. **eval/log.mdとの関係**: log.mdは集計＋記録ファイルへの相対リンクのみ（SSOT、内容二重化なし）。

## フォーマット
`records/TEMPLATE.md` を正とする（対象 / 系統別判定 / 適用した修正 / 設計判断 / 破棄した指摘 / 事後結果 / メタ）。

## 初号記録
PR1045のレビュー（2026-07-22実行）を、セッションログから復元した素材で遡及作成し、テンプレートの実例とする。

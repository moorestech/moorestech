## 1. ゲーム概要

半導体チップ生産に「空気純度」を導入する。プレイヤーは壁で密閉した部屋を建て、電力とフィルターを消費する空気清浄機で不純物濃度を下げる。部屋の清浄度クラスが、その部屋の半導体製造機が出せるチップの最大レベルと、汚れによる格下げ率を決める。

プレイヤーに要求する判断は次の3つ。

- 部屋の広さ: 大部屋は搬入出バーストに強いが、体積比例の基礎汚染と換気要求で維持費が重い。小部屋は安いが濃度が跳ねやすい
- 開口部の数: ハッチ・パイプ・ドアは接続点として恒常汚染を出す。搬送口を減らして集約するほど綺麗
- 清浄機への投資: 台数・電力・フィルター補充が上位チップの生産レートを決める

## 2. 部屋の検出

- クリーンルーム境界ブロック（壁・ドア・アイテムハッチ・パイプハッチ）で完全に囲まれた3D空間を1つの部屋として検出する
- flood-fill は6近傍。斜めの隙間はリーク（密閉は面で塞ぐ）
- 境界ブロック以外のブロック（ベルト・機械等）は空間を密閉できない。壁の穴にベルトを置いても fill が素通りしてリークになる。アイテムの出入りは必然的にハッチ経由になる
- 機械等の非境界ブロックの占有セルは部屋の内部セルに含まれるが、体積 V（空セル数）には算入しない。機械を詰めるほど実効 V が減り、濃度が跳ねやすくなる
- リーク判定: fill 中に実際に触れた境界セルの外接箱(+1)から fill がはみ出したらリーク。部屋の内部セル数上限 4096 を安全網とする
  - 根拠: この方式は「その部屋を実際に囲う壁」だけで探索を縛れるため、隣接部屋・遠くの装飾壁の影響を受けず、未密閉構造の探索コストだけを抑えられる
- 再検出は同期即時の全走査を禁止する。ブロック設置/破壊の影響セルをシードとして積み、毎 tick 予算（8192 セル）内で局所的に差分更新し、超過分は次 tick へ繰り越す
- 部屋の状態は「密閉成立（クラス付き）」か「不成立」の2値。猶予タイマ・中間状態は存在しない

## 3. 純度シミュレーション

各部屋は不純物総数 N（個, double）を持ち、濃度 C = N / V（個/m³）を派生する。

### 汚染（毎秒加算）

```
A_total = A_machine × 稼働中機械数
        + k_hatch  × ハッチ搬送レート(個/秒)
        + a_volume · V + a_surface · S + a_connector · 接続点数
```

- S = 空セルが境界ブロックに接する面の数
- 接続点 = 境界のうち壁以外（ドア・アイテムハッチ・パイプハッチ）の個数
- ハッチ搬送レートは直近1秒窓の実搬送個数から算出する。アイテム1個ごとの固定加算にはしない（ベルト速度やスタックサイズで回避できてしまうため）
- ドアの通過バーストは存在しない。ドアは接続点として恒常汚染に寄与するのみ
- 1つの境界ブロックが2部屋に面する場合（共有壁のハッチ等）、搬送汚染・接続点数は面する全部屋にフル計上する

### 除去と平衡

- 空気清浄機は毎秒一定体積 q(m³/秒) を処理し、除去レート = n·q·C。固定個数除去にはしない
  - 根拠: 固定除去は dN/dt が定数になり「0固定か発散か」に崩壊して多段クラスが成立しない。体積処理式は平衡濃度 C_eq = A_total/(n·q) に安定収束する
- 毎 tick の積分: N' = max(0, N + (A_total − n·q·C) × Δt)。Δt = 50ms

### クラス判定（二条件＋ヒステリシス）

クラスは A > B > C > D > Out。成立条件は次の2つを両方満たす最良クラス。

```
C ≤ 濃度閾値(クラス)  かつ  ACH = n·q / V ≥ 必要換気(クラス)
```

- ACH 要求により、大部屋ほど大きな清浄能力が必要になる（局所流体シミュなしで「大部屋の代償」を作る）
- ヒステリシス: 降格は素の閾値で即発火。昇格は「濃度 ≤ 閾値×0.8 かつ ACH ≥ 必要換気×1.25」を満たして初めて発火（電力変動等による ACH 起因の点滅も防ぐ）

## 4. ブロック

### 境界ブロック4種

| ブロック | 密閉 | 追加挙動 |
|---|---|---|
| クリーンルーム壁 | ✔ | なし |
| クリーンルームドア | ✔ | プレイヤー通行（クライアント側挙動）。サーバー的には接続点1 |
| アイテムハッチ | ✔ | 壁越しアイテム中継（下記） |
| パイプハッチ | ✔ | 壁越し流体中継（既存流体パイプ相当の容量を持つ） |

### アイテムハッチ

- 既存のインベントリ接続システムに乗る。マスタの inventoryConnectors で入力面/出力面を向き付きで定義（一方通行。逆向きが要るならもう1個置く）
- 挿入は中継バッファ（上限4スタック）へ積み、満杯なら受け取り拒否＝上流のベルトが詰まる
- 毎 tick、出力面の接続先へ押し出す。押し出した個数を直近1秒のリング窓で集計し、搬送レートとして汚染計算へ公開する

### 空気清浄機

- 電力駆動（既存 ElectricMachine 系の消費者として電力網に自動接続）
- 除去能力 = q × (供給電力 / 要求電力)。複数台は加算
- フィルターは仕事量ベースで消費: 自台が除去した不純物の累計が filterCapacity に達するごとにフィルターアイテムを1個消費。フィルター切れ（在庫0）の台は除去能力0
  - 平衡時のフィルター消費は A_total に比例する＝汚い部屋ほどフィルターを食う。開口部を減らす動機が直接生まれる

### 半導体製造機（クリーンルーム機械）

- 部屋状態の影響を受ける専用機械。レシピ処理は既存機械と同じ（入力→時間→出力、電力、モジュール対応）
- ステートは Idle / Processing / **Halted（稼働不能）** の3値。Halted は Idle とは別の明示的な状態
- Halted 遷移条件: 自機の全占有セルが同一の密閉部屋に属していない、または属する部屋のクラスが Out
- Halted 中: 新規レシピを開始しない。加工途中だった進捗と投入済み材料は保持する（プレイヤーの材料を消さない）。要求電力は0（電力を消費しない）。条件が回復したら Processing/Idle へ復帰し、途中だった加工は続きから再開する
- クライアントへは既存の BlockStateDetail 経路で Halted を通知する（表示の作り込みは対象外）

## 5. チップ抽選

- チップのグレードは独立 ItemId（Lv1〜Lv4）。ItemStack に品質フィールドは持たせない
- 3レバーの直交分離:
  - 部屋クラス → 最大レベル天井 ＋ 格下げ率（down-bin）
  - 品質モジュール → 天井内で分布を上位へ寄せる（将来。今回のスコープでは中立）
  - EUV 失敗率 → 出力なしの割合
- 抽選の処理順: ①EUV失敗判定（失敗なら出力なし） → ②クラスの天井でレベル分布を切り詰めて再正規化 → ③基礎分布から1レベル抽選 → ④格下げ率で1段格下げ（Lv1は据え置き） → ⑤確定
- 抽選はサイクル完了ごとに、blockInstanceId とセーブされるサイクルカウンタから導出した決定的シードで行う（セーブ/ロードで結果が再現される。共有 Random は使わない）
- レベル分布はマスタデータ（machineRecipe の出力アイテム単位）。分布定義が無い出力アイテムは抽選対象外の副産物としてそのまま出す
- アイテムを部屋の外に出しても変化しない。純度は「製造の瞬間に参照される部屋の属性」であり、アイテムの状態ではない

## 6. 部屋の同一性と永続化

- 部屋 ID は揮発。永続キーにしない
- 再検出をまたぐ状態引き継ぎはセル重なりで行う: 結合は N 合算、分割は N_old × 重なりセル数 / 旧セル数 で按分（N の総量を保存）。クラス行は最大重なりの旧部屋から引き継ぐ
  - 目的: 部屋の中にブロックを置いた・間仕切りしただけでクラスが Out に戻る事態を防ぐ
- 密閉が崩れて部屋が消えたら、その部屋の状態は即消滅する。猶予・幽霊・後からの復活は無い。再密閉した部屋は N=0・クラス Out から証明し直し
- 新規に検出された（引き継ぎ元の無い）部屋は N=0・クラス Out で開始する
- セーブ: 部屋ごとに N・クラス行・全セル座標。V/S は保存せず再検出から再導出。ロードはブロック復元 → 全再検出 → 保存レコードを最大セル重なりで照合して N・行を復元
- ハッチの中継バッファ内アイテムは各ブロックのセーブ状態として保存する

## 7. マスタデータ

- クラス閾値表（濃度上限・最大レベル・格下げ率・必要換気）はマスタデータ（新規スキーマ）
- チップレベル分布（レシピ×出力アイテム → レベル別重み）はマスタデータ（新規スキーマ）
- 汚染係数・清浄機性能などの数値も可能な限りマスタ（ブロックパラメータ等）に置き、コード内ハードコードの定数を残さない
- 新マスタが未定義の Mod でも起動はできること（クリーンルーム機能が無効なだけ）

## 8. 初期パラメータ（調整前提の出発値）

| クラス | 濃度上限(個/m³) | Lv天井 | 格下げ率 | 必要換気(1/秒) |
|---|---|---|---|---|
| A | 10 | Lv4 | 0% | 0.017 |
| B | 50 | Lv3 | 5% | 0.0083 |
| C | 200 | Lv2 | 15% | 0.0042 |
| D | 1000 | Lv1 | 35% | 0.0014 |
| Out | >1000 | 稼働不能 | — | — |

- 汚染係数: a_volume 0.10 / a_surface 0.05 / a_connector 0.50 / A_machine 2.0 / k_hatch 0.30
- 清浄機: q=5.0 m³/秒（満電1台）・フィルター容量 5000
- EUV 失敗率: 20%（レシピ側パラメータ）
- 昇格ヒステリシス: 濃度×0.8・ACH×1.25
- 検出: 部屋上限 4096 セル・再検出予算 8192 セル/tick・ハッチレート窓 1.0 秒・中継バッファ4スタック

検証用の基準例（テストで固定する）:

- 内寸 5×5×3（V=75, S=110）、接続点2、稼働機械1、清浄機1台満電 → A_total = 2.0 + 7.5 + 5.5 + 1.0 = 16.0 個/秒 → C_eq = 3.2 → クラスA成立（ACH 0.067 ≥ 0.017）
- 同構成で 10×10×5（V=500, S≈400）に拡大 → C_eq = 14.6 でクラスB相当、かつ ACH 0.01 < 0.017 で A 不成立 → 増設が必要

数値調整で壊してはいけない2不変条件: 「小部屋は清浄機1台で最良クラスに届く」「大部屋は増設しないと最良クラスに届かない」。

## 9. 受け入れ条件

1. 上記の基準例2つがテストで固定されている
2. 全機能が実プレイ経路で発火する。「テストからしか呼ばれない機能」「将来の統合を待つ seam」を残さない
3. 録画付きプレイテストで、部屋の建設 → クラス成立 → 機械稼働 → 壁破壊で Halted → 再密閉で復帰、が動画として確認できる
4. 新マスタ未定義の既存 Mod・cleanRoom キーの無い既存セーブがそのまま起動・ロードできる

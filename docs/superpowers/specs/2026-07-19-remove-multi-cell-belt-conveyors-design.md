# n連ベルトコンベア削除設計

## 目的

ベルトコンベアの2連・3連・4連・5連という長尺バリアントを、マスター、実行時システム、テスト、移行ツール、Prefabから削除する。ベルトコンベアファミリーマスターは、直線・上り・下りの対応関係を表す用途に限定して残す。

後方互換は設けない。既存の長尺ベルトを含むセーブデータは移行対象外とする。

## マスタースキーマ

`beltConveyorFamilies`の`straightBlocks`配列を、必須の`straightBlockGuid`へ置換する。1ファミリーが保持するブロックは次の3種類だけとする。

- 必須の1セル直線ブロック
- 任意の1セル上りブロック
- 任意の1セル下りブロック

配列とブロック長の概念を型からなくし、ファミリーマスターだけでは複数の直線バリアントを表現できないようにする。

## マスターデータ

スキーマを使用する全JSONを`straightBlockGuid`へ一括更新する。対象はサーバー、クライアントテスト、Sandbox、および`moorestech_master`の`server`から`server_v8`までの全世代とする。

現行マスターから長尺ベルトブロック12件、単体テストマスターから長尺ベルトブロック2件を削除する。削除対象GUIDへの参照は、ファミリー、破壊カテゴリなど参照元の種類を問わず同時に削除する。n連導入専用の`tools/belt_variant_migration`もディレクトリ単位で削除する。

## ドメインモデルとサーバー

`BeltConveyorFamily`は`StraightBlockId`、`UpBlockId`、`DownBlockId`だけを保持する。`RepresentativeBlockId`、長さ付き直線バリアント一覧、長尺非表示判定は削除する。

ファミリー解決は3つのGUIDを直接照合する。ビルドメニューとブロックピックでは、上り・下りだけを非表示または直線ブロックへ正規化する。設置プロトコルのアンロック判定も、ファミリー所属ブロックを直線ブロックへ正規化して判定する。

バリデーターは次を保証する。

- 各ファミリーに必須の直線ブロックが1件ある
- 直線・上り・下りはすべて`blockSize == [1,1,1]`
- ファミリーメンバーはベルト系ブロックだけである
- 同じブロックは複数ファミリーに所属しない
- すべてのベルト系ブロックはちょうど1ファミリーに所属する

## クライアント配置

水平ランの検出、長さ降順の貪欲割当、マルチセル原点計算を削除する。経路計算が返す各セルを、水平なら`StraightBlockId`、上りなら`UpBlockId`、下りなら`DownBlockId`へ1対1で変換する。

坂を持たないファミリーで傾斜セルが生成された場合は、そのセルを直線ブロックの設置不可プレビューとして残す。既存の連続ドラッグ配置、衝突判定、所持アイテム数判定、サーバー送信は維持する。

## Prefabと共有アセット

次の長尺専用Prefab 8個をUnity Editorの`AssetDatabase.DeleteAsset`経由で削除する。

- `BeltConveyor_Straight 2`から`BeltConveyor_Straight 5`
- `gear belt conveyor 2`と`gear belt conveyor 3`
- `metal_gear belt conveyor 2`と`metal_gear belt conveyor 3`

UnityにPrefabと`.meta`を一緒に削除させ、Addressables参照もEditor管理下で更新する。単連Prefabも利用しているFBX、Material、Textureは残す。

## テストとQA

長尺分解テストと長尺返却テストは削除する。代わりに、水平セルの1対1変換、上り・下り変換、坂なしファミリーの設置不可化、ファミリー正規化、1セルサイズ制約をテストする。

QAでは問題がある前提で、次を順に確認する。

1. 旧スキーマ名、長さバリアント型、長尺クラス名、旧GUID、長尺Prefabアドレスの残存検索
2. Source Generator再生成を含むUnityコンパイル
3. ベルト配置、ファミリー、設置・解体プロトコルの限定テスト
4. クライアントプロジェクトの全テスト
5. Unity ConsoleのErrorログ

## 反例と再混入防止

`blockSize == [1,1,2]`のベルトを追加するケースを最強の反例とする。ファミリー内へ追加しようとしても、単一GUID契約により既存直線との併記はできず、既存直線と置換しても1セルサイズ検証で拒否される。ファミリー外へ追加しても、全ベルト系ブロックの所属必須検証で拒否される。

これにより、n連を再導入するにはスキーマとバリデーターを明示的に変更する必要があり、データ追加だけで概念が復活することはない。

# アップグレードシステム 設計仕様

- 日付: 2026-06-05
- 対象: moorestech 全ゲーム共通の機械アップグレード（モジュール）基盤
- 起点: V8 mod (`moorestechAlphaMod_8`) の nodeGraph 半導体エリアに散在する「品質モジュール / 歩留まり改善モジュール / チップアップグレードモジュール / チップのレベル概念」メモ

---

## 1. 目的とスコープ

全機械共通で挿せる「アップグレードモジュール」基盤を作る。
モジュールを機械内の専用スロットに挿すと、その機械のレシピ処理の挙動（速度・産出・消費）や産物の品質（レベル）が変わる。

半導体エリアの品質モジュール等は、この汎用基盤の上に載る一適用例として扱う。

スコープ外（今回作らない）:
- ブロック置換型ティア強化（Mk1機械→Mk2機械への建て替え）
- 隣接ビーコン型の範囲効果
- モジュールの消費品（燃料）化

---

## 2. 全体構造：2レイヤー

```
┌─ モジュール基盤（汎用・全機械共通）────────────────────┐
│  機械ブロックが N 個のモジュールスロットを持つ              │
│  スロットにモジュールアイテムを挿す → 機械の挙動が変わる       │
│                                                          │
│  効果軸A（機械処理系）: 速度 / 生産性 / 省エネ              │
│    └ アイテムの正体に触れない。レシピ処理パラメータを変更      │
│                                                          │
│  効果軸B（品質系）: 出力アイテムのレベル分布を変更            │
│    └ 下のレベルファミリー機構に依存                         │
└────────────────────────────────────────────────────────┘
┌─ レベルファミリー機構（品質軸が使う・オプション）──────────┐
│  「レベルを持つアイテム」をスキーマで family 定義             │
│  → SourceGenerator が Lv1..LvN の独立アイテムを自動生成       │
│  → 「LvK × M 個 → Lv(K+1)」の合成レシピも自動生成            │
└────────────────────────────────────────────────────────┘
```

**設計の核心：基盤と品質を分離する。**
速度/生産性/省エネだけ使う機械は、レベルファミリー機構に一切依存しない。品質軸はオプションで載る能力。

ただし**完全分離ではない**点に注意：品質モジュールは速度ペナルティを持つ（=Aの処理時間計算に入る）し、品質は出力 ItemId を変える（=出力容量判定に入る）。そのため A の段階で、効果集計の統一結果（`EffectiveMachineParameters` 相当：処理時間/消費/産出/`qualityShift`/出力変換フック）に **品質用のフックの置き場所だけは確保**しておく。Bを後付けする際にAを作り直さないため。

### なぜ品質だけ別レイヤーなのか（決定の根拠）

効果軸は性質が2グループに分かれる:

| グループ | 軸 | 何に作用するか | 難易度 |
|---|---|---|---|
| A. 機械処理系 | 速度 / 生産性 / 省エネ | レシピ処理（時間・成功率・消費）。アイテムの正体に触れない | 軽い |
| B. 品質系 | 品質 / レベル | 産物の「レベル違いアイテム」をどう表現するかを強制する | 重い |

「アップグレードシステム全般」の大部分（A）は素直に作れる。ハードな判断（後述のレベル表現）は品質軸1つだけの問題で、基盤全体に波及させない。

## 3. レベル表現の決定：独立 ItemId（メタデータではない）

レベル違いアイテムを「別 ItemId」とするか「ItemStack メタデータ」とするかは設計上の分岐点。**独立 ItemId を採用する。**

### コードベースの事実（確認済み）

- `IItemStack` に `ItemStackMetaData` のスケルトンが存在するが、コメント通り「概念だけ存在、実際には使われていない」。
- `ItemStack.IsAllowedToAdd`（スタック結合判定）は `Id` のみ見てメタデータを見ない → 現状ではレベル違いも同一スタックに混ざる。
- セーブ `ItemStackSaveJsonObject` は `itemGuid` + `count` のみ。メタデータは永続化されない。
- 通信 `ItemMessagePack` は `Id` + `Count` のみ。メタデータは同期されない。
- レシピマッチングは `ItemId` 基準。

→ メタデータ路線は**「部分対応があるぶん、むしろ危険」**。`ItemStack.AddItem`/`Equals` は meta を比較するが、`IsAllowedToAdd`/`IsAllowedToAddWithRemain` は `Id` のみ・通信は `Id+Count` のみ・セーブは meta なし。一部だけ meta を見るため、永続化・ネット同期・レシピ照合・スタック結合判定をコアの最もデータ破損/desync に弱い経路でゼロから整合させる必要があり、半端な対応が潜在バグになる。

### 決定の決め手

下流レシピはレベルを**常に使用時点で意味を持つ**形で消費する（高Lvは高tier専用）。
「どのレベルでも可」消費が頻出しないため、メタデータ案の唯一の利点（"任意レベル一致"がタダ）が効かない。

### 採用案：独立 ItemId ＋ スキーマ自動生成

- レベルごとに別アイテム（`チップLv1` / `チップLv2` …）。
- スタック・セーブ・通信・レシピ照合は既存のまま全部動く＝コア改修ほぼゼロ。
- 唯一の弱点（N×種類ぶんのアイテムと「Lv1×N→Lv2」レシピの手書き）は、スキーマに「レベルファミリー」定義を追加し SourceGenerator で変種アイテムと合成レシピを自動生成して解消する。
- 想定 N ≈ 5 なので ID 数は爆発しない。

## 4. データ定義（マスタ / スキーマ）

### 4.1 新スキーマ `modules.yml`

各モジュールを定義する。モジュールは普通のアイテムでもある（インベントリに持てる・作れる）。

- `moduleGuid`
- `itemGuid`（対応するアイテム。インベントリ上の実体）
- `effectAxis`: `Speed` | `Productivity` | `Efficiency` | `Quality`
- `tier`: Mk1〜（上位ほど高効果）
- `effectValue`: 主効果量（%）
- `tradeoff`: トレードオフ量（軸ごとに意味が決まる。下表）
- （品質のみ）`qualityShift`: レベル抽選テーブルに対する確率シフト量

### 4.2 `blocks.yml` に `moduleSlotCount` 追加

機械ブロックごとにモジュールスロット数をマスタで定義する。
- `0` ならモジュール非対応。
- 上位tier機械ほどスロットが多い、といった差別化が可能。

### 4.3 レベルファミリー定義（`items.yml` または新規スキーマ）

- `baseItem`: 基準アイテム
- `maxLevel`: 最大レベル N
- `combineCount`: Lv上げに必要な個数 M（LvK × M → Lv(K+1)）

SourceGenerator が `baseItemLv1..LvN` の独立アイテムと、各段の合成レシピを自動生成する。

**重要（セーブ互換）**: セーブは `itemGuid + count` のみで識別するため、Lv派生アイテムの GUID は**決定的に生成**しなければならない（`baseGuid + level` から導出、またはマスタに固定値で記録）。ランダム GUID 生成は禁止。毎回変わるとセーブが即破損する。

## 5. モジュールの効果モデル

### 5.1 効果軸とトレードオフ

| モジュール | 主効果 | トレードオフ |
|---|---|---|
| 速度 (Speed) | 処理時間 −% | 電力/トルク消費 +% |
| 生産性 (Productivity) | 機械のレシピ種別で2モード: 確率出力機械→成功率↑ / 確定出力機械→確率で追加産出 | **処理速度 −%（Factorio準拠）** |
| 省エネ (Efficiency) | 消費 −% | （単独では弱、速度の相殺用） |
| 品質 (Quality) | 高レベル産物の出現率 +% | 処理速度 −% |

### 5.2 スタック（重ねがけ）の合成

- 同一機械内の複数モジュールの効果は**加算**（速度Mk1×2 = 効果合計）。
- スロット数が有限なので「速度を盛るか品質を盛るか」がレイアウト最適化の判断になる。

### 5.3 数値不変条件（clamp）— 必須

加算は単純合算のままだと必ず破綻する（処理時間0以下・消費0以下・成功率100%超・品質分布の負確率）。これはバランス問題ではなく**不変条件の問題**。効果集計の最終段で必ず clamp する:

- `processingTicks >= 1`（0以下にしない）
- `powerMultiplier >= 設定下限`（消費を0以下にしない）
- `0 <= 確率 <= 1`（成功率・追加産出確率）
- `Σ レベル分布 = 1`（品質シフト後に正規化、負確率を出さない）

clamp の下限値は modules.yml / グローバル設定で定義する。

### 5.4 効果のスナップショット方針 — 必須

現状の機械処理は開始時に `_processingRecipeTicks` を固定し完了時に出力する（`VanillaMachineProcessorComponent`）。処理途中でモジュールを抜き差ししたときの挙動を定義しないと「開始だけ速度盛り→完了直前だけ品質盛り」の exploit が出る。

**決定：効果は処理開始時にスナップショットする。** 処理中のスロット変更は次の処理サイクルから反映。処理中はスロットを読み取り専用ロック（または変更を次サイクル予約）にする。

### 5.3 tier

- 各効果軸に Mk1〜Mk3 等の段階。上位ほど主効果が高い。
- 製造コストと研究で解放をゲートし、長期的な成長軸にする。

## 6. 機械への組み込み（実装方針）

> **設計改訂（2026-06-06・ユーザーレビュー反映）:** 当初は「モジュールを既存 `VanillaMachineBlockInventoryComponent` の連結ビューに混ぜず、独立した専用サブインベントリ＋専用同期/移動経路にする」方針だった（下の旧記述）。しかし新規プロトコル・新 `InventoryType`・別コンポーネントを増やす過剰設計だった。**改訂後は、既存の統一スロット番号（input→output に続く第3レンジ＝module）としてモジュールスロットを足す。** これで同期（`InventoryRequestProtocol`）・移動（`InventoryItemMoveProtocol` ＋ `InventoryType.Block`）・セーブが既存のまま使え、新規はほぼゼロになる。

**改訂後の方針（採用）:**
- モジュールスロットは `VanillaMachineModuleInventory`（入力/出力と同じ作り）として機械インベントリに足し、`VanillaMachineBlockInventoryComponent` のスロット番号ルーティング末尾に**第3レンジ**として連結する。
- 旧§6が挙げた各干渉点は、第3レンジを**末尾に置く**ことと小さな対処で解消する:
  - **出力slot offset**: モジュールは output の**後ろ**に足すので、既存の input/output の slot index は不変。
  - **通常搬入**: 搬送系の `InsertItem` は入力サブインベントリにしか入らないため、モジュールスロットは自動投入で埋まらない（追加対処不要）。
  - **手動移動**: 挿入制限は設けず、既存の `InventoryItemMoveProtocol`／`InventoryItemMoveService` を**無改変**で使う（ユーザーレビュー第2弾）。非モジュールがスロットに入っても、効果コンポーネントがモジュールだけを数えるので無視されるだけ（出力スロットが任意投入を許す既存挙動に揃える）。よって移動プロトコルにもインベントリにも per-slot ガードを足さない。
  - **整理（InventorySortService）**: ⚠ 要対処。`InventorySortService.Sort` は `Enumerable.Range(0, GetSlotSize())` で全スロットを並べ替え `SetItem` で書き戻すため、放置するとモジュールが整理で散らばる。ただし**同サービスは既に `excludeSlots` 引数を持つ**ので、module レンジを除外指定するだけでよい（新機構は不要）。
  - **UI・購読同期**: モジュールスロットも全スロット同期に自動で乗る。クライアント側は末尾 `moduleCount` 個を専用の行として描画する。
- モジュールスロットへの**通常搬入は禁止**（上記のとおり自動で満たされる）。プレイヤーの明示装着のみ。
- 永続化は既存の機械インベントリ保存（`VanillaMachineSaveComponent`）にモジュールスロットを含める。アイテムは独立 ItemId なので既存のアイテム保存スキーマに手を入れない。

  - 産出（生産性の追加産出 / 成功率）
  - 出力レベル分布（品質）

## 7. 品質軸の動作（メモ準拠）

- レベル抽選を持つ機械（露光装置等）は出力レベル分布テーブルを持つ。例: `Lv1:80% / Lv2:18% / Lv3:2%`。
- 品質モジュールを挿すと、低レベルの確率を圧迫して高レベル率を底上げ（`qualityShift`、重ねがけで +%）。
- 完成時にチップのレベルが確定する（メモ「最後の一層を完了させる時にチップのレベルが確定する」に対応）。
- 「Lv1 × M個 → Lv2」はレベルファミリー機構の自動生成レシピで別途実行可能。
- 半導体特有の「歩留まり改善モジュール（失敗パターン率を下げる）」は、効果軸 = 生産性（成功率↑）として表現する。

### 7.1 生産性の追加産出と出力容量 — 必須

現状は開始前にベースレシピ出力だけ `IsAllowedToOutputItem` で確認し、完了時に出力する。追加産出が確率で発生すると、開始時の容量チェックを超える可能性があり、放置するとアイテム消失か停止不能バグになる。

**決定：開始前に「最大追加産出まで」を出力容量に予約してから処理を開始する。** 予約できなければ処理を開始しない（既存の Idle 条件に最大見積もりを加える）。完了時に実抽選結果を出力。

### 7.2 抽選順序の決定性 — 必須

品質と生産性の相互作用・乱数順序を固定しないと、セーブ/リプレイ/テストの決定性が壊れる。**順序を固定する**:

1. レシピ完了
2. 生産性で出力個数を決定（成功率 or 追加産出）
3. 各出力アイテムに対し品質（レベル）を決定

追加産出分にも品質抽選を行う（主産物と独立に抽選）。乱数源は決定的なシード経路を使う。

## 8. ユニット分割（責務境界）

| ユニット | 責務 | 依存 |
|---|---|---|
| `ModuleMaster` | modules.yml のロードと参照 | MasterHolder |
| モジュールインベントリコンポーネント | 機械のスロット保持・state 保存・ネット同期 | 既存ブロックインベントリ基盤 |
| モジュール効果集計 | スロット内容→効果係数（速度/消費/産出/品質シフト）に変換 | ModuleMaster |
| レシピ処理への適用 | 集計済み係数を処理時間・消費・産出・レベル分布に反映 | 既存機械処理 + 効果集計 |
| レベルファミリー生成（SourceGenerator） | family 定義→変種アイテム + 合成レシピ自動生成 | スキーマ |
| レベル抽選 | 出力レベル分布テーブル + 品質シフト → 抽選 | 効果集計 |

各ユニットは独立して理解・テスト可能。効果集計はモジュール集合を入力、係数を出力する純粋関数に寄せてテストしやすくする。

## 8.5 フェーズ分割

実装難易度が違うため **A先行 → B（品質）追加** の2フェーズに割る。

**フェーズA（基盤）の完了条件**（「速度/生産性/省エネが動く」だけでは不足）:
- モジュール専用サブインベントリの保存／同期／移動制限（通常搬入禁止・不正挿入拒否）
- 処理開始時の効果スナップショット（5.4）
- 加算後の clamp（5.3）
- 生産性追加産出の出力容量・最大見積もり予約（7.1）
- 効果集計の統一結果に品質フックの置き場所を確保（2章末）

**フェーズB（品質）**: レベルファミリー機構（SourceGenerator自動生成・決定的GUID）、レベル抽選テーブル＋品質シフト、抽選順序の決定性（7.2）。
Aでスロットや効果集計を雑に作るとB実装時に作り直しになるため、Aの完了条件は上記まで満たすこと。

## 10. テスト方針

- 効果集計の純粋関数: モジュール集合→係数のユニットテスト（加算・tier・トレードオフ）。
- 数値不変条件: 極端なモジュール構成でも処理時間≥1・消費≥下限・確率∈[0,1]・分布和=1 に clamp されることを検証（5.3）。
- 効果スナップショット: 処理中にモジュールを抜いても当該サイクルの効果が変わらないことを検証（5.4）。
- 出力容量予約: 追加産出の最大見積もりで出力が埋まる場合に処理が開始されない／アイテム消失しないことを検証（7.1）。
- 移動制限: モジュールスロットへの通常搬入・不正アイテム挿入が拒否されることを検証。
- レベル抽選: 固定乱数シードで分布が品質シフトに応じて変わることを検証。
- レベルファミリー生成: family 定義から期待どおりの変種アイテム数と合成レシピが生成されることを検証。
- 機械統合: モジュール挿入で処理時間・消費・産出が変わることのサーバー側統合テスト。
- セーブ/ロード: モジュールスロット内容がブロック state 経由で round-trip することを検証。

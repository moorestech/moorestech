# クリーンルーム（空気純度）システム 設計書

- 日付: 2026-06-05
- 対象: moorestech V8 mod（半導体／ICチップ生産）
- ステータス: 設計承認済み（数値バランスは実装プランで確定）。Codex 外部監査の指摘を反映した改訂版。

## 1. 概要

半導体（EUV露光〜ICチップ）生産に「空気純度」の概念を導入する。プレイヤーは壁で密閉したクリーンルームを建て、空気清浄機とフィルターで不純物濃度を維持する。部屋の清浄度（クラス）が、製造できる最大チップグレードと、汚れによる格下げ（ダウングレード）を決める。

設計の核となる決定:

- クリーンルームは **壁で完全に囲った3D密閉空間** として検出する。
- 純度は **不純物濃度（個/m³）** で表し、閾値で **クラス** が決まる。
- クラスは **製造できる最大チップグレードの天井** と、**汚れによる下位グレードへの格下げ率** を規定する（binning型）。
- 不純物は複数の汚染源から増え、空気清浄機（電力＋フィルター消費）で除去する。
- **体積・表面積に比例する基礎汚染** と **クラスごとの換気回数（ACH）要求** により、部屋を大きくする／小さくする意思決定に意味を持たせる。
- アイテムを部屋外へ取り出しても、そのアイテムは変化しない（純度は製造の瞬間にのみ参照される、部屋の状態属性）。

## 2. 部屋検出（3D密閉箱）

- **クリーンルームブロック（壁）** で完全に囲まれた立体空間を1つの部屋として検出する。
- 内部の空セルから3D flood-fill を行い、フィルが外部へ漏れる／規定の上限体積を超えた場合は **密閉していない＝部屋不成立** とする。
- 境界を貫通してよいのは **ドア／アイテムハッチ／パイプコネクタ** のみ。それ以外の壁の欠け（隙間）は即リークとして扱う。
- ブロックの設置・破壊のたびに、影響範囲の部屋を再検出する。
- 各部屋は不純物状態（不純物総数 N、体積 V、表面積 S）を保持し、セーブ／ロード対象とする。
- **クラス判定にはヒステリシスを入れる**（閾値付近でのクラス揺れ・点滅を防ぐ）。

### 実装前に確定すべき仕様（実装プランで詳細化）

- flood-fill の近傍: 6近傍か26近傍か。斜めの隙間をリークと見なすか。
- 機械・コンベア・パイプ・清浄機は内部空間（空セル）を塞ぐ扱いか、体積Vにどう算入するか。
- 内部パーティションで部屋が割れる条件。
- ハッチが2部屋の境界にある場合、2部屋を接続するのか隔離するのか。
- 部屋の結合時は N を合算、分割時は濃度を等分配でよいか（閾値付近のクラス揺れに注意）。
- ロード時に保存済み room id と再検出後の room id をどう対応づけるか。

### 再検出の実装方針（同期即時は禁止）

巨大な未密閉空間に flood-fill して上限まで走るケースが頻発するため、同期即時の全再検出は破綻する。以下を前提とする:

- 変更セルを dirty area として積む。
- 次tick以降に分割処理する。
- flood-fill が上限体積に到達したら invalid としてキャッシュする。
- 近傍の既知 room 境界からの差分更新を基本とする。

### リーク判定境界の局所化（フェーズ2最適化）

フェーズ1は「全境界ブロックの外接箱（グローバルAABB）の外に出たらリーク」で判定する。これは正しいが、遠くに無関係な境界ブロックがあるとAABBが膨らみ、未密閉構造のリーク探索が `MaxRoomVolume` 到達まで長く走る（正常な部屋は自分の壁で閉じるので影響を受けない／劣化するのは未密閉時の探索コストだけ）。フェーズ2で次のいずれかに局所化する。

- **連結成分AABB**: 境界ブロックを面隣接で連結成分に分け、各成分のAABBで内部 flood-fill を縛る。グローバルより大幅にタイト。ただし隣接2部屋が壁を共有すると成分が1つになりAABBが両部屋を覆う緩さが残り、装飾や通路で境界ブロックを長く繋げると成分AABBが伸びる。
- **触れた壁AABB（推奨）**: 事前の連結成分計算をせず、内部 flood-fill 中に**フロンティアで実際に触れた境界セルの外接箱（+1）**で縛る。fillセルがこのbboxの外に出たらリーク。「その部屋を実際に囲っている壁だけ」を見るため連結成分AABBより厳密に狭く、(a) 共有壁の隣接部屋は各部屋を自分の壁だけで縛れる、(b) 部屋外の装飾壁はfillが触れないのでbboxに入らない、(c) コーナーだけで密閉した（面非接続だが6気密の）2壁片も両方を触れてbboxを合体するため自然に正しくなる。

いずれも正しさは保たれ（正常な密閉部屋は自分の壁で閉じる）、損なうのは未密閉構造のリーク探索コストだけ。`MaxRoomVolume` は病的に巨大なケースの安全網として残す。「触れた壁AABB」は外への一歩を検知するため必ず **+1** して判定する。

## 3. 純度モデル

各部屋は **不純物濃度 C = 不純物総数 N ÷ 部屋体積 V（個/m³）** を持つ。

汚染の毎秒加算量 A は、イベント性の汚染に加え、**空間そのものに比例する基礎汚染**を含む:

```
A_total = A_machine + A_hatch + A_door
        + a_volume   · V              // 空間維持コスト（巨大空室のデメリット）
        + a_surface  · S              // 壁面・シール面からの漏れ
        + a_connector · 接続点数        // ハッチ/ドア/パイプコネクタの数
```

不純物総数の時間変化と平衡:

- **空気清浄機は「毎秒一定体積 q(m³/s) の空気を処理する」**。除去量 = `n台 × q × C(濃度)`。固定個数除去ではない。
- `dN/dt = A_total − n·q·C = A_total − n·q·(N/V)`。
- **平衡濃度 `C_eq = A_total / (n·q)`** に安定収束する。

### クラス成立の二条件（濃度＋換気回数）

クラスを成立させるには、平衡濃度の閾値に加えて **換気回数（ACH）要求** を満たす必要がある:

```
クラス成立条件:
  C ≤ classThreshold                         // 平衡濃度が閾値以下
  かつ
  ACH = n·q / V ≥ requiredAirChangeRate(class) // クラスごとの必要換気量
```

ACH 要求により、体積 V が大きいほど必要な清浄能力 `n·q` が増える。これが「大部屋の代償」を局所流体シミュレーションなしで作る。

### なぜ「体積処理式」か（固定個数除去を採用しない理由）

固定個数除去（除去 = n×R、濃度に依存しない定数）にすると、`dN/dt = A − n·R` が定数になり、
N は 0 にクランプ（常に最良クラス）か、無限大に発散（常に最悪クラス）の二択に崩壊する。
安定した内部値が存在せず、多段クラスが機能しない。体積処理式なら平衡濃度に収束し、クラスが連続スペクトルになる。

### 体積 V の役割（定常＋過渡の両方で効く）

- **定常**: `a_volume·V` の基礎汚染と ACH 要求により、大部屋ほど維持コスト（清浄機台数・電力・フィルター）が重い。
- **過渡（バースト）**: ドア開閉やハッチ搬入出の固定個数スパイクを、大部屋は薄めて吸収（許容的だが回復は遅い `V/(n·q)`）、小部屋は濃度が大きく跳ねる。

→ 大部屋＝バースト耐性は高いが空間維持コストと必要換気量が重い／小部屋＝安いがスパイクに弱い。3D体積判定がこの両面で意味を持つ。

## 4. クラスと効果（binning型・既存歩留まりレバーとの役割分担）

製造1回の結果は「❌不良（捨て）／レベル1／レベル2／レベル3…」のいずれか。3つのレバーが直交した役割を持つ:

| レバー | 役割 |
|---|---|
| **クリーンルームクラス（純度）** | 製造できる **最大グレードの天井** ＋ 汚れによる **下位グレードへの格下げ（down-bin）率** |
| **品質モジュール** | 天井の範囲内で **上位グレードへ寄せる分布** |
| **EUV 20%失敗 / 歩留まり改善モジュール** | **出力なしの catastrophic 失敗率**（不良＝捨て） |

処理順のイメージ:

1. EUV失敗（例: 20%）でまず一定割合が ❌不良 として脱落。
2. 残りを、クラスが決める最大グレードの範囲で生成。
3. 汚れ（クラス）に応じて一部を下位グレードへ格下げ。
4. 品質モジュールで天井内の分布を上位へ寄せる。

→ EUV＝捨てる率 / クラス＝上限＋格下げ / モジュール＝上位寄せ、と各レバーが「失敗・格付け・分布」に直交し、出力をプレイヤーが推論しやすい。

## 5. 汚染源（A_total に合算）

- **常時の自然増加**: 部屋ごと固定ではなく **体積 V・表面積 S に比例**（`a_volume·V + a_surface·S`）。巨大空室を不利にする。
- **製造機の稼働**: EUV 等の稼働中に不純物を排出（`A_machine`）。機械を詰めるほど汚染が増える。
- **ドアの開閉（人の出入り）**: プレイヤー通過で大きめのバースト加算（`A_door`）。自動化を促す。
- **ハッチのアイテム搬入出**: **単位時間あたりの搬送量で集計**する（`A_hatch`）。1アイテムごとの加算だとベルト速度・スタックサイズで不自然に回避できるため、レート換算にする。
- **接続点数**: ハッチ／ドア／パイプコネクタの数に比例（`a_connector·接続点数`）。搬送口を増やすほど汚れる＝集約の根拠。

## 6. 空気清浄機 ＋ フィルター

- 常時 **電力** を消費する（ElectricMachine 系）。
- **フィルターは仕事量ベースで消費する** ＝ 除去した不純物量に比例。
  - 平衡時のフィルター消費は総汚染レート A_total に比例する。
  - → 汚い部屋ほどフィルターを食う。ハッチ／ドア／接続点を減らす動機が直接生まれる。
- 複数台で除去能力 `n·q` が加算される。機械を詰めるほど、また体積が大きいほど（ACH要求）清浄機の増設が必要になる。
- 清浄機台数は「クラス閾値を超える最小台数」に収束しやすいが、ACH要求・電力・占有面積が追加の設計圧になる。さらに調整幅が欲しい場合は **高級フィルター種別**（除去効率の高い上位フィルター）を後から足せる。

## 7. I/O の役割分担

- **アイテムハッチ**: アイテムの搬入出。低スループット（集約の根拠）かつ汚染源。
- **パイプコネクタ**: 流体の搬入出（超純水・IPA 等）。
- **ドア**: プレイヤーの出入り。汚染が大きく、自動化を促す。

## 8. 製造機とクリーンルームの関係

- 半導体製造機（EUV 露光式半導体製造装置 等）は **有効なクリーンルーム内でのみ稼働** する。
- 部屋が無効化された場合（壁破壊・リークで密閉が崩れる、または機械が部屋外にある）、機械は **外に置いたのと同じ状態＝単純に稼働を停止** する。
- 部屋のクラスが製造対象グレードの要求に満たない場合は、最大グレードの天井で出力を制限する（稼働自体は継続）。

### 一瞬無効化への耐性（Valid / Degraded / Invalid ＋ 猶予）

壁の差し替え・建築途中・チャンクロード順・ハッチ/ドアの向きミスで部屋が一瞬無効化され、全機械が即停止する事故を防ぐため、部屋状態に段階と猶予を設ける:

- **Valid**: 密閉成立・クラス維持。通常稼働。
- **Degraded**: 一時的に密閉判定が崩れた／クラスが揺れた直後。**数秒の再評価猶予**の間は停止しない。
- **Invalid**: 猶予後もリークが継続。停止扱い。

これにより、建築編集直後の即時停止 flicker を避けつつ、本当のリークは確実に停止させる。

## 9. アイテム外部取り出し時の挙動

- アイテムをクリーンルーム外へ取り出しても、そのアイテムは変化しない。
- 純度はアイテムが持つ状態ではなく、製造が行われる瞬間に参照される部屋の属性として扱う（アイテム単位の汚染度トラッキングは行わない）。

## 11. 実装プランで確定する数値・詳細パラメータ

- クラスの濃度閾値（個/m³）と、クラスごとの **最大グレード／格下げ率／必要換気回数 requiredAirChangeRate**。
- ヒステリシスのマージン（クラス境界の上げ／下げで別閾値）。
- 清浄機の処理体積 q、フィルター1個あたりの除去容量、電力消費。
- 汚染係数: `A_machine`、`A_hatch`（搬送レート換算係数）、`A_door`（バースト量）、`a_volume`、`a_surface`、`a_connector`。
- Degraded 状態の再評価猶予時間。
- 3D flood-fill の近傍種別・上限体積・機械の体積算入・部屋の結合/分割時の N 配分・ロード時の room id 対応。
- 再検出の dirty-queue 処理・差分更新・1tickあたりの処理量上限。
- 部屋状態（N, V, S, クラス, 状態列挙）のセーブ／ロード形式。

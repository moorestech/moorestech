# AIミス検知レビュープロンプト チューニング — 収束サマリー

## 目的
moorestech の git 履歴で「AIが書いたコード → 人間が直したコード」のペアを回帰テスト化し、
**人間が直すことになるAIのミスを、マージ前にレビューで先回り指摘できる観点プロンプト**を育てる。
最終成果物 = `prompts/prompt-vN.md`（コードレビュープロンプト）。

## 方法論（このリポジトリ特化）
1. **ケースの来歴を blame で確定**（最重要）。`mine-valid-cases.py` で、人間の修正コミットが
   消した各行を `git blame <human>^` し、**その行を最後に書いたのが Claude 共著コミットか**を判定。
   AI由来の行を直したケースだけを有効とする。
   - 「人間が触ったファイル＝AIが触ったファイル」で対応付けると**誤帰属**する（実証済み。下記）。
2. 有効ケースごとに `inputs/caseX-ai-diff.md`（base→ai、レビュー対象）と
   `inputs/caseX-golden.md`（人間の修正＝捕捉すべき正解）を作る。golden はレビュアーに見せない。
3. **生成（レビュアー）subagent**: 観点プロンプト＋ai-diff だけでレビュー。
4. **評価**: golden の指摘を捕捉できたか（捕捉率）で pass/fail。
5. **過学習スキャン必須**: 観点プロンプトがケースの“答え”そのもの（具体API名・固有名）を
   例示していたら抽象カテゴリへ書き換える。

## 実証された教訓
- **誤帰属の危険**: 初回の「ファイル重複」ベース分析は、AIのせいでないミスをAIに帰属していた。
  - caseL（チェーンポール復元の再登録漏れ）= 2025年の古いコードで、対象AIコミットのdiffに無い → 無効。
  - caseS（インベントリ index 取り違え `_subInventory.Count`）= 2024年の**人間コード** bd007d41b 由来 → AIのミスでない。
  → 正しい来歴判定（blame）なしに「AIのミス」を語ると誤る。
- **過学習の危険**: v1 は答え（marker.Position/transform.position, GetKeyDown/GetKey 等）を例示しており、
  caseDismount の合格が水増しだった。v2 で抽象化したところ、答えを教えずとも本物の捕捉を再現。

## 結果（prompt-v2 = 非過学習）
| case | AIコミット | 人間修正 | 観点 | 結果 |
|---|---|---|---|---|
| caseV (不要な後方互換null) | 46797bfd1 | 338630bfa | V | ✅ 捕捉 |
| caseDup (コピペ重複) | 46797bfd1 | 4a6ddc108 | D | ✅ 捕捉（おまけにVも） |
| caseDismount (入力API取り違え) | b117406b5 | 72af4f1d1 | S | ✅ 捕捉（v2で答えなしでも） |

3/3 合格で収束。

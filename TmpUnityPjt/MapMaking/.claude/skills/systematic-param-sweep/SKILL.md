# Systematic Parameter Sweep

パラメーターを**1つずつ**変更し、効果を観察してから戻す。これを全パラメーターに対して繰り返し、
各パラメーターの影響を個別に把握してから最終調整を行うワークフロー。

## 鉄則

1. **一度に変えるのは1つだけ** — 複数同時変更は禁止
2. **カメラは固定** — 観察中にカメラ角度・位置を変えない
3. **必ず戻す** — 観察後はパラメーターを元の値に戻してから次へ進む
4. **記録する** — 各変更の効果をメモし、最後に知見を整理する

## ワークフロー

### Phase 0: 準備

1. 現在のパラメーター値を**すべて記録**する（ベースライン）
2. カメラ位置・角度を固定する（以降触らない）
3. ベースラインのスクリーンショットを撮る + 外部監査でベースライン評価を取得

```
# ベースラインスクリーンショット
uloop screenshot --window-name Scene
node tools/codex-audit.mjs <screenshot_path> --ask "【評価基準】[対象バイオーム]【確認観点】ベースライン状態の記録"
```

### Phase 1: パラメーター分離スイープ

対象パラメーターを1つ選び、以下のサイクルを回す。

#### Step 1: 大胆な変更で影響範囲を把握

目的: そのパラメーターが**何に影響するか**を大まかに掴む。

- **2倍** にして観察 → 戻す
- **10倍** にして観察 → 戻す
- **100倍** にして観察 → 戻す（意味がある場合のみ）
- **0** または **極小値** にして観察 → 戻す

各変更後:
1. マップを再生成する
2. スクリーンショットを撮る
3. ベースラインとの差分を記録する（何が変わったか、何が変わらなかったか）

```
# パラメーター変更 → 再生成 → スクリーンショット のサイクル
# 1. パラメーターを変更（コード編集 or dynamic code）
# 2. 再生成
uloop execute-dynamic-code --code '
var go = UnityEngine.GameObject.Find("MapGenerator");
var facade = go.GetComponent<MapGenerator.MapGeneratorFacade>();
facade.Generate();
return "Generated";
'
# 3. スクリーンショット
uloop screenshot --window-name Scene
# 4. 外部監査で差分確認
node tools/codex-audit.mjs <screenshot_path> --ask "【確認観点】[パラメータ名]を[元値]→[新値]に変更。ベースラインとの差分は？" --session <UUID>
# 5. パラメーターを元に戻す
```

#### Step 2: 微調整で感度を把握

目的: そのパラメーターの**感度**（少し変えたらどれくらい変わるか）を掴む。

- **+10%** にして観察 → 戻す
- **+30%** にして観察 → 戻す
- **-10%** にして観察 → 戻す
- **-30%** にして観察 → 戻す

#### Step 3: 記録

そのパラメーターについて以下を記録:
- **影響対象**: 何に影響するか（地形の高さ、ノイズの粒度、密度など）
- **感度**: 高感度（少し変えると大きく変わる）/ 低感度（大きく変えても変化が小さい）
- **方向性**: 値を上げると何が起き、下げると何が起きるか

#### Step 4: 戻して次へ

パラメーターを**必ずベースライン値に戻し**、次のパラメーターへ進む。

### Phase 2: 知見の整理

全パラメーターのスイープ完了後、知見を整理する:

```
## パラメーター影響マップ
| パラメーター | 影響対象 | 感度 | 備考 |
|---|---|---|---|
| exponent | 地形の急峻さ | 高 | 2→4で劇的に変化 |
| frequency | ノイズの粒度 | 中 | ... |
| ... | ... | ... | ... |
```

### Phase 3: 最終調整

知見に基づいて、目標の地形に向けてパラメーターを調整する。

1. Phase 2 の影響マップから**高感度パラメーター**を特定する
2. 目標に最も影響するパラメーターから順に調整する
3. 1回の変更は**微調整レベル**（10-30%）に留める
4. 変更ごとにスクリーンショット + 外部監査で確認する
5. **A評価になるまで** 調整→再生成→監査のループを回す

この段階では複数パラメーターを組み合わせて変更してよいが、
相互作用が強いパラメーター（例: frequency と amplitude）は
片方を固定してもう片方を調整すること。

## Gotchas

- **bool型/enum型パラメーター**: 倍率変更は意味がない。ON/OFF切替で影響を観察する
- **0にするとクラッシュ**: 除算に使われるパラメーターは0にできない。極小値（0.001等）で代用
- **変化が見えない**: スクリーンショットの画角が狭い可能性。Scene全体を俯瞰する画角で確認
- **コンパイルエラー**: コード定数を変更した場合、再生成前にコンパイル成功を確認する

## 注意事項

- パラメーターがコード内の定数の場合はコード編集 → コンパイル → 再生成の手順になる
- SerializedField の場合は dynamic code で直接変更可能
- スイープ対象が多い場合は、まず大胆な変更だけで全パラメーターをざっと見て、
  影響の大きいものを優先的に微調整する（全部を微調整する必要はない）
- 外部監査セッションは最初に1つ作り、全スイープで同一セッションを使い回す

# バイオームハイトマップ調整ガイド

## Phase 0: バイオーム要件定義

パラメータに触れる前に、**どんな地形を作りたいか**を具体的に定義する。
抽象的な印象語（「いい感じ」「自然な感じ」）ではなく、**ゲームプレイに直結する具体的な仕様**を引き出す。

### 簡易モード（ユーザーが参考画像や具体的フィードバックを出した場合）

参考画像・スクリーンショット付きフィードバックがある場合は質問フローを省略し、画像から要件を読み取って確認するだけでよい。

### 標準モード（要件定義ヒアリング）

**1問ずつ**聞く。各質問で**3個の選択肢を提示**する。質問内容は以下の4つの観点をカバーするようにその場で考える。

1. **地形の骨格（マクロ構造）** — どんな大きな形か。丘・山・台地・峡谷など。高さやスケール感
2. **表面の質感（ミクロ構造）** — 歩いたとき足元はどう感じるか。滑らか・デコボコ・ゴツゴツ
3. **プレイアビリティ** — この地形でプレイヤーは何をするか。建築・探索・通過。歩行制限の有無
4. **制約** — パラメータのみ or アルゴリズム変更OK。参考画像の有無

質問はバイオームの世界観やゲームプレイに踏み込む形で深掘りし、「いい感じ」「自然な感じ」のような曖昧な印象語を具体的な仕様に変換することを目指す。

### 要件サマリー

```
## バイオーム要件
- 対象: [バイオーム名]
- 骨格: [丘陵 / 山岳 / 台地] — 高さ: [低/中/高]、幅: [広/狭]
- 質感: [滑らか / ゆるやかデコボコ / ゴツゴツ]
- 用途: [建築 / 探索 / 通過]
- 歩行: [全域歩行可能 / 一部崖あり]
- 制約: [パラメータのみ / アルゴリズム変更OK]
- 外部監査の評価基準: [上記要件をチェックリスト化したもの]
```

**承認を得てから Phase 1 に進む。**

## Phase 1: 調整ワークフロー

**原則: 3D地形評価を最優先する。** 2Dハイトマップは補助的なアルゴリズムデバッグ用。

### メインフロー（3D優先）

1. **現状パラメータ取得** — uloop dynamic-codeで現在値を読む
2. **地形再生成 → スクリーンショット → 外部監査（3D）** — 下記の手順を**一連の不可分な操作**として実行:
   1. `facade.Generate()` で地形再生成
   2. `uloop screenshot` でSceneビューをキャプチャ
   3. **即座に** `node tools/gemini-audit.mjs` で外部監査を実行
   4. 監査結果をユーザーに報告
   **スクリーンショットを撮ったら、自分の目視判断を述べる前に必ず外部監査を実行すること。**
3. **パラメータ変更 → 再生成 → 3D監査（反復）** — 大きな変更から始め、徐々に微調整
4. **必要に応じて2Dハイトマップで確認** — アルゴリズムのノイズパターンをデバッグする場合のみ

### 2D監査の限界（重要）

BiomeHeightmapExporterはmin-max正規化するため、**以下のパラメータの効果は2D PNGに反映されない**:
- `amplitude` — 3Dの高低差に直結するが、正規化で消える
- `exponent` — 値分布は変わるが見た目の変化は微妙
- `baseHeight` — 完全に正規化で消える

**2D監査で「良い」と判断されても、3Dでは全く違う結果になることがある。** 2Dでの反復調整に時間をかけすぎず、早期に3Dで確認すること。

### ユーザーからのフィードバック時の要件明確化フロー

ユーザーが3D地形に対してフィードバックを出した場合（「もっと山感を」「歩きにくい」等）:

1. **要件を具体的な評価基準に変換する** — ユーザーの言葉を外部監査に渡せる明確な基準に落とし込む。曖昧なまま調整を始めない。
   - 例: 「山頂が尖ってる」→「山頂に平坦面積があるか」「斜面の勾配が歩行可能か」
   - 例: 「面白くない」→「高低差がダイナミックか」「谷と尾根の変化があるか」
   - 例: 「デコボコすぎる」→「ディテールの振幅が過剰か」「歩行時にガタつかないか」
2. **変換した要件をユーザーに確認する** — 「以下の基準で評価しますがよいですか？」
3. **パラメータ調整後、外部監査に変換した要件を渡す** — `--prompt` に要件チェックリストとして含める

---

## 禁止事項

- **スクリーンショット確認後、外部監査なしにユーザーに報告してはならない。** 自身の目視確認は外部監査の代替にならない。CLAUDE.mdの「重要事項」に明記されたルールである。
- 外部監査を省略する理由（「明らかに良さそう」「前回と似ている」等）は一切認めない。

## パラメータ影響マップ

症状から原因パラメータを特定する早見表。

### 基本パラメータ

| 症状 | 主な原因 |
|---|---|
| 液体/メタリック感 | octaves過多 + warp過多 + terrace量子化の組み合わせ |
| 平坦すぎ/起伏なし | terraceHeight高すぎ（テラスが元の起伏を消す） |
| 鋭い稜線/スパイク | ridgeBlend（エッジへの加算ノイズ）が強すぎ |
| のたうち回る線/血管状の溝 | abs()ノイズ（ValleyNetwork, Ridged）のゼロ交差アーティファクト → absSmoothing で解決 |
| 斜面のザラつき | octaves or persistence が高すぎ |
| 段差境界が直線的 | boundaryNoiseStrength不足 |
| のっぺり/情報量不足 | octaves不足 or warpStrength不足 |
| 流体/マーブル模様の表面 | ドメインワープ座標にディテールノイズを適用している → **元座標(pos)に適用** |
| 崖面のジャギ/階段アーティファクト | smoothstep遷移が狭すぎ or 高周波ノイズがハイトマップ解像度を超過 |
| 薄い壁状のスパイク | pow()で尖らせた高周波fBmピークがsmoothstep閾値をギリギリ超過 |

### 3D固有の症状（2D PNGでは見えない）

| 症状 | 主な原因 |
|---|---|
| 3Dで平坦すぎる/山感がない | amplitude不足（2D正規化では見えない） |
| 山頂が尖って歩けない | plateauFlattenなし or exponent高すぎ |
| 谷が水没する | lowlandCutoff設定時にbaseHeightが低すぎる |
| 斜面が波打つ/シワ状 | warpStrengthやridgeBlendが斜面に干渉 |
| デコボコが谷だけ弱い | `detail * weight * terrain`の乗算マスク（均一にするなら`* terrain`を除去） |
| デコボコが均一すぎて単調 | ディテールノイズにマスクやDomain Warpがない |
| 台地エッジが角張る | ハードクランプ（math.min）使用時 → ソフトクランプに変更 |

## アルゴリズム変更ガイド

パラメータ調整だけで解決できない場合、以下のアルゴリズム変更パターンを検討する。

### 山頂平坦化の方式

| 方式 | コード例 | 特徴 |
|---|---|---|
| smoothstep | `s = t²(3-2t); lerp(terrain, s, blend)` | 全体を丸める。山頂も谷も平坦化。ダイナミクスが減る |
| ハードクランプ | `math.min(terrain, limit)` | 確実に平坦だが台地エッジが角張る |
| ソフトクランプ | 閾値以上をsmoothstepで圧縮 | 台地＋自然なエッジ丸み。**推奨** |

### リッジ合成の方式

| 方式 | コード例 | 特徴 |
|---|---|---|
| lerp | `lerp(fbm, ridge, blend)` | リッジが支配的になりやすい（リッジ値がFBmより極端なため） |
| 乗算 | `fbm * (1 + ridge * blend)` | FBmの塊感を維持しつつリッジでシャープさを付与。**推奨** |
| 加算 | `terrain += ridge * blend * terrain` | terrainマスクで高い場所にだけリッジ。山頂強調向け |

### abs()折り返しアーティファクトの対処

`math.abs(noise)` や `ridgeOffset - math.abs(...)` はゼロ交差点で微分不連続を起こし、「のたうち回る線」状のアーティファクトを生む。パラメータを0にすれば線は消えるが起伏ごと消失する。

| 方式 | コード例 | 特徴 |
|---|---|---|
| 従来のabs | `math.abs(x)` | ゼロ交差に鋭い折り目。アーティファクトの原因 |
| SmoothAbs | `sqrt(x*x + k*k)` | k>0でゼロ交差を丸める。k=0で従来absと同等。**推奨** |

**適用時の原則: ノイズステージごとにスムーズ化強度を分ける。**
- **渓谷（SimpleValleyNetwork）** — 平地の線の主原因。フルの `absSmoothing` を適用
- **リッジ（Ridged）** — 山の稜線のシャープさを保つため `absSmoothing * 0.5` に抑える

砂漠バイオームでの実績値: `absSmoothing=0.1`（渓谷=0.1、リッジ=0.05）、`cliffAmplitude=0.22`。

### ディテール適用の方式

| 方式 | コード例 | 特徴 |
|---|---|---|
| 均一加算 | `terrain += detail * weight` | 全域に均一なデコボコ。**多くの場合推奨** |
| terrainマスク付き | `terrain += detail * weight * terrain` | 山頂は強く谷は弱い。意図的に差を付けたい場合のみ |

### ビュート/台地の孤立化方式

| 方式 | コード例 | 特徴 |
|---|---|---|
| pow()ピーク狭窄 | `pow(fBm, 3.0)` | 簡単だが薄い壁/スパイクが生じやすい。**ゲーム品質で問題あり** |
| 積ノイズ | `fBm1 * fBm2` | 2つの独立ノイズ場の積。薄い壁を回避しつつ自然な孤立ビュートを生成。**推奨** |
| 高閾値 | `smoothstep(0.6, 0.8, fBm)` | 単純だがfBmのピークが広く繋がりやすい |

**積ノイズの原理**: 独立した2つのfBm場の積は、両方が高い地点だけがピークになるため、自然に孤立した塊状のビュートを形成する。pow()のように薄い壁を生じない。

## ゲーム品質制約（Unity Terrain）

**ハイトマップ方式のTerrainでは以下の物理的制約がある。参考画像が垂直な崖を含む場合でも、これらの制約を優先すること。**

### 斜面の勾配制限
- **cliffSteepness（smoothstep halfWidth）の上限**: 5-7（halfWidth=0.14-0.2）。これ以上急にすると階段状ジャギが発生
- **ハイトマップ解像度**: 513x513で2000mの場合、1サンプル≈3.9m。急勾配ほどサンプル間の高低差が大きくなりジャギが目立つ
- **テクスチャUV**: 垂直に近い面ではUVが極端に引き伸ばされる。斜面角度50-65度が実用上の上限

### ドメインワープの制限
- **warpStrength上限**: 100-200（バイオームにより異なる）。300以上で流体/マーブル模様が発生
- **ディテールノイズにはワープ座標を使わない**: 高周波ノイズをドメインワープ座標で評価すると、ワープによる伸縮で流体的なパターンになる。**元座標(pos)を使うこと**

### ノイズ周波数の制限
- ディテール/ラフネスの最高周波数: `frequency * 10-15`。これ以上はハイトマップの解像度限界でエイリアシングを起こす
- ディテールノイズのオクターブは2-3で十分。高オクターブ+高persistenceで薄い壁やスパイクの原因になる

### メサバイオームの実績値（2026-03-20確認）
```
積ノイズ孤立化 + 二段smoothstep方式:
- octaves=3, persistence=0.35, isolationFreqMult=1.7
- butteThreshold=0.45, cliffSteepness=10 (halfWidth=0.1)
- plateauFlatten=0.45 (二段smoothstepで頂上フラット化)
- warpStrength=100, frequency=0.001, amplitude=0.35
- floorVariation=0.08
- ラフネス: pos座標, freq*10, 3oct, 強度0.8*floorVariation
- 谷底ディテール: pos座標, freq*4, 2oct
→ 外部監査B評価（構図A, フラットトップA, ディテールB, 歩行A, ゲーム品質B）
→ 崖面の岩肌質感はテクスチャ/マテリアル側で補完する領域
```

## 重要な注意点

### エクスポーターのmin-max正規化
BiomeHeightmapExporterは出力を0-1に正規化する。**amplitude, exponent, baseHeightの効果は2D PNGに反映されない。** 2D PNGはノイズの「形状パターン」の確認にのみ使い、地形の高低差・傾斜・歩行感の評価は必ず3Dで行うこと。

### terraceHeightはブレンド比率
`math.lerp(terrain, terraced, terraceHeight)` で実装。1.0=完全テラス（平坦）、0=テラスなし。中間値でテラスの段差を保ちつつ元のFBm起伏を残せる。

### ridgeBlendの副作用
edgeMask^1.5にリッジノイズを**加算**するため、段差境界に鋭い突起が生じる。穏やかなバイオームでは0にし、急峻なバイオームでのみ慎重に使う。

### 外部監査の運用
- 異なるシード間の「配置」一致は不可能。**スタイル**類似度で評価する
- **2D監査は矛盾しやすい**（正規化により情報が欠落するため）。3D監査を優先する
- **Phase 0で合意した基準を毎回 --prompt に渡す**ことで一貫性を保つ
- `--session` でキャッシュを再利用する

# External Audit — 外部AI監査・相談ワークフロー

`tools/codex-audit.mjs` は Codex CLI (`codex exec`) をラップした外部監査ツール。スクリーンショット/ハイトマップ画像を渡して評価を得るだけでなく、画像なしでアルゴリズム・パラメータの改善相談もできる。

## 前提条件

- `codex` CLI がインストール・ログイン済み（`which codex` で確認）
- カレントディレクトリはプロジェクトルート（`pwd` が `.../MapMaking` またはその worktree）
- `tools/audit-requirements.md` が存在する（新規セッション時のシステムプロンプトとして自動挿入される）
- Node.js が利用可能。画像を渡す場合は `sharp` が `node_modules/` にあること（1MB超の画像を自動圧縮するため）

## モード選択

| モード | 用途 | 渡すもの | 使いどころ |
|---|---|---|---|
| **評価モード** | スクショの品質判定 | 画像 1 枚以上 + `--ask` | バイオーム見た目チェック、ハイトマップ評価、Scene View 確認 |
| **比較モード** | 複数画像の差分評価 | 画像 2 枚以上 + `--ask` | 参考画像との類似度、before/after（専用スキルあり → `before-after-audit`） |
| **相談モード** | コード/アルゴリズム相談 | `--ask` のみ（画像省略可） | パラメータ調整方針の検討、アルゴリズム改善相談、A 達成のための戦略立案 |

評価と相談はセッションを跨いで混在可能。**同一セッション内で評価 → 相談 → 修正 → 再評価のループ**を回すのが典型パターン。

## 基本コマンド

### 新規セッション（初回）

```bash
# 評価モード
node tools/codex-audit.mjs <画像> [画像2 ...] --ask "【評価基準】... 【確認観点】..."

# 相談モード（画像なし）
node tools/codex-audit.mjs --ask "【背景】... 【相談内容】... 【現在のパラメータ】..."
```

初回呼び出し後、**stderr** に `--session <UUID> を指定してください` が出る。stdout ではないので注意。以降は同一セッションに再利用する。

### セッション再開（2 回目以降）

```bash
node tools/codex-audit.mjs [画像...] --ask "..." --session <UUID>
```

セッション再開時は `audit-requirements.md` は再挿入されない（初回のみ）。監査人は前回の文脈を保持している。

## `--ask` の書き方

`--ask` はスキル発動の唯一のトリガー。曖昧だと監査結果が薄くなる。以下の 2 要素を必ず含める：

1. **評価基準** — 対象がどのバイオームか、どんな地形/見た目を期待するか、参考にすべき要件
2. **確認観点** — 今回のスクショ/相談で具体的に何を判断してほしいか

### 評価モード例

```
--ask "【評価基準】Grassland バイオームのハイトマップ。なだらかな丘陵と広い平原が中心で、
急峻な山脈は含まれない。fBm ベースのノイズで有機的なパターンを期待する。
【確認観点】前回 C 評価だった「勾配の単調さ」と「高周波ノイズの不足」が改善されているか。
特に中央部の平原の起伏感と、丘の稜線の自然さを重点的にチェックしてほしい。"
```

### 相談モード例

```
--ask "【背景】MapGenerator の Desert バイオームで、砂丘のスケール感が現実離れして大きすぎる。
SampleHeight は fBm 3 オクターブ + Domain Warp 0.3。スケールは baseFrequency=0.002。
【現在のパラメータ】hillAmplitude=80, exponent=1.8, duneScale=0.005
【相談内容】自然な砂丘サイズに近づけるにはどのパラメータをどう変えるべきか。
関連コード: Assets/MapGenerator/Pipeline/Biomes/Desert/DesertBiomeConfig.cs を読んで提案してほしい。"
```

### コード相談を強化するコツ

監査人はコードパスを渡すと**自主的にファイルを読みに行く**。改善提案の質が上がるので積極的に渡す：

- 関連する `*BiomeConfig.cs` のパス
- 設定されている具体的なパラメータ値（Inspector の値そのまま）
- 現在のアルゴリズム概要（「fBm 3 オクターブ」「Domain Warp 有り」等）
- これまでに試した修正と結果

## セッション管理

- **ユーザーから特別な指示がない限り、常に同一セッションを使い続ける**。監査人が前回の文脈（試した修正・前のスクショ・議論の経緯）を保持しているほうが評価が正確になる
- セッション ID は必ずメモ（TodoList やチャット中に残す）。一度失うと前回文脈が復元不能
- セッションが長期間（数日以上）放置されると `codex exec resume` で「セッション期限切れ」エラーが出ることがある。その場合は新規セッション作成のみ可能
- **別バイオーム・別テーマ**の監査を始める時は新規セッションを切る（文脈混線を防ぐため）

## 出力の読み方

監査結果は `audit-requirements.md` で定義されたフォーマットで返る：

1. **要件チェックリスト**（表形式、項目ごとに A〜F）
2. **詳細分析**（観察事実に基づく）
3. **総合評価: [A/B/C/D/F]**
4. **A 達成までに必要なこと**（次の打ち手）

stdout に監査結果本文、stderr にセッション ID 案内（新規時のみ）が出る。スクリプトで結果を受けるときは stdout のみ拾えばよい。

## A 評価未達時の動作ルール

**A 評価以外が返ったらユーザー確認を待たず自主的に修正ループを回す。** CLAUDE.md に明記されたプロジェクトルール：

1. 監査結果の「A 達成までに必要なこと」を読む
2. パラメータ調整を優先（コード変更は最終手段）
3. 再生成 → 再スクショ → **同一セッションで**再監査
4. A 評価に到達するまでループ
5. A 評価達成後に初めてユーザーに報告

自身の目視と外部監査の結果が矛盾する場合は**外部監査を優先**する（自分の判断より第三者視点を信頼する）。

## Gotchas

- **画像は省略可能だが `--ask` は必須。** 省略すると即エラー終了
- **セッション ID は stderr に出る。** `2>&1` でマージしていないと見落とす。Bash で実行するときは stderr も確認
- **同一セッションでの `--ask` だけ渡すパターンは相談モード**。画像を付ければ評価モード。モードを切り替えても文脈は保持される
- **1MB 超の画像は自動圧縮される**（長辺 2048px、JPEG 品質 85 → 55 段階）。圧縮失敗時は元画像で続行（警告のみ）
- **複数画像を渡した場合、監査人は画像間の比較も行う**。参考画像と出力を並べるときに有用
- **画像に写っていないものは評価しない**（監査人ルール）。例えば「遠景の山脈」を評価したいなら、その山脈がフレームに入っているスクショを渡す
- **新規セッション時のみ `audit-requirements.md` がシステムプロンプトとして挿入される**。セッション再開時は `--ask` の指示のみが追加される。「最初の監査基準が欲しい」場合は新規セッションを切る
- **`before/after` 比較が目的なら `before-after-audit` スキルを使う。** 視点固定など別の制約が加わる
- **ハードコードされた座標→出力マッピングの提案**（「座標(x,y)のとき出力 Z」等）は受け入れない。プロシージャル生成の原則に反する（CLAUDE.md）
- **セッションID の取り違え**: 複数バイオームを並行で監査していると ID を混同しがち。バイオームごとに別メモで管理する

## 典型ワークフロー

```
1. Scene View スクショ (uloop screenshot --window-name Scene)
2. 新規セッションで評価
   → node tools/codex-audit.mjs screenshot.png --ask "【評価基準】...【確認観点】..."
   → stderr から UUID 取得
3. 結果が A 未満なら「A 達成までに必要なこと」を読む
4. 必要なら同セッションで相談
   → node tools/codex-audit.mjs --ask "【相談内容】この指摘について具体的にどのパラメータを..." --session UUID
5. パラメータ調整 → 再生成（facade.Generate()） → 再スクショ
6. 同セッションで再評価
   → node tools/codex-audit.mjs new_screenshot.png --ask "修正後。前回からの改善点を..." --session UUID
7. A 評価到達までループ
8. A 評価到達後にユーザー報告
```
